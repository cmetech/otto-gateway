# Phase 08.4 Plan 01: US Address PII Coverage Summary

One-liner: three regex recognizers (USAddress, USState, USZIP) + validator + integration test + smoke-fixture extension that closes PII-01 and silences the 2026-06-04 splunk-box probe's PERSON false positives on US street names via overlap arbitration.

## Goal

Add US-address coverage to the PII redactor so encrypt-mode tokenizes `1111 Main Street, Austin, TX 27584` byte-for-byte across all three API surfaces, AND silence the prose-v2 NER's PERSON false positives on street names ("Main Street" → PERSON, etc.) by registering the USAddress regex BEFORE the NER stage so the existing overlap arbiter at `internal/plugin/pii/pii.go:258-277` drops the NER candidate.

## What Built

**Mechanically additive — no engine, hook-chain, or config-loading changes:**

1. **Three regex literals** in `internal/plugin/pii/recognizers.go`'s existing `var (...)` block:
   - `usZIPRe`: `\b\d{5}(?:-\d{4})?\b` + `validateUSZIPRange` rejects all-same-digit codes.
   - `usStateRe`: two-arm alternation — `(?:,\s+|^|\n)` lead but with line-start arm requiring `\s+\d{5}` ZIP trail (AP-2 strict mitigation; see Decisions §1).
   - `usAddressRe`: `\b\d{1,6}[ \t]+[A-Z][A-Za-z]*(?:[ \t]+[A-Z][A-Za-z]*)*[ \t]+<16-suffix-vocab>\b\.?` — tightened to `[ \t]+` (no `\n` in `\s`) to enforce single-line addresses per Pitfall 3.
2. **One validator function** `validateUSZIPRange` — mirrors `validateSSNRange` shape.
3. **Three `Recognizer{}` entries** appended to the `Recognizers` slice in order `USAddress → USState → USZIP` (largest span first, AP-4 mitigation).
4. **Three keys** added to `piiAllowedEntities` in `internal/config/config.go` between `"SITE"` and the PERSON/LOCATION NER block.
5. **Six new captured-span / reject tests + one paired regex+validator test** in `recognizers_test.go`, all using exact-string equality on `r.Pattern.FindString` (AP-5 mitigation).
6. **One new integration test** `TestPIIRedactionHook_USAddressFullCoverage` in `pii_test.go` — drives the full `*PIIRedactionHook` with `NER: NewNEREngine()` against the canonical address fixture; asserts `Counts["USAddress"]==1, USState==1, USZIP==1, LOCATION==1, PERSON==0`.
7. **`TestRecognizers_RegistryShape`** `wantNames` slice grew 13 → 16 in registration order.
8. **REQUIREMENTS.md**: new `### PII — Recognizer coverage` section before `## v2 Requirements`, PII-01 entry, Traceability row, v1 count `62 total → 63 total` with footnote extension.
9. **scripts/test-pii.ps1 + scripts/test-pii.sh**: extended in lockstep (ONE commit, PS1/SH parity gate) with the `Site address: 1111 Main Street, Austin, TX 27584` fixture line and three needles (`1111 Main Street`, `TX`, `27584`). ASCII-only invariant preserved (PS1 still `ASCII text` per `file(1)`; SH still no BOM).

## Decisions Made

1. **USState regex restructured to two alternation arms** (Rule-1 deviation — see `Deviations` below). The literal RESEARCH.md shape `(?:,\s+|^|\n)<STATES>(?:\b|\s+\d{5}|\.|,)` allowed line-start English-word collisions to match (`OK, that works`, `TX is a state code`). The fix splits the regex into two arms: comma-prefixed (any trail allowed) and line-start-or-newline (ZIP trail required). Negative-control RED tests now pass.
2. **USAddress whitespace tightened** to `[ \t]+` so multi-line text cannot smuggle into a single address span. Honors RESEARCH §Pitfall 3.
3. **Pre-existing gofumpt drift in `internal/acp/client.go`** split into a separate `chore(gofumpt)` commit (`8b44262`) — keeps the REFACTOR atomic at exactly the five planned files while unblocking `make fmt-check`. The drift was confirmed pre-existing on base commit `315f1cc` via stash + recheck.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] USState regex shape needed two-arm restructure to honor AP-2**

- **Found during:** Task G GREEN — RED test `TestUSStateRecognizer_RejectsEnglishWords/ok-as-acknowledgment` and `/bare-tx-no-context` failed against the literal RESEARCH.md regex.
- **Issue:** The literal `(?:,\s+|^|\n)<STATES>(?:\b|\s+\d{5}|\.|,)` lead alternation allowed line-start matches with ANY trail. Result: `OK, that works for me.` matched (lead `^`, code `OK`, trail `,`), and `TX is a state code but lacks context here.` matched (lead `^`, code `TX`, trail `\b`). Plan's AP-2 truth says these MUST not match.
- **Fix:** Split the regex into two alternation arms: arm 1 (comma-prefixed) keeps the original permissive trail; arm 2 (line-start) requires `\s+\d{5}` (ZIP must follow). Line-start English-word collisions are now rejected at the regex level.
- **Files modified:** internal/plugin/pii/recognizers.go (usStateRe literal + extended doc comment).
- **Commit:** `8ac2da6`.

**2. [Rule 1 — Bug] USAddress regex `\s+` let newlines smuggle multi-line text into a single span**

- **Found during:** Task G GREEN — RED test `TestUSAddressRecognizer_RejectsNonAddressShapes/multiline-name` failed: `1111 Main\nStreet today.` matched as a single span because RE2's `\s` includes `\n`.
- **Issue:** The literal regex used `\s+` between TitleCase words. Per RESEARCH §Pitfall 3, the intent is single-space/tab inter-word whitespace, NOT cross-line.
- **Fix:** Replaced `\s+` with `[ \t]+` inside `usAddressRe`. Multi-line names now correctly fail to match.
- **Files modified:** internal/plugin/pii/recognizers.go (usAddressRe literal + Pitfall 3 doc comment).
- **Commit:** `8ac2da6`.

**3. [Rule 1 — Test correction] RED `TestUSStateRecognizer_CapturedSpan` expectations had spurious trailing space**

- **Found during:** Task G GREEN.
- **Issue:** RED test asserted `wantCapture = ", TX "` (with trailing space) for input `, Austin, TX 27584.`. RE2 leftmost-first alternation picks the zero-width `\b` trail over the longer `\s+\d{5}` arm; actual captured span is `, TX`. USZIP fires separately on the trailing `27584` per Pitfall 7 — the load-bearing contract (three separate tokens) is preserved.
- **Fix:** Updated RED test expectations to `", TX"`, `", MA"`, `", HI"` with an explanatory comment.
- **Files modified:** internal/plugin/pii/recognizers_test.go.
- **Commit:** `8ac2da6` (folded into the minimum-diff GREEN commit).
