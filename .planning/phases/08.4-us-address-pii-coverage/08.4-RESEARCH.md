**Researched:** 2026-06-04

## Summary

Phase 8.4 adds three new regex-only `Recognizer` entries to `internal/plugin/pii/recognizers.go` — `USAddress`, `USZIP`, `USState` — and one entry to the `piiAllowedEntities` map in `internal/config/config.go` for each. The work is mechanical: the existing pipeline at `pii.go:215` (regex collect) and `pii.go:277` ("Skip NER candidates that overlap any regex span") already gives the new recognizers precedence over the prose-v2 NER's PERSON / LOCATION emissions, so the documented address-region PERSON false positives ("Main Street" → PERSON, "Pennsylvania Avenue" → PERSON, "Apple Park" → PERSON) are suppressed automatically as soon as a `USAddress` span covers the same bytes. No engine, hook-chain, or config-loading-flow changes are required.

The phase's load-bearing decisions are not technical — they are surface decisions: (1) which REQ-ID neighborhood the new requirement lives under (PII-* is a new section vs. PLUG-07 continuation), (2) whether `USAddress` swallows adjacent `, City, ST ZIP` or each component emits its own atomic token, and (3) how the smoke-test fixture extension is shaped. The recognizers themselves are short, RE2-compatible, and well-precedented inside the existing 13-recognizer registry.

**Primary recommendation:** Three concrete regex patterns plus one `validateUSZIPRange` validator (rejects all-same-digit ZIPs); registration order `USAddress → USState → USZIP` (largest span first so the "first recognizer wins" overlap rule produces the expected coverage); one new section `### PII — Recognizer coverage` in REQUIREMENTS.md owning `PII-01` for the new requirement; smoke-test fixture extension by adding one address line + three needles to the existing handover-note prompt rather than a new scenario. The `USAddress` recognizer should emit a token per atomic entity (street separate from ZIP separate from state) — matches the existing model where COORDINATES emits one token and IPv4/IPv6 emit individually adjacent tokens. Each of the three new recognizers fires independently, the existing `mergeSpansGreedy` correctly handles three non-overlapping adjacent regex spans.

### Pattern 1: Init-Time Compiled Regex

**What:** All regex literals compile at package init via `regexp.MustCompile`. Malformed regex panics at binary boot — surfaces regressions before any request is handled.

**When to use:** Always for the new patterns.

**Example:**

```go
// Source: internal/plugin/pii/recognizers.go:80-100
var (
    usPhoneRe = regexp.MustCompile(`(?:\+?1[ .\-]?)?\(?\b[2-9][0-9]{2}\)?[ .\-]?[0-9]{3}[ .\-]?[0-9]{4}\b`)
    // ...add the three new ones here, alphabetized within their grouping or appended:
    usZIPRe     = regexp.MustCompile(`\b\d{5}(?:-\d{4})?\b`)
    usStateRe   = regexp.MustCompile(`(?:,\s+|^|\n)([A-Z]{2})(?:\b|\s+\d{5}|\.|,)`)  // see USState design notes below
    usAddressRe = regexp.MustCompile(`\b\d{1,6}\s+[A-Z][A-Za-z]*(?:\s+[A-Z][A-Za-z]*)*\s+(?:St|Street|Ave|Avenue|Blvd|Boulevard|Rd|Road|Dr|Drive|Ln|Lane|Way|Pl|Place|Ct|Court|Pkwy|Parkway|Cir|Circle|Ter|Terrace|Sq|Square|Hwy|Highway)\b\.?`)
)
```

`TestRecognizers_CompiledAtPackageInit_NoPerRequestCompile` at `recognizers_test.go:514` is the source-level guard — it greps `recognizers.go` for `regexp.MustCompile(` and asserts at least one call per unique pattern. Three new MustCompile calls will satisfy the existing assertion automatically.

### Pattern 2: Registration Order Controls Span Priority

**What:** `collectRegexSpans` at `pii.go:227-233` rejects any candidate span that overlaps an already-accepted span:

```go
// Source: internal/plugin/pii/pii.go:227-233
for _, a := range out {
    if cand.overlaps(a) {
        conflict = true
        break
    }
}
if conflict {
    continue
}
```

The first recognizer registered for a byte range wins. If `USAddress` is registered BEFORE `USState`, the recognizer matching "1111 Main Street" takes priority over a `USState` candidate inside it — but because `USAddress`'s pattern won't match `, TX ` and `USState` won't match `1111 Main Street`, they don't actually conflict in practice. **Register order: `USAddress, USState, USZIP`** so that any future regex tightening doesn't accidentally let a smaller match steal bytes from a larger one.

**When to use:** This is the engine. Phase 8.4 only chooses the registration order in the `Recognizers` slice.

**Example:** See "Code Examples" section below for the proposed slice diff.

### Pattern 3: Optional Validator Function for RE2-Inexpressible Filters

**What:** When the regex shape is intentionally permissive, a `Validate func(string) bool` post-filter rejects false positives.

**When to use:** USZIP needs this. The raw regex `\b\d{5}(?:-\d{4})?\b` matches things like `00000`, `99999`, and `12345` (a PIN code, a date, etc.). The validator should reject obvious non-ZIP shapes:

```go
// Source: internal/plugin/pii/recognizers.go pattern (proposed)
//
// validateUSZIPRange rejects obvious non-ZIP shapes that the permissive
// \b\d{5}(?:-\d{4})?\b regex accepts. Rules per USPS pattern + common
// sense:
//   - All-same-digit codes are rejected (00000, 11111, ..., 99999).
//     These are not valid USPS ZIP assignments; the regex shape is
//     intentionally loose, validator tightens it.
//   - 00000 is explicitly reserved (not assigned).
//   - We DO NOT validate against the USPS first-digit region table
//     (0 = Northeast, 1 = NY/PA/DE, ..., 9 = West Coast / Hawaii /
//     Alaska) because new ZIPs are assigned occasionally and the table
//     drifts. The regex is conservative enough at the shape level.
func validateUSZIPRange(s string) bool {
    // Strip the optional "-NNNN" extension; we only validate the 5-digit base.
    base := s
    if dash := strings.IndexByte(s, '-'); dash > 0 {
        base = s[:dash]
    }
    if len(base) != 5 {
        return false
    }
    // Reject all-same-digit runs (00000, 11111, ..., 99999).
    allSame := true
    for i := 1; i < len(base); i++ {
        if base[i] != base[0] {
            allSame = false
            break
        }
    }
    return !allSame
}
```

**When NOT to use:** USAddress and USState are intentionally validator-less. USState's anchor regex already does the disambiguation; USAddress's `\d+ ... <suffix>` structure is distinctive enough.

### Anti-Patterns to Avoid

- **Bare two-letter state alternation without context anchor.** A regex like `\b(?:AL|AK|...|WY)\b` will fire on every English-language `OR`, `IN`, `OK`, `HI`, `ME`, `MD`, `MA`. The recognizer must require a preceding `,` or newline OR a trailing ZIP. The proposed `usStateRe` uses `(?:,\s+|^|\n)` as the lead anchor and `(?:\b|\s+\d{5}|\.|,)` as the trailing anchor.
- **Greedy USAddress that swallows City + ST + ZIP.** Each entity should emit its own token. Reasons: (1) referential-identity counter (`[USZIP_1]` reused across the prompt for the same ZIP); (2) operator-side `PII_ENTITY_ACTIONS` granularity (encrypt the street, mask the ZIP, drop the state — impossible if it's one blob); (3) the existing model where `IPv4` and `IPv6` and `COORDINATES` all emit separately is the precedent.
- **Hand-rolled `strings.Contains` loops to validate state codes.** RE2 alternation is the standard pattern for this; it's optimized internally and reads cleanly.
- **Adding a validator to USState.** The context anchor IS the validator. Adding a 55-entry hash-set check on top would double-gate the same constraint.
- **Forgetting `piiAllowedEntities` in `config/config.go:720`.** The new names must be in this whitelist or `PII_ENABLED_ENTITIES=USAddress` causes a boot error.
- **Forgetting `TestRecognizers_RegistryShape` update at `recognizers_test.go:489`.** This test asserts the exact 13-name list and exact count of 13. New entries must be added or the test breaks.

## Common Pitfalls

### Pitfall 1: USState alternation matches English words

**What goes wrong:** `\b(?:AL|...|OR|IN|HI|OK|ME|MD|...)\b` fires on "I think it might be OR something", "Hi there!", "ID is set to ..."

**Why it happens:** Two-letter state codes are also common English words / abbreviations. Word boundaries (`\b`) don't distinguish.

**How to avoid:** Anchor USState to `, ST` or `\nST` context. Proposed shape: `(?:,\s+|^|\n)([A-Z]{2})(?:\b|\s+\d{5}|\.|,)`. Operationally: a state code is preceded by a comma + space (after a city) OR sits at a line start, and is followed by either a word boundary + ZIP, a period, or a comma. The capturing group `([A-Z]{2})` should be inside an alternation containing all 55 codes. (Full alternation given below in Code Examples.)

**Warning signs:** A test fixture like `"I OR my colleague will handle it"` matching as a `[USState_1]`. Add this as a reject-invalid test case.

### Pitfall 2: USZIP matches 5-digit non-ZIP runs

**What goes wrong:** Bare `\b\d{5}\b` matches PIN codes, order numbers, year ranges, etc.

**Why it happens:** Five-digit numeric runs are not unique to ZIP codes.

**How to avoid:** Two layers — (a) the validator `validateUSZIPRange` rejects all-same-digit codes; (b) accept that the encrypt round-trip is the ONLY consumer for the recognizer's match, so a false positive on a non-PII 5-digit number becomes a tokenized number on the way to kiro-cli and a plaintext number on the way back. This is the same trade-off the IPv4 recognizer accepts (matches version numbers like `1.2.3.4`). The operator-trust model + `PII_ENTITY_ACTIONS` give per-entity opt-out.

**Warning signs:** A smoke-test fixture with `"order #54321"` getting `[USZIP_1]`. Document the trade-off in the requirement text rather than over-engineering the validator.

### Pitfall 3: USAddress's `<word>+` repetition is too greedy

**What goes wrong:** A naive `\d+\s+[A-Z][\w\s]+?\s+<suffix>` with `[\w\s]+?` will backtrack and produce odd captured spans, or worse, consume newlines and grab two streets.

**Why it happens:** RE2 doesn't backtrack (it's an NFA-based matcher), but the `[\w\s]+?` style still produces unintuitive captures because RE2 prefers left-most-then-longest. Also, `\s` matches `\n`.

**How to avoid:** Use `[A-Z][A-Za-z]*(?:\s+[A-Z][A-Za-z]*)*\s+(?:St|Street|...)` — a sequence of TitleCase words separated by single spaces, followed by the suffix vocabulary. Each word is `[A-Z][A-Za-z]*` (capital letter then letters; no digits in the street name, no underscores, no newlines, no tabs). This is the standard "street-name word" tokenization.

**Warning signs:** A test like `"Visit 1111 Main\nStreet today"` (with literal newline) matching `1111 Main\nStreet`. Reject this via the `[A-Za-z]*(?:\s+...)` shape (whitespace is single space or tab between TitleCase words; multi-line addresses are not supported v1).

### Pitfall 6: The `decryptTokenRe` letters-only bug regression

**What goes wrong:** Encrypt mode produces `[PII:USAddress:base64url...]` but the decrypt regex doesn't accept the entity name.

**Why it happens:** The historical bug was that `decryptTokenRe` used `[A-Za-z]+` for the entity name group, dropping numeric-suffix names like `IPv4` and `IPv6`. Fixed in v1.9.5 — see comment at `pii.go:336-339`.

**How to avoid:** The current regex is `\[PII:([A-Za-z0-9]+):([A-Za-z0-9_-]+)\]` — alphanumeric entity name group. The new names `USAddress`, `USState`, `USZIP` are all alphanumeric → no regex change needed. Verified by reading `pii.go:339`. The matching smoke-script regex at `test-pii.ps1:432` and `test-pii.sh:324` is also `[A-Za-z0-9]+` — also no change needed. `[VERIFIED: internal/plugin/pii/pii.go:339 + scripts/test-pii.ps1:432 + scripts/test-pii.sh:324]`

**Warning signs:** Decryption fails silently and ciphertext leaks to client. The "no ciphertext leak" assertion in the smoke test catches this. The Phase 8.4 plan should include a smoke-test assertion that the USAddress / USState / USZIP plaintext appears in the response (round-trip verified).

### Pitfall 7: Adjacent regex spans accidentally overlap due to context anchor consumption

**What goes wrong:** `usStateRe` consumes `, TX ` as part of its match (capture group `(TX)`), but if the regex match includes the leading comma+space, the capture span includes `, ` — and then `usAddressRe`'s trailing `, Austin` collides.

**Why it happens:** Anchored regex must distinguish "matches what we want to anchor against" from "captures what we want to tokenize". `FindAllStringIndex` returns the full match span, not the capture group span.

**How to avoid:** TWO options — (a) use a non-capturing assertion that RE2 doesn't have natively (lookbehind), so this isn't available; (b) accept the match span includes the lead context (`, TX`), and the resulting `[USState_1]` token replaces `, TX` with `, [USState_1]` (still correct, because the comma+space pairs are common boilerplate). Option (b) is the standard RE2 workaround.

**Concrete consequence:** `1111 Main Street, Austin, TX 27584` → `[USAddress_1], Austin, [USState_1] [USZIP_1]`. The leading `, ` in front of `[USState_1]` comes from `usStateRe` consuming the anchor as part of its match. Acceptable — round-trips byte-for-byte through encrypt because the comma is OUTSIDE the encrypted blob.

**Better alternative — use `FindAllStringSubmatchIndex` and key off the capture group:** This would let `usStateRe` capture only `TX` while requiring `, ` as a lookahead-style anchor. BUT this requires a code change in `collectRegexSpans` (currently uses `FindAllStringIndex`, not `FindAllStringSubmatchIndex`). **Warning signs:** Smoke-test fixture `"1111 Main Street, Austin, TX 27584"` round-trips as `1111 Main Street, Austin, TX 27584` (the comma is preserved by encrypt-decrypt because it's outside the token; the token replaces the regex-matched bytes verbatim on encrypt, and decrypt replaces them with the captured-original bytes). ### Three new regex literals + one validator (proposed)

```go
// Source: internal/plugin/pii/recognizers.go (Phase 8.4 additions)
//
// Append to the var ( ... ) block at recognizers.go:80-145, after siteRe.

var (
    // usZIPRe — US ZIP code: 5-digit base, optional ZIP+4 extension.
    // validateUSZIPRange rejects obvious non-ZIP shapes (all-same-digit
    // runs like 00000, 99999). Bare 5-digit runs that pass the validator
    // get tokenized — accepts the false-positive trade-off (a 5-digit
    // order number gets encrypted on the way to kiro-cli and decrypted
    // back on return).
    usZIPRe = regexp.MustCompile(`\b\d{5}(?:-\d{4})?\b`)

    // usStateRe — US state / territory two-letter code (all 50 states +
    // DC + 5 territories = 56 codes). Context-anchored: a state code
    // is preceded by ", " (after a city) or sits at line start, and is
    // followed by either a ZIP code, a period, or a comma.
    //
    // Without the context anchor this recognizer would fire on every
    // English-language "OR", "IN", "HI", "ME", "OK" etc. (Pitfall 1).
    // The leading ", " is consumed by the match span — acceptable per
    // Pitfall 7; the comma falls outside the encrypted blob and
    // round-trips byte-for-byte.
    //
    // Alternation list MUST be kept in sync with USPS state-code
    // assignments. As of 2026: 50 states + DC + AS + GU + MP + PR + VI.
    usStateRe = regexp.MustCompile(
        `(?:,\s+|^|\n)` +
        `(?:AL|AK|AZ|AR|CA|CO|CT|DE|DC|FL|GA|HI|ID|IL|IN|IA|KS|KY|LA|ME|` +
        `MD|MA|MI|MN|MS|MO|MT|NE|NV|NH|NJ|NM|NY|NC|ND|OH|OK|OR|PA|RI|SC|` +
        `SD|TN|TX|UT|VT|VA|WA|WV|WI|WY|AS|GU|MP|PR|VI)` +
        `(?:\b|\s+\d{5}|\.|,)`)

    // usAddressRe — US street address: house number + TitleCase word(s) +
    // street suffix from a controlled vocabulary. Captures the common
    // `<digits> <name> <suffix>` shape.
    //
    // Suffix vocabulary (16 forms covering full + standard USPS abbrev):
    //   St / Street, Ave / Avenue, Blvd / Boulevard, Rd / Road,
    //   Dr / Drive, Ln / Lane, Way, Pl / Place, Ct / Court,
    //   Pkwy / Parkway, Cir / Circle, Ter / Terrace, Sq / Square,
    //   Hwy / Highway.
    //
    // Trailing `\.?` accepts the period after abbreviated forms ("Ave.").
    //
    // Constraints:
    //   - House number 1-6 digits (\d{1,6}); empirically captures all
    //     US street numbers.
    //   - Street-name word is TitleCase letters only: [A-Z][A-Za-z]*.
    //     Multi-word names join via single space: (?:\s+[A-Z][A-Za-z]*)*.
    //   - No newlines inside the name (\s would match \n which is wrong
    //     for multi-line text — Pitfall 3).
    usAddressRe = regexp.MustCompile(
        `\b\d{1,6}\s+[A-Z][A-Za-z]*(?:\s+[A-Z][A-Za-z]*)*\s+` +
        `(?:St|Street|Ave|Avenue|Blvd|Boulevard|Rd|Road|Dr|Drive|Ln|Lane|` +
        `Way|Pl|Place|Ct|Court|Pkwy|Parkway|Cir|Circle|Ter|Terrace|Sq|` +
        `Square|Hwy|Highway)\b\.?`)
)

// validateUSZIPRange rejects obvious non-ZIP shapes the permissive
// \b\d{5}(?:-\d{4})?\b regex accepts. Rules:
//   - All-same-digit codes are rejected (00000, 11111, ..., 99999).
//     These are not valid USPS ZIP assignments.
//   - The validator inspects only the 5-digit base; the optional ZIP+4
//     extension is stripped before checking.
//   - We do NOT validate against the USPS first-digit region table;
//     new ZIPs are assigned occasionally and the table drifts.
func validateUSZIPRange(s string) bool {
    base := s
    if dash := strings.IndexByte(s, '-'); dash > 0 {
        base = s[:dash]
    }
    if len(base) != 5 {
        return false
    }
    allSame := true
    for i := 1; i < len(base); i++ {
        if base[i] != base[0] {
            allSame = false
            break
        }
    }
    return !allSame
}
```

## Open Questions

1. **REQ-ID neighborhood**
   - What we know: REQUIREMENTS.md has 14 section groupings; PII coverage is currently inside `### Plugins — Guardrails / hook chain` as PLUG-06.
   - What's unclear: Whether the planner should create a new `### PII — Recognizer coverage` section (with PII-01 for this work and a v2 placeholder for future address-PII expansion) OR continue the PLUG-* sequence with PLUG-07.
   - Recommendation: New section `### PII — Recognizer coverage` at PII-01. Rationale: the recognizer corpus is an axis distinct from the hook-chain mechanism (PLUG-* is about hook lifecycle / chain dispatch / short-circuit semantics; PII-* would be about which specific entities are detected). Future address-related work (international addresses, apartment parsing) lands cleanly in the new section. Planner: ratify in CONTEXT.md or override.

2. **USState match span includes leading `, ` — acceptable or refactor `collectRegexSpans` to use capture groups?**
   - What we know: The current code uses `FindAllStringIndex` which returns the full match (anchor + capture). Refactoring to `FindAllStringSubmatchIndex` would let the recognizer capture only `TX` and use `, ` as a (non-consuming) constraint.
   - What's unclear: Whether the trailing-`, ` artifact in the redacted text (e.g., `[USADDRESS_1], Austin, [USSTATE_1] [USZIP_1]`) is operator-acceptable, or whether the planner wants a Wave 0 task to refactor `collectRegexSpans` to support capture-group span extraction.
   - Recommendation: Accept the artifact. The pipeline refactor would touch every recognizer's contract and is a bigger change than the phase's stated goal. Document the artifact in the requirement text. If a future operator complains, revisit.

3. **Should `USAddress` also match `<num> <name>` without a suffix (e.g., bare "1600 Pennsylvania")?**
   - What we know: ROADMAP §SC 3 explicitly says "shape `\d+\s+[A-Z][\w\s]+?\s+(?:St|Street|...)`" — suffix-required.
   - What's unclear: Whether real PHI fixtures sometimes drop the suffix (e.g., "1600 Pennsylvania" alone).
   - Recommendation: Suffix-required for v1. Bare `<num> <name>` would be too generic and would false-positive on `<num> <city>` (e.g., "200 Boston" inside "200 Boston-area employees"). Defer to a follow-up phase if operators report misses.

4. **Should `validateUSZIPRange` use a stricter USPS region-table check?**
   - What we know: The first digit of a US ZIP encodes a region (0 = Northeast, 9 = West Coast). The full table is published by USPS.
   - What's unclear: Whether tightening this matters for the false-positive rate, or just adds maintenance burden when USPS reassigns.
   - Recommendation: Skip the region table. Reject only all-same-digit codes. Operators can opt out via `PII_ENABLED_ENTITIES` if false positives become a problem.

### Known Threat Patterns for Go regex + plugin pipeline

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| ReDoS (regex catastrophic backtracking) | Denial of Service | Go's `regexp` uses RE2 (linear-time NFA). Catastrophic backtracking is structurally impossible. No mitigation needed. `[VERIFIED: RE2 design documented at https://github.com/google/re2/wiki/WhyRE2 ; Go regexp implementation matches]` |
| Slopsquatted regex literal (copy-paste from untrusted source) | Tampering | All three regex literals reviewed inline in this RESEARCH.md; planner reviews the diff; reviewer reads the test fixtures. No external code introduced. |
| Validator function leaks PII to logs | Information Disclosure | `validateUSZIPRange` does not log. Other validators (e.g., `validateLuhn`) also don't log. Pattern preserved. |
| Encrypt-key reuse across new entity names | Cryptographic weakness | The existing AES-256-GCM encrypt uses AAD = entity name. New entity names get distinct AAD, so encrypted tokens for `USAddress` cannot be confused with tokens for `Email`. `[VERIFIED: encrypt.go uses entity as AAD per Phase 8 work]` |
| ZIP `00000`/`99999` as ciphertext bypass | Information Disclosure | The validator rejects these so they pass through plaintext to kiro-cli. This is a known and accepted trade-off — same model as IPv4 accepting `1.2.3.4` as a version number. Operators concerned about residual exposure can use `PII_ENABLED_ENTITIES` to disable the entity OR rely on PII_REDACTION_MODE=hash/drop. |
| New recognizer accidentally matches AUTH_TOKEN, KIRO_CMD, or other secrets | Information Disclosure | Verified: `usZIPRe` (5-digit), `usStateRe` (2-letter alpha), `usAddressRe` (digit-then-TitleCase-then-suffix) cannot match a bearer token shape (32+ char alphanumeric), command paths, or env var names. No mitigation needed beyond review. |
