#### A1. Init-time-compiled regex literal — analog `ssnRe` (`recognizers.go:84`) + `creditCardRe` (`recognizers.go:85`)

**Why this analog:** Both are simple `regexp.MustCompile` in the same `var (...)` block, both have a numeric-shape regex paired with a Go validator (`validateSSNRange`, `validateLuhn`). `usZIPRe` follows the exact same idiom.

**Exact pattern to copy** (`recognizers.go:80-100`, focus on the doc-comment style + the `MustCompile` invocation):

```go
//	ssnRe         — permissive 3-2-4 segment shape per RESEARCH Pitfall 1
//	                (RE2 has no negative lookahead); validateSSNRange
//	                rejects SSA-reserved ranges.
//	creditCardRe  — 13-19 digit runs with optional spaces/hyphens
//	                between groups; validateLuhn (luhn.go) confirms.
var (
    // ... existing literals ...
    ssnRe        = regexp.MustCompile(`\b[0-9]{3}-[0-9]{2}-[0-9]{4}\b`)
    creditCardRe = regexp.MustCompile(`\b(?:[0-9][ \-]?){12,18}[0-9]\b`)
)
```

**New code shape** (paste in same `var (...)` block, after `siteRe` at `recognizers.go:143-145`):

```go
// usZIPRe — US ZIP code: 5-digit base, optional ZIP+4 extension.
// validateUSZIPRange rejects all-same-digit shapes (00000, 11111, …,
// 99999) which the permissive regex would otherwise accept. False-
// positive trade-off documented in 08.4-RESEARCH §Pitfall 2: a 5-digit
// order number gets encrypted on the way to kiro-cli and decrypted
// back unchanged on return; same trade-off IPv4 already accepts.
usZIPRe = regexp.MustCompile(`\b\d{5}(?:-\d{4})?\b`)
```

#### A2. Context-anchored regex without ContextKeywords — analog `coordinatesRe` (`recognizers.go:121-129`)

**Why this analog:** `coordinatesRe` uses surrounding-letter context (`[NS]`, `[EW]`) to disambiguate a numeric shape from arbitrary decimals. `usStateRe` uses surrounding-comma/newline context (`(?:,\s+|^|\n)` lead + `(?:\b|\s+\d{5}|\.|,)` trail) to disambiguate two-letter state codes from English words. Both anchor INSIDE the regex (no ContextKeywords field needed).

**Exact pattern to copy** (`recognizers.go:121-129`):

```go
//	coordinatesRe — decimal-degrees lat/long with N/S and E/W
//	          hemisphere markers (e.g., "37.7749 N, 122.4194 W"). The
//	          optional °-sign and whitespace are tolerated. Context-
//	          free; the hemisphere letters are distinctive.
//
//	Word-boundary at start (\b) ensures the integer-degree portion
//	doesn't bleed into an adjacent number. The trailing E/W letter is
//	a word char so \b on the tail-end is implicit.
coordinatesRe = regexp.MustCompile(`\b\d{1,3}\.\d+\s*°?\s*[NS][,\s]+\d{1,3}\.\d+\s*°?\s*[EW]\b`)
```

**New code shape** (paste in same `var (...)` block — note multi-line string concatenation idiom mirrors `siteRe` at `recognizers.go:143-145`):

```go
// usStateRe — US state / DC / territory two-letter code (50 + DC + 5
// = 56 codes). Context-anchored INSIDE the regex (no ContextKeywords
// — same idiom as coordinatesRe's [NS]/[EW] hemisphere anchor). The
// leading "(?:,\s+|^|\n)" requires the code to follow ", " (after a
// city) OR sit at a line start. Without this anchor the recognizer
// would fire on every English-language "OR", "IN", "HI", "ME", "OK"
// (08.4-RESEARCH Pitfall 1). The leading ", " is consumed by the
// match span — acceptable per 08.4-RESEARCH Pitfall 7; the comma
// falls outside any encrypted blob and round-trips byte-for-byte.
usStateRe = regexp.MustCompile(
    `(?:,\s+|^|\n)` +
        `(?:AL|AK|AZ|AR|CA|CO|CT|DE|DC|FL|GA|HI|ID|IL|IN|IA|KS|KY|LA|ME|` +
        `MD|MA|MI|MN|MS|MO|MT|NE|NV|NH|NJ|NM|NY|NC|ND|OH|OK|OR|PA|RI|SC|` +
        `SD|TN|TX|UT|VT|VA|WA|WV|WI|WY|AS|GU|MP|PR|VI)` +
        `(?:\b|\s+\d{5}|\.|,)`)
```

#### A3. Alternation-with-vocabulary regex — analog `siteRe` (`recognizers.go:143-145`) + `creditCardRe` (`recognizers.go:85`)

**Why this analog:** `siteRe` uses two arms of literal-prefix alternation (`site[-_\s]?...` | `(?:ENB|BTS|NB|...)...`). `usAddressRe` uses the same idiom for street suffixes (`St|Street|Ave|Avenue|...`). Multi-line string concatenation idiom is identical.

**Exact pattern to copy** (`recognizers.go:143-145`):

```go
siteRe = regexp.MustCompile(
    `\bsite[-_\s]?[A-Z0-9]{1,2}[A-Z0-9_\-]{1,12}\b` +
        `|\b(?:ENB|BTS|NB|CELL|NODE|RAN|BSC|RNC|MSC|HLR|MME|SGW|PGW)[-_]?[A-Z0-9]{2,12}\b`)
```

**New code shape** (paste in same `var (...)` block, after `usStateRe`):

```go
// usAddressRe — US street address: 1-6 digit house number + one or
// more TitleCase street-name words + street suffix from a controlled
// vocabulary (16 forms: full + USPS-standard abbreviation). Trailing
// "\.?" accepts the period after abbreviated forms ("Ave.").
//
// Anti-Pattern AP-1 mitigation: a bare "<digits> <words>" without
// the suffix vocabulary would match phone-number-shaped strings,
// room numbers, table cells. The suffix list is load-bearing.
//
// House number 1–6 digits empirically captures every US street
// address. The street-name word is "[A-Z][A-Za-z]*" (capital then
// letters only — no digits, no underscores, no newlines per
// 08.4-RESEARCH Pitfall 3).
usAddressRe = regexp.MustCompile(
    `\b\d{1,6}\s+[A-Z][A-Za-z]*(?:\s+[A-Z][A-Za-z]*)*\s+` +
        `(?:St|Street|Ave|Avenue|Blvd|Boulevard|Rd|Road|Dr|Drive|Ln|Lane|` +
        `Way|Pl|Place|Ct|Court|Pkwy|Parkway|Cir|Circle|Ter|Terrace|Sq|` +
        `Square|Hwy|Highway)\b\.?`)
```

#### A4. Validator function — analog `validateSSNRange` (`recognizers.go:185-205`) + `validateLuhn` (`luhn.go:64-66`)

**Why this analog:** `validateSSNRange` is the closest structural match — string → bool, defensive length/shape check first, then domain-specific rejection rules. `validateLuhn` shows the closure-adapter idiom (a one-liner that defers to a more general `LuhnCheck`); we DON'T need that level of indirection for ZIP — inline the rule.

**Exact pattern to copy** (`recognizers.go:185-205`):

```go
// validateSSNRange rejects SSA reserved-range SSNs per the published
// SSA assignment table (RESEARCH Example 3):
//   - aaa == 000
//   - aaa == 666
//   - aaa starts with '9' (900-999 is reserved for ITIN/etc.)
//   - gg == 00
//   - ssss == 0000
//
// The regex shape guarantees aaa-gg-ssss segments of widths 3-2-4, so
// the parse is segment-position fixed.
func validateSSNRange(s string) bool {
    // Defensive segment-shape check (handles the unlikely case the
    // regex was overridden externally).
    if len(s) != 11 || s[3] != '-' || s[6] != '-' {
        return false
    }
    aaa := s[0:3]
    // ...
    return true
}
```

**New code shape** (paste alongside `validateSSNRange`, after `recognizers.go:205`):

```go
// validateUSZIPRange rejects obvious non-ZIP shapes the permissive
// \b\d{5}(?:-\d{4})?\b regex accepts. Rules:
//   - All-same-digit codes are rejected (00000, 11111, ..., 99999).
//     These are not valid USPS ZIP assignments.
//   - The validator inspects only the 5-digit base; the optional
//     ZIP+4 extension is stripped before checking.
//   - We do NOT validate against the USPS first-digit region table
//     (0 = Northeast, ..., 9 = West Coast / HI / AK) because new
//     ZIPs are assigned occasionally and the table drifts. The
//     regex is conservative enough at the shape level.
func validateUSZIPRange(s string) bool {
    base := s
    if dash := strings.IndexByte(s, '-'); dash > 0 {
        base = s[:dash]
    }
    if len(base) != 5 {
        return false
    }
    allSame := true
    for i := 1; i < len(base); i++ {
        if base[i] != base[0] {
            allSame = false
            break
        }
    }
    return !allSame
}
```

## Shared Patterns

### Cross-cutting: RE2 limitation respect

**Source:** `recognizers.go:64-78` (top-of-var docstring listing RE2 workarounds)
**Apply to:** All three new regex literals in §A.

The RE2 stdlib regexp has no lookbehind, no backreferences, no negative lookahead. Each new regex MUST either (a) accept the leading-anchor consumption (Pitfall 7 — `usStateRe`), or (b) push the disambiguation into a validator (`validateUSZIPRange`), or (c) make the regex shape distinctive enough that no validator is needed (`usAddressRe`'s suffix vocabulary). DO NOT use `(?=...)` or `(?!...)` — RE2 supports `(?=...)` only in lookahead position with bounded length and the package style is to avoid it (no existing recognizer uses it).

### Cross-cutting: init-time MustCompile + source-level guard

**Source:** `recognizers.go:80` (`var (...)` block) + `recognizers_test.go:514-541` (`TestRecognizers_CompiledAtPackageInit_NoPerRequestCompile`)
**Apply to:** All three new regex literals.

Use `regexp.MustCompile` (not `regexp.Compile`). The existing source-level test greps `recognizers.go` for `regexp.MustCompile(` and counts unique patterns. Adding three new MustCompile calls satisfies the existing assertion automatically — **no edit needed** to that test.

### Cross-cutting: ASCII-only smoke-script invariant

**Source:** Project memory `feedback-pii-bracket-shape.md` + commit `f7ccd40` (UTF-8 BOM + ASCII-only chars for PowerShell 5.1)
**Apply to:** §E + §F edits.

The PS1 and SH scripts must use only ASCII characters. No smart quotes, no em-dashes, no `<...>` markers (kiro-cli treats `<...>` as an XML tag and hangs). Use `[...]` for any placeholder.

### Cross-cutting: PS1/SH parity gate

**Source:** STATE.md drift incident `f7ccd40`
**Apply to:** §E + §F edits land in the SAME plan (one task), not two.

PS1 and SH are intentionally parallel; drift between them was a real incident documented in STATE.md. The planner should put both file edits under one task so a partial commit can't ship.
