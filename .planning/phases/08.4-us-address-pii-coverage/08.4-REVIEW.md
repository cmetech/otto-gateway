# Phase 08.4: Code Review Report

**Reviewed:** 2026-06-04T00:00:00Z
**Depth:** standard
**Files Reviewed:** 7
**Status:** issues_found

## Summary

Phase 08.4 introduces three new regex PII recognizers (USAddress, USState, USZIP) and a validator (validateUSZIPRange) into the OTTO Gateway redactor. Registration order, regex shapes, and the validator generally match what the plan calls for, and the integration-level test (TestPIIRedactionHook_USAddressFullCoverage) does pin the canonical address fixture end to end. Unit tests cover the captured-span and rejection cases described in the phase context.

However, the adversarial pass surfaced two BLOCKER-class defects that fall directly inside the "Pay attention to" list in the phase prompt:

1. The USState recognizer's AP-2 "must not match English words" mitigation is incomplete. Arm 1 (comma-prefixed) accepts the word boundary `\b` as a valid trail with no ZIP requirement, so every sentence that puts the words `OR`, `IN`, `OK`, `HI`, `ME`, `ID`, `OH`, `AL`, `MS`, etc. after a comma matches as a US state code. Only arm 2 (line-start) actually requires the `\s+\d{5}` ZIP-trail anchor that the docstring says is "load-bearing" against English-word collisions. The existing AP-2 test (`TestUSStateRecognizer_RejectsEnglishWords`) only covers cases without a leading comma, so it passes despite the bug.
2. The same USState arm 1 uses `\s+` between the comma and the state code. Per the explicit "Pitfall 3" warning the file itself cites for usAddressRe (`[ \t]+` not `\s+` to avoid newline smuggling), this `\s+` allows the comma and the state code to be split by a newline. Bulleted lists with comma-terminated items immediately followed by an English-word continuation (`",\n  OR"`, `",\n  IN"`) match as a USState span.

A third BLOCKER is the script ASCII-only invariant called out in the phase prompt: `scripts/test-pii.sh` contains many em-dashes (U+2014) and right-arrows (U+2192) in comments and `ok`/`fail` text. The Windows-parity PowerShell script (`test-pii.ps1`) was visibly de-Unicoded but the POSIX twin was not — this is the literal regression the f7ccd40 invariant exists to prevent.

A fourth BLOCKER concerns `validateUSZIPRange`'s dash-handling, which fails open on inputs the regex shouldn't be able to deliver but the code path doesn't defend against.

Lower-severity findings include: the `IndexByte` "dash > 0" guard in the ZIP validator is misleading (5-digit base is required so dash should be == 5 to be valid, not > 0); the test fixtures don't exercise the false positives surfaced above; and the docstring for the USState recognizer claims arm 1's trail anchor "MUST" include a ZIP requirement, but the implementation accepts plain `\b`.

## Critical Issues

### CR-01: USState arm 1 lacks AP-2 mitigation — common English words after comma match as state codes (BLOCKER)

**File:** `internal/plugin/pii/recognizers.go:177-190`
**Issue:** The docstring at lines 162-173 promises that the regex anchors against English-word collisions by requiring "ZIP trail" anchoring, citing the AP-2 mitigation. Arm 2 (line-start) does enforce this with `\s+\d{5}`. Arm 1 (comma-prefixed) does NOT — its trail alternation is `(?:\b|\s+\d{5}|\.|,)`, where the leading `\b` (zero-width word boundary) is satisfied at the very next character after any two-letter state code. The result: ordinary English prose collides.

Reproduced (RE2):
```
"Hello, OK then"                   ->  matches ", OK"
"I want this, OR that"             ->  matches ", OR"
"Submitted, IN summary"            ->  matches ", IN"
"Send to me, ME tomorrow"          ->  matches ", ME"
"Hi, ID this please"               ->  matches ", ID"
```

The existing `TestUSStateRecognizer_RejectsEnglishWords` (recognizers_test.go:635-657) only tests cases where the English word has NO preceding comma+space, so it passes despite the bug. The full-coverage integration test (`TestPIIRedactionHook_USAddressFullCoverage`) doesn't probe this surface either.

Operational impact in encrypt mode: ordinary user prose like "Yes, OR no?" gets a substring encrypted as a USState span — the LLM never sees those four bytes, and on decrypt the round-tripped text is identical only if the LLM echoes the cipher token verbatim. If the LLM does not echo it, the decrypt path leaves the cipher token in the response, which the test scripts' `[PII:Entity:base64url]` leak detector will flag as a failure. In replace mode, ", OR" becomes "[USState_N]" — the request that reaches kiro-cli is semantically wrong.

**Fix:** Either remove the bare `\b` arm and require a ZIP trail on the comma-prefixed arm too (mirrors arm 2), or extend the ContextKeywords / AP-2 test to actually exercise comma-prefixed two-letter English words and tighten the trail. Suggested regex change:
```go
usStateRe = regexp.MustCompile(
    `(?:,[ \t]+` +
        `(?:AL|AK|AZ|...|VI)` +
        `(?:[ \t]+\d{5}|\.|,))` +     // drop bare \b — REQUIRE ZIP / period / comma
        `|(?:(?:^|\n)` +
        `(?:AL|AK|AZ|...|VI)` +
        `[ \t]+\d{5})`,
)
```
Then add fixtures to `TestUSStateRecognizer_RejectsEnglishWords` like `"Yes, OR no"`, `"Submitted, IN summary"`, `"Send to me, ME tomorrow"`, `"OK, that works"` (with preceding comma) — these must NOT match.

### CR-02: USState `\s+` admits newline smuggling between comma and state code (BLOCKER, Pitfall 3 violation)

**File:** `internal/plugin/pii/recognizers.go:179` (arm 1) and `:189` (arm 2)
**Issue:** The recognizers.go file at lines 202-204 calls out RE2 Pitfall 3 for usAddressRe: "Whitespace between tokens is `[ \t]+` (NOT `\s+`) — RE2's `\s` includes newlines, which would let multi-line text smuggle into a single address span." The same hazard exists in usStateRe arm 1 (`,\s+`) and arm 2 (`\s+\d{5}`), but the recognizer here uses `\s+`.

Reproduced (RE2) — given a bulleted list:
```
Please pick:
- Option A,
  OR Option B,
  IN that case
```
The regex returns matches `[",\n  OR", ",\n  IN"]`. The match span includes the literal newline.

When fed to the redactor in replace mode the span ", \n  OR" (or its byte equivalent) becomes `[USState_N]` — collapsing a list-item boundary into a single token and changing layout the LLM sees. In encrypt mode the captured span (including the newline) is what is encrypted; the rewriter inserts the ciphertext token, and on decrypt the newline is restored verbatim only if the LLM did not strip surrounding whitespace.

**Fix:** Replace `\s+` with `[ \t]+` in both arms. Update the arm-2 line-start guard so it explicitly uses `[ \t]+`:
```go
usStateRe = regexp.MustCompile(
    `(?:,[ \t]+` +
        `(?:AL|AK|...)` +
        `(?:[ \t]+\d{5}|\.|,))` +
        `|(?:(?:^|\n)` +
        `(?:AL|AK|...)` +
        `[ \t]+\d{5})`,
)
```
Add a fixture to `TestUSStateRecognizer_RejectsEnglishWords` that pins multi-line lists with comma-terminated items followed by a two-letter English word continuation.

## Blockers (additional)

### BL-03: scripts/test-pii.sh violates the ASCII-only script invariant from f7ccd40 (BLOCKER)

**File:** `scripts/test-pii.sh` (multiple lines: 2, 6, 8, 14, 125, 128, 148, 161, 199, 200, 223, 224, 229, 248, 258, 267, 321, 324, 336, 354, 359)
**Issue:** The phase prompt and commit f7ccd40 establish that PowerShell and POSIX test scripts must be ASCII-only (no em-dashes, no smart quotes, no UTF-8 BOM). `file scripts/test-pii.sh` reports `Unicode text, UTF-8 text`; the matching `.ps1` reports `ASCII text`. The non-ASCII bytes are em-dashes (U+2014, `—`) in the comment block and right-arrows (U+2192, `→`) in `ok`/`fail` messages. Examples:

```
scripts/test-pii.sh:2:# scripts/test-pii.sh — quick PII / streaming smoke test ...
scripts/test-pii.sh:199:    ${want_stream_ct}*) ok "stream=true → Content-Type: $ct" ;;
scripts/test-pii.sh:223:    ${want_nonstream_ct}*) ok "stream=false → Content-Type: $ct2" ;;
scripts/test-pii.sh:359:    section "PII round-trip — encrypt → worker → decrypt → client"
```

The `→` arrows render badly under stock Windows PowerShell hosts on a non-UTF-8 console codepage and under POSIX terminals that don't honor `$LANG` (CI containers, ssh into busybox), producing mojibake in the operator-visible PASS/FAIL output. The PowerShell twin has been visibly converted (e.g., `"encrypt -> worker -> decrypt -> client"`), but the SH twin has drifted.

**Fix:** Replace every em-dash with `--` and every right-arrow with `->`. After the change, `file scripts/test-pii.sh` must report `ASCII text` and `LC_ALL=C grep -nP '[^\x00-\x7F]' scripts/test-pii.sh` must return nothing.

### BL-04: validateUSZIPRange's "dash > 0" guard masks malformed inputs and accepts a wrong base when `len(s) > 5` with no dash

**File:** `internal/plugin/pii/recognizers.go:284-300`
**Issue:** Two related problems:

1. The validator strips the ZIP+4 extension via `if dash := strings.IndexByte(s, '-'); dash > 0`. The regex shape guarantees `\d{5}(?:-\d{4})?` so a legitimate dashed input always has the dash at offset 5. Using `> 0` rather than `== 5` is "defensive in the wrong direction": if the regex were ever overridden externally (the same scenario the SSN validator's comment explicitly defends against at line 252) and produced `"00000-12345-6789"`-shaped input, the dash at offset 5 would still trigger the strip — but a leading dash at offset 0 silently falls through (good), while a stray dash at any other offset (e.g., `"12345-6"`) gets accepted because `len(base) == 5` after the strip and the `allSame` rule still holds for `12345`.
2. More importantly, `len(base) != 5` returns `false` for both the genuinely-malformed case AND the case where the regex was overridden to allow a 6-9 digit run without a dash. The validator should also explicitly reject `len(base) != 5` when there is no dash separator, which it does, but it does NOT reject the case where the input has multiple dashes (e.g., `"12345-1234-1234"` — `IndexByte` returns the first dash at 5, base becomes `"12345"`, validator accepts).

This is a latent bug today because the regex shape is tight, but the SSN validator's "defensive segment-shape check" comment at line 252 establishes the convention that validators should not trust the regex; this validator violates that convention.

**Fix:**
```go
func validateUSZIPRange(s string) bool {
    base := s
    switch {
    case len(s) == 5:
        // bare 5-digit, no dash — base = s
    case len(s) == 10 && s[5] == '-':
        base = s[:5]
    default:
        return false
    }
    for i := 1; i < len(base); i++ {
        if base[i] != base[0] {
            return true
        }
    }
    return false
}
```

Add fixtures to `TestUSZIPRecognizer_ValidatorRejectsAllSameDigit` for the off-shape inputs (`"12345-1234-1234"`, `"-12345"`, `"123-45"`, `"123456"`) and assert they all return `wantValidate: false`.

## Warnings

### WR-01: USAddress regex pattern misses common real-world street names (silent under-redaction)

**File:** `internal/plugin/pii/recognizers.go:205-210`
**Issue:** The street-suffix vocabulary at line 207-209 is fixed at 16 forms. Adversarial probing turned up three classes of common real address shapes that silently fall through with NO recognizer firing:

- Apostrophes and hyphens in street names: `"100 St John's Way today."` and `"100 O'Hara Drive today."` both fail because `[A-Z][A-Za-z]*` excludes `'` and `-`. Real PII (e.g., `"100 St. John's Cathedral Way"`) walks through unredacted.
- Ordinals: `"100 42nd Street today."` fails because the name token `[A-Z][A-Za-z]*` excludes digits — but `42nd Street`, `5th Avenue`, `21st Place` are extremely common. Plan 08.4 explicitly targets US street address coverage; this is a near-100% miss for any city with numbered streets (NYC, DC).
- Missing suffixes: `Trail`, `Loop`, `Walk`, `Run`, `Crossing`, `Park`, `Plaza` are all USPS-standard suffixes per Pub 28; the current vocabulary omits them. `"100 Forest Trail"` is not redacted.

These are warnings, not blockers, because Phase 08.4 documents partial coverage as acceptable and offers NER as the fallback for prose addresses. But the LOCATION NER recognizer is now disabled by default (config.go:351 `PII_NER_ENABLED=false`), so the regex IS the only address recognizer in the default deployment. The recall floor is lower than the plan implies.

**Fix:** Either expand the suffix vocabulary to match Pub 28 and allow optional digits / apostrophes / hyphens inside the name segment, OR re-enable NER LOCATION by default and accept the prior PERSON false positives. At minimum, document the recall floor in the docstring above usAddressRe so future debugging doesn't loop back to "I thought we covered street addresses".

### WR-02: Test fixture `TestUSStateRecognizer_RejectsEnglishWords` does not actually exercise the AP-2 surface

**File:** `internal/plugin/pii/recognizers_test.go:635-657`
**Issue:** Every fixture in this test omits the comma-prefix that arm 1 requires. They probe arm 2 (line-start), where the regex correctly demands `\s+\d{5}` — but they never probe the realistic case where the operator writes "Yes, OR no?". As a result, the test passes even though CR-01 above shows arm 1 matches mid-sentence English words.

This is a test-coverage hole: the load-bearing AP-2 mitigation has no negative test that exercises arm 1.

**Fix:** Add comma-prefixed negative cases:
```go
{"comma-or", "Should we ship today, OR wait until next week?"},
{"comma-in", "I keep my keys, IN my pocket."},
{"comma-me", "Send to me, ME, tomorrow."},
{"comma-ok", "Yes that works, OK go ahead."},
{"comma-id", "Look here, ID is not enabled."},
```

After CR-01 is fixed, these must all `FindString == ""`.

### WR-03: USState arm 1 trail anchor list comment / implementation drift

**File:** `internal/plugin/pii/recognizers.go:162-173, 178-183`
**Issue:** The docstring at lines 162-173 documents arm 1's trail as "Trail is `\b` / `\s+\d{5}` / `.` / `,`" and then explicitly states "Without arm-2's strict ZIP requirement, every line-start 'OK,' would tokenize as a USState". The implementation at line 183 says `(?:\b|\s+\d{5}|\.|,)` — so the `\b` alternative IS present in arm 1. This is internally inconsistent: arm 2 is described as "strict" because it requires ZIP, but arm 1's bare `\b` undermines the strictness.

Either the docstring is wrong about what was implemented, or the implementation regressed against the design intent. Combined with CR-01, this suggests the regex's design intent was sound but the implementation drifted between plan and code.

**Fix:** Resolve in lockstep with CR-01. Update the docstring to either (a) document that arm 1 deliberately accepts non-ZIP trails because the leading `, ` is itself the AP-2 anchor (and then add positive tests showing this is safe), OR (b) tighten the regex so the docstring's "ZIP trail" claim is true.

### WR-04: USPhone area-code first digit `[2-9]` rejects legitimate NANP regions starting with 1 — and the test doesn't notice

**File:** `internal/plugin/pii/recognizers.go:100`
**Issue:** The USPhone regex hard-codes `[2-9]` as the area-code first digit. The TestUSPhoneRecognizer_CapturedSpan fixture at recognizers_test.go:260-261 uses `"1-242-442-2424"` (Bahamas, NANP) which is captured as `"1-242-442-2424"` — but the `242` area code starts with `2` so this works. There are legitimate NANP-style placements (toll-free `800/888/...`) where the area code starts with `8` and is fine, BUT older operator-entered numbers in formats `"(0xx) xxx-xxxx"` from international shapes fall through.

This is technically pre-existing — not part of Phase 08.4 — but I am flagging it because the phase touches surrounding regex code and may regress against any future expansion.

**Fix:** No action recommended in Phase 08.4 (out of scope), but a one-line follow-up plan ticket: relax `[2-9]` to `[0-9]` or `[1-9]` and add fixtures for `800`/`888` toll-free.

## Info

### IN-01: Order-and-comment drift in `Recognizers` slice — comment claims order is "load-bearing" but downstream arbiter is by-iteration-order, not by-span-size

**File:** `internal/plugin/pii/recognizers.go:350-357`
**Issue:** The comment block at lines 350-354 says "Order: largest span first (USAddress) → context-anchored alphabet alternation (USState) → 5-digit numeric (USZIP). First-recognizer-wins overlap arbitration (pii.go:227-233) gives the largest match priority on the rare overlap case."

That's only true because the recognizer ordering HAPPENS to put the largest-span recognizer first. The arbiter at pii.go:227-233 does NOT prefer larger spans; it prefers earlier iteration order. If a future maintainer reorders this slice for any reason (e.g., adding USAddress2), the "largest first" property silently disappears. Make the property explicit in code rather than relying on slice-ordering discipline. At minimum, add a unit test that asserts the registration index of USAddress is less than the registration index of USZIP.

**Fix:** Add a test like:
```go
func TestRecognizers_USAddressBeforeUSZIP(t *testing.T) {
    var addrIdx, zipIdx = -1, -1
    for i, r := range Recognizers {
        switch r.Name {
        case "USAddress":
            addrIdx = i
        case "USZIP":
            zipIdx = i
        }
    }
    if addrIdx < 0 || zipIdx < 0 || addrIdx >= zipIdx {
        t.Fatalf("USAddress (%d) must precede USZIP (%d) so the overlap arbiter at pii.go:227-233 lets USAddress win on house-number ambiguity", addrIdx, zipIdx)
    }
}
```
