## Info

### IN-NEW-01: recognizers.go docstring lists `"Pick A, OR 27584 stuff"` ambiguously — could be misread as a fixed case

**File:** `internal/plugin/pii/recognizers.go:177`
**Issue:** The CR-NEW-01 doc block lists four shapes the prior `\.|,` alternates admitted:

```
"Yes, OK.", "Pick A, OR.", "Look, OK, here.", "Pick A, OR 27584 stuff" all
matched as USState spans.
```

The first three are now rejected (the `\.|,` alternates were dropped). The fourth — `"Pick A, OR 27584 stuff"` — **still matches** because `, OR 27584` satisfies the post-fix `[ \t]+\d{5}` trail requirement. A reader who skims this paragraph would reasonably conclude all four cases were fixed by the trail tightening; they were not. The test file at recognizers_test.go:727-734 correctly characterizes this as INHERENTLY ambiguous and unfixable in RE2, but that nuance is absent from the source-file docstring.

**Fix:** Split the docstring example list. Keep `"Yes, OK."`, `"Pick A, OR."`, `"Look, OK, here."` as the cases the fix closed; move `"Pick A, OR 27584 stuff"` to a separate paragraph noting it as the AP-2 residual that RE2 cannot mitigate. Suggested phrasing for the addition:

```go
// AP-2 residual (not fixed -- inherently ambiguous in RE2): the shape
// "<prose>, <2-letter-English-word>[ \t]+<5-digit-quantity>" still
// matches arm 1 because the structural trail `[ \t]+\d{5}` cannot
// distinguish ", OR 27584 stuff" from ", TX 27584 ASAP" without a
// city-name prefix lookbehind, which RE2 lacks. Accepted as a known
// AP-2 miss; LOCATION NER + downstream review is the catch-all when
// enabled. See recognizers_test.go:727-734 for the test-suite
// documentation of this trade-off.
```

This makes the source-file doc match the test-file doc, and ensures a future operator triaging a `", OR 27584"` false positive can find the design decision without spelunking the planning artifacts.

### IN-NEW-02: usAddressRe docstring self-contradicts on NER default in one sentence

**File:** `internal/plugin/pii/recognizers.go:249-250`
**Issue:** The PRECISION-OVER-RECALL paragraph for the suffix revert says, verbatim:

> They fall to LOCATION NER when enabled (default in production); when NER is disabled (config.go:351 default), recall on these shapes is best-effort.

The two clauses contradict each other. The parenthetical "(default in production)" claims NER is enabled by default, then the next clause correctly says "(config.go:351 default)" is disabled. Reading config.go:351 confirms: `piiNEREnabled, err := getEnvBool("PII_NER_ENABLED", false)` — the binary default IS disabled. There is no separate "production" config path that enables it; the env var is the only knob and its default is false.

This is a holdover from the original Phase-08 docstring drafted when NER was on-by-default before the 2026-06-03 flip documented at config.go:339-350. Fix: drop the "(default in production)" parenthetical entirely, since it's incorrect and the next clause already covers the true default state.

**Fix:**

```go
// PRECISION-OVER-RECALL trade-off: real-world addresses ending in
// Trail, Loop, Walk, Run, Crossing now miss the regex. They fall to
// LOCATION NER when PII_NER_ENABLED=true; when NER is disabled
// (config.go:351 default), recall on these shapes is best-effort.
```

This is a one-line cleanup; not behavior-affecting.
