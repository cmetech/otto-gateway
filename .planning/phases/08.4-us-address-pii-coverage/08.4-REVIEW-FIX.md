## Fixed Issues

### CR-NEW-01: USState arm 1 trail alternation `\.|,` still matches common English prose

**Files modified:** `internal/plugin/pii/recognizers.go`, `internal/plugin/pii/recognizers_test.go`
**Commit:** `b040ef4`
**Applied fix:** Tightened `usStateRe` arm 1's trail from `(?:[ \t]+\d{5}|\.|,)` to just `[ \t]+\d{5}`. Arm 1 now REQUIRES the structural ZIP trail, same as arm 2. Dropped the `\.|,` alternates that let comma-prefixed English-word two-letter codes followed by terminal punctuation match as USState spans (`"Yes, OK."` → `", OK."`, `"Pick A, OR."` → `", OR."`, etc.). Added 6 negative fixtures pinning the punctuation-trail shapes the prior alternates admitted. Documented the trade-off: sentence-end state codes WITHOUT a ZIP no longer match — but standalone state codes are inherently ambiguous with English words and are intentional misses; LOCATION NER is the catch-all. Canonical address fixture (`"Send a tech to 1111 Main Street, Austin, TX 27584 ASAP."`) still passes — `, TX 27584` matches because the ZIP trail is present.

**Known AP-2 residual (documented in test comment, NOT a regression of this fix):** The shape `", OR 27584 stuff"` (English-word two-letter code + 5-digit quantity) is INHERENTLY ambiguous with a real address `", TX 27584 ASAP"` under RE2 — there is no structural signal that distinguishes them without a city-name prefix lookbehind, which RE2 lacks. The review's suggested fixture list included three such shapes (`comma-or-digits`, `comma-in-digits`, `comma-ok-digits`); those fixtures CANNOT pass against any RE2 regex that admits the canonical address shape, so they were intentionally not added. The 6 punctuation fixtures (the clear AP-2 wins) are the realistic test coverage for this fix.

## Skipped Issues

### WR-NEW-03: USState span subsumes USZIP — `Summary.Counts()[USZIP]` traceability claim weakened

**Files referenced:** `internal/plugin/pii/recognizers.go:189-204`, `internal/plugin/pii/pii_test.go:1031-1091`
**Status:** `accepted_deviation`
**Commit:** `e1f156d` (documentation-only — updated test docstring and assertion-error message to call out the deliberate intent)
**Reason for skipping the structural change:** The review itself anticipated this outcome:
> If iteration 3 picks the strict-ZIP-only arm-1 fix proposed in CR-NEW-01, the subsuming behavior becomes the only correct shape, and this finding folds into 'documented intended behavior'.

CR-NEW-01's fix landed exactly that strict-ZIP-only arm 1 (no punctuation alternates), so the USState span always captures the ZIP on the canonical `City, ST ZIP` shape, and USZIP is structurally guaranteed to be 0 on that shape. Restructuring `usStateRe` to capture only `, ST` and let USZIP fire separately would require either:

1. A capture-group split + a Recognizer-struct change to surface a sub-span (load-bearing for the whole redact pipeline — out of scope for a review-fix iteration), or
2. A `(?=...)` lookahead which RE2 does not support.

Per the prompt's guidance ("Do NOT do a risky last-minute refactor of the Recognizer struct or overlap arbiter — those are load-bearing for the whole pipeline"), the accepted-deviation path is the right call.

**Operational impact documented in code:** The `TestPIIRedactionHook_USAddressFullCoverage` docstring now explicitly notes:
- Operators auditing gross ZIP volume must sum `USAddress + USState + USZIP` counts.
- Per-entity ZIP routing (`PII_ENTITY_ACTIONS=USZIP:encrypt`) applies only to standalone ZIPs; ZIPs adjacent to state codes inherit the USState action.

A future phase that needs per-entity ZIP routing on ZIPs-inside-states can revisit by either (a) refactoring `Recognizer` to expose multi-capture or (b) post-processing the USState span at the action-application stage. Both are explicit scope decisions, not silent regressions.
