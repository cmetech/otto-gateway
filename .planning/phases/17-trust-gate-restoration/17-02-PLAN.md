<objective>
Fix the goleak flake in `TestRegression_REL_POOL_02_CtrlCOrphansChildren`
(`internal/pool/regression_rel_pool_02_test.go`) so it passes 20/20 iterations
under `-race`. The test currently fails ~1/8 runs because two orphan goroutines
spawned at line 109 — `go func() { _, _ = stream.Result() }()` — are NOT
tracked by the outer `wg` and are sometimes still inside `stream.Result()`
when the deferred `goleak.VerifyNone(t)` at line 63 fires.

Per D-17-04, the root cause is a race between two parallel close paths in the
test scaffolding:
1. `p.Close()` at line 132 cancels every slot's client context, which causes
   each `blockingPromptClient`'s goroutine (lines 33-39) to take the
   `<-ctx.Done()` branch and call `stream.CloseForTest(nil, ctx.Err())`.
2. `close(bc0.gate); close(bc1.gate)` at lines 164-165 unblocks the
   `<-c.gate` branch of the same select, which calls
   `stream.CloseForTest(&acp.FinalResult{...}, nil)`.

Whichever fires first wins; the loser races to call `CloseForTest` on an
already-closed stream. The orphan `stream.Result()` goroutines block on the
stream's done channel and only return once `CloseForTest` lands — and under
the race they sometimes return AFTER goleak's check window opens.

Per D-17-04's 3-iteration budget, this plan investigates fixes in order of
expected effectiveness and stops at the first one that produces 20/20 green:

1. **Iteration 1 (cheapest):** Add a dedicated `resultWg` that the orphan
   `stream.Result()` goroutines join; wait on it BEFORE allowing the deferred
   goleak check to run.
2. **Iteration 2 (if 1 still flakes):** Reorder cleanup — close the
   blocking-client gates BEFORE `p.Close()` so streams close via the
   gate-path (deterministic `FinalResult`) instead of racing with
   Cancel-via-ctx.Done.
3. **Iteration 3 (if 2 still flakes):** Add explicit `<-stream.Done()` waits
   inside the resultWg goroutines so they don't return until the underlying
   `acp.Stream` has fully transitioned to closed state.

Per D-17-05, scope is strictly bounded to test-scaffolding edits in
`regression_rel_pool_02_test.go`. Do NOT modify production code under
`internal/pool/pool.go`, `internal/acp/stream.go`, or anywhere else. If after
3 iterations the test still flakes, STOP and surface to the operator —
per the D-17-04 risk-mitigation row, escalation options are (a) test
redesign or (b) a documented goleak-suppression annotation with a known-flake
follow-up. Do NOT take either escalation path unilaterally.

Per D-17-05, this is a test-hygiene fix; production REL-POOL-02 behavior
(pool.Close cancels in-flight sessions) is already in place from Phase 15-01
and is asserted by the test at lines 138-159 — those assertions stay.
