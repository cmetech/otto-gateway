# Phase 6.1: Admin Observability UI - Context

**Gathered:** 2026-05-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Lean, dark-mode admin page served at `GET /admin` that renders the current `/health` + `/health/agents` JSON in a styled HTML UI using the OTTO brand palette. Self-refreshes on a fixed 30-second interval via client-side polling so operators see live pool/session/process state without manual refresh. Pure consumer of existing observability endpoints — zero new gateway request-path logic, no new ACP plumbing, no impact on the engine, pool, or session registry.

Includes a live log tail panel that streams `/tmp/otto-gateway.log` lines into the UI via SSE without holding an exclusive file handle and without modifying any existing log-writing path.

Single static binary preserved: templates + CSS + JS shipped as compiled-in assets via `embed.FS`; no node toolchain, no cgo, cross-compile with `CGO_ENABLED=0` still works.

**This phase supersedes** the PROJECT.md "Out of Scope" entry that previously said "A web UI / status dashboard — `/health` JSON is enough for v1." The scope cut from 2026-05-27 (Phase 7 / embeddings) freed budget for this slice.

</domain>

<decisions>
## Implementation Decisions

### Auth posture (v1 only)
- **D-01:** No auth on `/admin` or any of its sub-routes in v1. The page is served regardless of whether `AUTH_TOKEN` is set. Reason: lean delivery; the operator's network is assumed trusted (localhost / private VPC). Risk: anyone with HTTP access to the gateway can see pool slot labels, session IDs, last-used timestamps, and (via the log tail) every log line including DEBUG-level traces of request paths. Treated as acceptable for v1 because the same network already trusts `/health` and `/health/agents` (which are auth-exempt per AUTH-03). Worth revisiting if/when the gateway ships outside trusted networks.
- **D-02:** Auth is explicitly **deferred** as a follow-on. Phase 8 (plugin hook chain) is the right home for a per-route auth policy that could gate `/admin` behind the same `AuthHook` that protects `/v1/*` and `/api/*`. Not blocking v1.

### Page layout / information density
- **D-03:** Single-page admin view with four stacked sections:
  1. **Summary header strip** — status pill, uptime, version, pool counts (size/alive/busy), session active count
  2. **Pool slots grid** — 3-column responsive card grid; one card per slot showing label, alive/busy indicators, and current_session_id (or "—" when idle)
  3. **Active sessions table** — id (truncated), alive/busy badges, last_used (relative time), model
  4. **Log tail panel** — pinned at the bottom, SSE-driven (see D-08..D-12)
- **D-04:** The `embeddings` field in `/health` JSON is left in place (zero-value) for wire backward compatibility but is NOT rendered on the admin page. Phase 7 cut on 2026-05-27 made the field dead but stripping it from the wire risks breaking any external consumer that decodes the strict struct.

### Refresh / polling architecture
- **D-05:** Single aggregator endpoint `GET /admin/api/snapshot` returns a unified JSON snapshot composing `pool.Stats()` + `registry.Stats()` + `poolDetail.Detail()` + the registry's session detail. One JSON struct, atomic snapshot taken at one instant on the server. Avoids skew between two separate polls and halves the auth/log/HTTP overhead per refresh. Existing `/health` and `/health/agents` stay unchanged for backward compat.
- **D-06:** Client-side polling at **30s fixed interval** (no UI control in v1; sessions/pool state don't change fast enough to need sub-30s refresh). Failed polls show a `#FF8C0A` warning badge with the failure timestamp; successful polls show a green dot + "Last updated Xs ago". Easy to add a UI interval picker in a later phase if needed.
- **D-07:** Snapshot endpoint is registered at the **router root** (alongside `/health`, NOT under a surface-mount block). Auth-exempt in v1 per D-01.

### Log tail architecture
- **D-08:** SSE-based live tail at `GET /admin/logs/stream`. Streams new log lines as they appear in `/tmp/otto-gateway.log` (or whatever `OTTO_LOG` points at). One TCP connection per connected operator; no per-poll HTTP overhead while the page is open.
- **D-09:** **Shared single-tailer pattern** — exactly ONE goroutine across the whole gateway tails the log file at a time. It maintains a 500-line in-memory ring buffer and fans out new lines to N subscribed SSE clients via a channel. Started lazily on first SSE connect; exits when the last subscriber disconnects. Prevents goroutine + file-handle multiplication when multiple operators open the page.
- **D-10:** **File access is strictly read-only and non-blocking.** Open with `os.Open` (no exclusive locks). On read error (file rename / missing / permission revoked), close and re-open at the new file's EOF — log rotation must not crash the tailer or backfill old content. Zero changes to the existing log-writing code path.
- **D-11:** SSE handler tears down cleanly on client disconnect: context cancel → goroutine exits → unsubscribe from the fan-out → if last subscriber gone, the shared tailer goroutine returns.
- **D-12:** UI surface for log tail (all filtering is client-side over the accumulated buffer — no server-side query support):
  - Level filter dropdown: `ALL / DEBUG / INFO / WARN / ERROR`
  - Live grep box: regex filter against parsed `msg` and `path` fields
  - Pause/resume button: stops appending new lines so operator can scroll back without auto-scroll
  - "Newest" badge: appears when paused and new lines arrive
  - Color-coded by level: INFO `#FAFAFA`, DEBUG `#A0A0A0` (brandGray3), WARN `#FF8C0A`, ERROR `#FF3232`
  - On connect, server sends the current ring buffer contents (last ≤500 lines) immediately so the operator has context before the live stream starts

### Distribution / single-binary
- **D-13:** Templates (`html/template`), CSS, and JS shipped via `embed.FS` declared in a new `internal/admin/` package. `go build -o bin/otto-gateway ./cmd/otto-gateway` from a fresh checkout produces one binary; no external file dependencies at runtime; cross-compile from macOS to `linux/amd64` and `windows/amd64` with `CGO_ENABLED=0` is verified as part of phase exit.

### Brand palette (from oscar-adminui UserThemeOptions.js customColors)
- **D-14:** Palette tokens to use across the admin page:
  - **Primary accent** (sparing — selected tab indicators, focused buttons, the "live" dot): `#FAD22D` (brand yellow/gold). Use sparingly — bright yellow on dark surfaces is jarring at scale.
  - **Body background**: `#28243D` (dark purple-navy `bodyBg`)
  - **Card / panel surface**: `#3A3A3A` (paper dark)
  - **Foreground text**: `#FAFAFA` (brandWhite); muted text: `#A0A0A0` (brandGray3); subtle separators: `#4A4A4A` (brandGray1c)
  - **Status — healthy / alive**: `#0FC373` (brandGreen)
  - **Status — warning / poll failed**: `#FF8C0A` (brandOrange)
  - **Status — failed / dead**: `#FF3232` (brandRed)
  - **Activity / live indicator dot, log-stream pulse**: `#AF78D2` (brandPurple)
  - **Secondary CTA / links**: `#1174E6` (brandSecondary)

### Non-interference contract
- **D-15:** Surface gating via `ENABLED_SURFACES` does NOT affect `/admin` or `/admin/*` — those routes are always mounted (when not explicitly disabled by a future config flag).
- **D-16:** Mounting the admin routes does NOT change the registration, ordering, or middleware chain of any existing `/api/*`, `/v1/*`, `/health`, `/health/agents` routes. Verified by existing route-registration tests plus a new admin-route smoke.

### Claude's Discretion
- Exact CSS class names, file layout inside `internal/admin/`, ring buffer struct shape, and SSE event framing details are implementation choices for the planner / executor.
- The 500-line ring buffer size, 30s poll interval default, and whether to wrap the log tail in a fixed-height scrollable region are all reasonable starting points — the planner can adjust if a more sensible default surfaces during research.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `*server.Server` struct in `internal/server/server.go` already holds the `pool`, `registry`, `poolDetail`, `logger`, `version`, `commit`, `start` fields the snapshot endpoint needs. No new dependency wiring at the cmd/otto-gateway main.go level — just call existing `Stats()` and `Detail()` methods.
- `s.healthHandler` / `s.agentsHandler` patterns in `internal/server/health.go` and `internal/server/agents.go` are the templates the new `snapshotHandler` should mirror: nil-safe accessors, `json.NewEncoder(w).Encode(resp)`, error-logging on encode failure.
- The existing access-log middleware (`internal/server/server.go` accessLog chain) already logs `/admin/*` requests for free — no new logging code needed.

### Established Patterns
- **Router root vs surface mounts:** `/admin`, `/admin/api/snapshot`, `/admin/logs/stream` register at the router root (like `/health`) — NOT under `SurfaceMount` blocks. Mirrors AUTH-03's exemption posture for observability endpoints.
- **JSON wire stability:** existing handlers declare typed structs with `json:"..."` tags as the contract. Snapshot endpoint follows the same pattern with a new typed `AdminSnapshot` struct (or equivalent) that downstream JS can rely on.
- **D-05 single-goroutine invariant for streaming:** the Anthropic SSE emitter (`internal/adapter/anthropic/sse.go`) already establishes the pattern of a single goroutine owning a writer + flusher across the stream's lifetime. The log-tail SSE handler should use the same shape — single goroutine per HTTP request, fan-in from the shared tailer's broadcast channel.

### Integration Points
- **Where snapshot data comes from:** `server.pool.Stats()` (pool counts), `server.registry.Stats()` (session counts), `server.poolDetail.Detail()` (per-slot detail). All exist today; snapshot composes them into one struct.
- **Where the log file lives:** `OTTO_LOG` env var (defaults to `/tmp/otto-gateway.log` per `scripts/otto-gw`). The tailer reads from whatever the env resolves to, not a hardcoded path.
- **Where embed.FS lives:** New `internal/admin/` package with its own `templates/`, `static/css/`, `static/js/` directories. `embed.FS` declared at package level alongside the `Handler()` constructor that the server calls during `NewFromConfig`.

</code_context>

<specifics>
## Specific Ideas

- **Brand palette source is fixed**, not negotiable: `/Users/coreyellis/code/github.com/cmetech/oscar_app/oscar-adminui/src/layouts/UserThemeOptions.js` `customColors` block. Use the exact hex values from there.
- **Look-and-feel anchor**: oscar-adminui's general dark-mode admin shell. Lean, simple, stylish — NOT a full dashboard product with sidebars and nav. Single scrollable page.
- **The page should make it obvious at a glance when something is wrong.** A dead pool slot should be visually unmissable (red border on the slot card, not just a tiny indicator).

</specifics>

<deferred>
## Deferred Ideas

- **Auth on `/admin`** — Postponed from v1 per D-01/D-02. Right home is Phase 8 (plugin hook chain) where the existing `AuthHook` can gate the admin routes the same way it gates `/v1/*` and `/api/*`. Should ship before the gateway is deployed outside trusted networks.
- **Operator-selectable refresh interval** — D-06 locks it at 30s for v1. A UI dropdown (1s / 5s / 30s / manual) is trivial to add later if operators ask for it.
- **Historical metrics / sparklines** — Phase 6.1 surfaces the current snapshot only, no time-series. If operators want pool-utilization-over-time graphs or session-count history, that's a separate observability phase (would need a server-side metric store).
- **Request rate / throughput counters** — Same reasoning. Not in the current `/health*` data, would need new instrumentation in the engine. Out of scope.
- **Server-side log filtering** — D-12 keeps level/grep filtering client-side. If logs get noisy enough that the 500-line ring buffer churns through too fast, a follow-up phase can add server-side filtering with query params.
- **Stripping the dead `embeddings` field from `/health`** — D-04 leaves it for backward compat. A future cleanup phase could strip it after a sufficient deprecation window.

</deferred>
