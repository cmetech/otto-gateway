## Page contents / density

| Option | Description | Selected |
|--------|-------------|----------|
| Summary stats only | Pool size/alive/busy + sessions active count. Dense single-screen view. | |
| Summary + per-slot grid | Adds card-grid showing each pool slot's label, alive/busy, current_session_id. | |

**User's choice:** Full layout, with log tail panel added per separate decision.
**Notes:** Brief was "show/expose what we have via API." Summary-only would duplicate `/health` JSON and not justify the slice. Full layout makes the page operationally useful — operator can spot dead slots and stale sessions at a glance.

## Auth UX when AUTH_TOKEN set

| Option | Description | Selected |
|--------|-------------|----------|
| HTTP Basic Auth (browser native) | Browser prompts; password = `AUTH_TOKEN`. Recommended by Claude as low-code + secure. | |
| Query-param token (?token=…) | Token in URL — leaks to history / access logs. Discouraged. | |
| Login form → cookie session | Builds a small `/admin/login` page; sets a session cookie. More code, friendlier UX. | |
| **No auth on /admin in v1** | `/admin` and sub-routes served regardless of `AUTH_TOKEN`. Deferred to Phase 8 / future. | ✓ |

**User's choice:** No auth in v1 — "just show the page."
**Notes:** Operator trusts the network the gateway runs on (localhost / private VPC). Same posture as existing auth-exempt routes (`/health`, `/api/version`). Risk documented in CONTEXT.md D-01: anyone with HTTP access can see pool slot labels, session IDs, last-used timestamps, and via the log tail every log line. Acceptable for v1; should revisit when gateway ships outside trusted networks. Auth deferred to Phase 8 (plugin hook chain) — the existing `AuthHook` is the natural home.

## Polling architecture

| Option | Description | Selected |
|--------|-------------|----------|
| Direct polling | Client polls `/health` AND `/health/agents` separately on each refresh (2 requests). | |
| **Aggregator endpoint** | New `GET /admin/api/snapshot` composes pool + session + slot data into one JSON. Single fetch per refresh; atomic snapshot. | ✓ |

**User's choice:** Aggregator.
**Notes:** Halves auth/log/HTTP overhead per refresh. Atomic snapshot avoids skew between pool count and session count across two requests. Pure server-side composition (no new business logic) — reads existing `pool.Stats()`, `registry.Stats()`, `poolDetail.Detail()`. Existing `/health` and `/health/agents` stay unchanged for wire backward compat.

## Log tail scope

| Option | Description | Selected |
|--------|-------------|----------|
| Defer to later phase | Ship 6.1 without log tail to keep the slice small. | |
| Snapshot last-N-lines (poll) | Each refresh fetches the last 200 lines. No live stream; lines between polls may be missed if logs scroll fast. | |
| **SSE live tail with shared single-tailer** | `GET /admin/logs/stream` opens an SSE connection. ONE shared goroutine across the gateway tails the file and maintains a 500-line ring buffer, fanning out to N subscribed clients. Read-only file access, rotation-tolerant. | ✓ |

**User's choice:** SSE live tail (in scope for 6.1).
**Notes:** User explicitly directed "please include the logging as part of the deliverable." Conservative architecture to avoid impact on the running gateway: shared single-tailer pattern (one goroutine + one file handle regardless of how many operators have the page open), read-only file access, re-open on rotation. Zero changes to existing log-writing code. Client-side level filter + grep + pause/resume implemented over the accumulated buffer. Color-coding by level using the brand palette (`#FAFAFA` info, `#A0A0A0` debug, `#FF8C0A` warn, `#FF3232` error).

## Refresh interval

| Option | Description | Selected |
|--------|-------------|----------|
| 5s fixed | Originally proposed; aggressive but fine since aggregator is one request. | |
| **30s fixed** | Lower network chatter; pool/session state doesn't change fast enough to need sub-30s. | ✓ |
| Operator-selectable | UI dropdown to pick 1s/5s/30s/manual. Deferred. | |

**User's choice:** 30s fixed for v1.
**Notes:** Changed from initial 5s recommendation at user's request. Sessions and pool state don't typically churn fast enough to warrant sub-30s refresh. UI dropdown for operator-selectable interval deferred to a future phase. The SSE log tail is independent — logs still flow live since SSE is an open stream, not polling.

## Claude's Discretion

- Exact CSS class names, file layout inside `internal/admin/`, ring buffer struct shape, and SSE event framing details — implementation choices for planner / executor.
- The 500-line ring buffer size and 30s poll interval are reasonable starting defaults; planner may tune if research surfaces better numbers.
- Whether to wrap the log tail in a fixed-height scrollable region vs full-page-grow is a layout choice for the executor.
- Specific Go package boundary for the admin code (`internal/admin/` vs `internal/server/admin/`) is the planner's call.

## Deferred Ideas

- **Auth on `/admin`** — Phase 8 (plugin hook chain) is the right home. Must ship before the gateway is deployed outside trusted networks.
- **Operator-selectable refresh interval** — Trivial UI dropdown for a later phase.
- **Historical metrics / sparklines** — Would need a server-side metric store. Out of scope.
- **Request rate / throughput counters** — Would need new instrumentation in the engine. Out of scope.
- **Server-side log filtering** — Kept client-side per D-12. Future phase could add `?level=&grep=` query params if log volume warrants.
- **Stripping the dead `embeddings` field from `/health`** — Left present (zero-value) for wire backward compat. Future cleanup phase after a deprecation window.
