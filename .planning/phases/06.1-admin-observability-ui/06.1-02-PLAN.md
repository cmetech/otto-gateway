<objective>
## Phase Goal

**As an** OTTO Gateway operator, **I want to** see the full pool layout and active stateful sessions in the admin page **so that** I can spot a dead pool slot or an unexpected long-lived session at a glance, without parsing `/health/agents` JSON in a terminal.

## Slice Scope (Slice B: Pool slots grid + Active Sessions table)

This plan is **pure additive UI extension** of Slice A. The backend `snapshotHandler` from Plan 01 already returns `pool.slots[]` and `sessions[]` populated with real data; this plan teaches the page to render those arrays.

- Template: fills the previously-stubbed `data-slot-grid` and `data-sessions-card` containers. Adds an empty-state node for the sessions table.
- CSS: adds `.otto-slot-grid` (3-col responsive grid at ≥1024px, 2-col 768-1023px, 1-col <768px), `.otto-slot-card` (card surface with 2px red border override for dead slots — the "visually unmissable" contract from UI-SPEC), `.otto-sessions-table` (header underline, row dividers, no zebra), `.otto-badge` (alive/busy/dead variants at 12% opacity per UI-SPEC), `.otto-empty-state` (centered muted message).
- JS: `renderSlots(slots)` and `renderSessions(sessions)` functions called from the existing `fetchSnapshot` success path; both PATCH the DOM in place (no innerHTML wipe of the whole section; preserve scroll position by reusing existing nodes when slot/session counts are stable, replacing only when the array length changes); relative-time formatter (`Xs`/`Xm`/`Xh`/`Xd ago`) with full ISO timestamp set as the `title=` attribute on each `<td>`.

**End-state user check:** Run a few `POST /v1/chat/completions` and `X-Session-Id`-tagged requests; reload `/admin`; verify the pool grid shows one card per slot (label, alive/busy, current_session_id or `—`) and the sessions table fills with rows. Kill a pool slot (e.g., `kill <pid>` against a kiro-cli child); within 30s the slot card flips to dead state with the 2px red border.

**Out of scope (deferred to later plans):**
- Log Tail SSE panel (Plan 03)
- Cross-compile + binary-size gate (Plan 04)

**Why this is a vertical slice and not a horizontal layer:** Each section of the page is end-to-end (server data → JSON wire → DOM render → visible UI). Plan 02 takes the same handler/JSON path Plan 01 built and grows the rendered output. No new backend code, no new endpoints.
</objective>
