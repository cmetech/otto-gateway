# Phase 06.1: Code Review Report

**Reviewed:** 2026-05-27T20:30:00Z
**Depth:** standard
**Files Reviewed:** 17

## Summary

The implementation is solid. Goroutine lifecycle, race-detector cleanliness (per `goleak.VerifyTestMain` + per-test `defer goleak.VerifyNone(t)`), and the TRST-04 boundary (admin only depends on stdlib + chi) all hold up. Path traversal in the static file server is properly mitigated — `http.ServeFileFS` rejects `..` in `r.URL.Path` before opening the embed, and percent-encoded `%2e%2e` decodes into the raw `..` form caught by the same check (manually verified). XSS is correctly mitigated in `admin.js` via exclusive `.textContent` usage.

No Critical (blocker) findings were identified. Six Warnings exist:
1. **Lone `\r` characters in log lines corrupt SSE frames** (T-6.1-13 mitigation incomplete).
2. **Partial log lines (no trailing `\n`) are silently dropped** when the scanner is reset every tick.
3. **Reconnect dedup window (20 lines) is far smaller than backfill (≤500 lines)** — operator will see up to 480 duplicate lines per EventSource reconnect.
4. **Legitimate duplicate log lines are silently dropped** by the 20-deep dedup ring even in the steady-state (no reconnect).
5. **`pageHandler` / `snapshotHandler` write `WriteHeader(200)` BEFORE template/JSON execution** — a render failure produces a truncated 200 response with no error envelope.
6. **Level filter re-derivation after a dropdown change loses null-level information** — lines parsed as `level=null` become indistinguishable from `level=info` after one dropdown change, producing inconsistent filter behavior between initial-render and re-render paths.

Info findings cover dead code (`subscriber.ctx`, `logConsecutiveReconnects`, `slot.id` fallback), magic numbers (`io.SeekEnd == 2` written as literal `2`), missing `X-Content-Type-Options: nosniff` defense-in-depth header, hardcoded `/admin/` mount path coupling in template + JS, and minor test hygiene (`configIsland` declared and never asserted in `TestAdmin_AssetsFSContains`).

## Warnings

### WR-01: Lone `\r` (carriage return) in log lines corrupts SSE frames

**File:** `internal/admin/sse.go:182-198`
**Issue:** `writeSSELine` splits the payload on `\n` only. The HTML spec for EventSource (`https://html.spec.whatwg.org/multipage/server-sent-events.html#parsing-an-event-stream`) treats both `\r` and `\r\n` as line terminators in addition to `\n`. A log line containing a bare `\r` (e.g., progress-bar `\r`-rewrites, raw subprocess stdout from kiro-cli, Windows-formatted logs missing the `\n` half of `\r\n`) will be split mid-data on the client side, producing truncated visible text and spurious extra fields.

Example: payload `"hello\rworld"` produces `event: log\ndata: hello\rworld\n\n`. EventSource parses `data: hello\r` as one line (value = `hello`) and `world\n` as a new field with name `world` (no colon = empty value) → the event dispatches with `data` = `hello` and a stray field-name-only entry. The operator sees `hello` instead of `hello\rworld`.

This invalidates the T-6.1-13 mitigation in scenarios where logs include `\r` (which is common — slog text format does NOT emit `\r`, but subprocess stdout streams that include carriage-return-overwrites do).

**Fix:**
```go
// In writeSSELine, normalize \r\n to \n before splitting, and strip lone \r:
func writeSSELine(w io.Writer, eventName, payload string) {
    if eventName != "" {
        _, _ = fmt.Fprintf(w, "event: %s\n", eventName)
    }
    if payload == "" {
        _, _ = fmt.Fprint(w, "data:\n\n")
        return
    }
    // Normalize CRLF and lone CR into LF so the split below handles all
    // three EventSource line terminator forms.
    payload = strings.ReplaceAll(payload, "\r\n", "\n")
    payload = strings.ReplaceAll(payload, "\r", "\n")
    segments := strings.Split(payload, "\n")
    for _, seg := range segments {
        _, _ = fmt.Fprintf(w, "data: %s\n", seg)
    }
    _, _ = fmt.Fprint(w, "\n")
}
```

### WR-02: Partial log lines (no terminating `\n`) are silently dropped

**File:** `internal/admin/tail.go:343-359`
**Issue:** Inside the `case <-ticker.C:` branch, after `for scanner.Scan()` exits at EOF, the code creates a fresh `bufio.Scanner` over `f` and updates `lastSize = st.Size()`. The scanner's internal buffer may contain bytes the underlying `os.File` read but did not return (no `\n` terminator yet) — those bytes are discarded when the scanner is replaced. On the next tick, the new scanner reads from the file's current position, which is AHEAD of where the discarded buffer reached. The "tail" of the partial line is silently lost.

Concrete failure case: log writer appends `"INFO step 1\nINFO step 2 in progress"` (no final `\n`). Tailer reads both, `Scan()` returns `INFO step 1` and then returns false at EOF. The `"INFO step 2 in progress"` bytes are in the scanner's internal buffer (the scanner needs `\n` to return them). Scanner gets discarded. Next tick: writer appends `"\nINFO step 3\n"`. Scanner reads from `f.Tell()` which is AFTER the partial line. Operator sees `INFO step 1`, then `INFO step 3` — they never see `INFO step 2 in progress`.

This is more common than it looks: slog flushes per-record but Go process abnormal-termination paths may buffer; subprocess crash dumps; in-progress structured logging where the record write was interrupted.

**Fix:** Track the file position via `f.Seek(0, io.SeekCurrent)` before discarding the scanner, then on the next tick `f.Seek(lastReadOffset, io.SeekStart)` before reading. Alternatively, do not recreate the scanner — `bufio.Scanner` becomes terminal after EOF, but you can use `bufio.NewReader(f).ReadString('\n')` in a loop, which preserves partial-line buffering correctly across ticks. Recommended fix: switch from `bufio.Scanner` to manual `bufio.Reader.ReadString('\n')` with a partial-line carry-over buffer; or keep the scanner and use `f.Seek(currentScannerOffset, 0)` to back up. The current implementation prioritizes simplicity over partial-line correctness.

### WR-03: Reconnect dedup window is 25× smaller than backfill size

**File:** `internal/admin/static/js/admin.js:363-374, 461-462`
**Issue:** `logRecentLines` is a 20-entry ring used to dedupe lines on EventSource reconnect. But the backfill on connect is up to `RingBufferLines = 500` lines (`internal/admin/tail.go:40`). On every browser reconnect (default ~3s reconnect interval per EventSource spec), the SSE server replays up to 500 backfill lines, but the JS dedup buffer only remembers the last 20. The other 480 lines reappear in the UI.

This is a severe UX regression in any environment with intermittent network connectivity (corp VPN, sleep/resume on laptops, container restart of any intermediate proxy). The operator's log viewport will be flooded with duplicates each time.

**Fix:** Increase `LOG_DEDUP_MAX` to at least `RingBufferLines` (500), backed by a `Set` for O(1) lookup. Also consider tracking lines by content-hash or a monotonic line-counter prefix from the server, so identical-but-distinct events (see WR-04) are not collapsed.

```js
var LOG_DEDUP_MAX = 500;  // match server RingBufferLines
var logRecentLinesSet = new Set();
var logRecentLinesQueue = [];

function logIsDupe(line) { return logRecentLinesSet.has(line); }
function logTrackLine(line) {
  if (logRecentLinesSet.has(line)) return;
  logRecentLinesSet.add(line);
  logRecentLinesQueue.push(line);
  while (logRecentLinesQueue.length > LOG_DEDUP_MAX) {
    logRecentLinesSet.delete(logRecentLinesQueue.shift());
  }
}
```

### WR-04: Legitimate duplicate log lines are silently dropped in the steady-state

**File:** `internal/admin/static/js/admin.js:461-462, 366-368`
**Issue:** `logIsDupe(line)` returns true for any exact string match within the last 20 lines, even when no reconnect occurred. Repeated info logs are common in steady-state production (heartbeats, pings, "no work to do" idle messages, identical 200-response access logs). The current implementation silently drops them, making the operator's count of "things that happened" wrong.

Example: kiro-cli emits `"DEBUG ping ok"` every 60s. If the gateway has multiple PingInterval-driven goroutines or the slog handler emits identical messages back-to-back (which is normal), the UI shows the first one and silently swallows the next 19 within the dedup window.

This conflates two semantically distinct problems: (1) reconnect-time backfill replay (the comment says this is what the dedup is for), and (2) steady-state duplicate suppression (an unrelated and undesired side effect).

**Fix:** Tie the dedup to the SSE reconnect event lifecycle only — e.g., on `onSSEOpen`, set a flag `inBackfill = true`; in `onLogEvent`, skip dedup checks when `inBackfill === false`; flip `inBackfill = false` after the first non-duplicate line OR after a short timeout. Alternative: have the server attach a per-line monotonic ID and dedup by ID, not content. Either approach restores correct steady-state semantics.

### WR-05: `pageHandler` and `snapshotHandler` commit `WriteHeader(200)` before rendering

**File:** `internal/admin/admin.go:141-148`, `internal/admin/snapshot.go:108-114`
**Issue:** Both handlers call `w.WriteHeader(http.StatusOK)` BEFORE executing the template or JSON encoder. If `pageTemplate.Execute(w, data)` fails (corrupted embed, template panic during execution, broken-pipe write error), the response is already committed as 200 with `Content-Type: text/html` — the client receives a truncated HTML page with no indication of failure. Same hazard in `snapshotHandler`.

The code comments acknowledge this is intentional (`Cannot write error response after WriteHeader — just log it`), but the better pattern is to render into a `bytes.Buffer` first, then write the full body. Template execution against a buffer never partially commits to the wire, so a 500 envelope is still recoverable on failure.

**Fix:**
```go
func (h *handler) pageHandler(w http.ResponseWriter, r *http.Request) {
    data := struct{ Version, Commit string }{h.deps.Version, h.deps.Commit}
    var buf bytes.Buffer
    if err := pageTemplate.Execute(&buf, data); err != nil {
        h.deps.Logger.Error("admin: page render", "err", err)
        http.Error(w, "admin page render failed", http.StatusInternalServerError)
        return
    }
    w.Header().Set("Content-Type", "text/html; charset=utf-8")
    w.WriteHeader(http.StatusOK)
    _, _ = w.Write(buf.Bytes())
}
```

Same pattern for `snapshotHandler` — marshal with `json.Marshal` first, write on success.

### WR-06: Level filter re-derivation loses null-level information

**File:** `internal/admin/static/js/admin.js:413-436, 517-539, 564-579`
**Issue:** When `appendLine` adds a log line, the CSS `is-debug` / `is-warn` / `is-error` class is added only when `parsed.level` is non-null AND non-info. Both `parsed.level === null` (unrecognized format) and `parsed.level === 'info'` produce zero classes.

Later, when the operator changes the level dropdown, the JS re-derives the level from `el.classList.contains('is-X')` — falling through to `level = 'info'` for any line with no level class. This conflates two distinct cases: (a) genuine INFO lines and (b) lines whose level the parser could not determine.

Symptom: a non-slog log entry (e.g., a raw subprocess panic stack trace) appears in the viewport. Initial render with `logLevelFilter='all'` shows it. Operator changes filter to `warn`. Initial-render path used `parsed.level=null` and `matchesFilters` returned `false` (filter is `warn`, parsed.level is null → null !== 'warn' → hide). After the change, re-derive treats it as `info` → hide. Same result here, but it's coincidence. Now operator selects `info` — re-derived path shows it (info matches info), but on the original `appendLine` it would have been hidden (null !== info). The two paths disagree.

**Fix:** Persist the parsed level (including `null`) on the DOM element via `dataset.level` so the level filter handler reads the same source of truth as the initial render:

```js
function appendLine(line, parsed) {
  ...
  div.dataset.level = parsed.level || 'unknown';
  ...
}

// In level-filter handler:
var level = el.dataset.level === 'unknown' ? null : el.dataset.level;
```

## Info

### IN-01: Hardcoded `/admin/` mount path coupling

**File:** `internal/admin/templates/index.html.tmpl:7,126`, `internal/admin/static/js/admin.js:307,590`
**Issue:** The HTML template and JS hardcode `/admin/static/...`, `/admin/api/snapshot`, and `/admin/logs/stream`. The mount path is fixed in `internal/server/server.go:219` as `/admin`, so this works today, but the admin package's surface is brittle if the mount path is ever changed (CONTEXT mentions Phase 8 may re-mount under an authenticated prefix). Either accept the coupling and document it explicitly, or template the mount path through `Deps.MountPath` and string-substitute into both the HTML and JS at render time. (Note: this would break the `embed.FS`-served static JS asset — JS would need to read mount path from `OTTO_ADMIN_CONFIG`, which is already a string island in the template.)
**Fix:** Add `MountPath` to `Deps` and inject into `OTTO_ADMIN_CONFIG`:
```js
var es = new EventSource(cfg.mountPath + '/logs/stream');
fetch(cfg.mountPath + '/api/snapshot', ...)
```

### IN-02: Magic number `2` instead of `io.SeekEnd`

**File:** `internal/admin/tail.go:290`
**Issue:** `nf.Seek(0, 2) // io.SeekEnd == 2` — the comment correctly identifies that `2` is `io.SeekEnd`, but the constant exists in the `io` package for exactly this purpose. Using the literal is unidiomatic Go.
**Fix:** Import `io` and write `nf.Seek(0, io.SeekEnd)`.

### IN-03: Missing `X-Content-Type-Options: nosniff` header on admin responses

**File:** `internal/admin/admin.go:141`, `internal/admin/snapshot.go:108`
**Issue:** Static assets, HTML page, and JSON snapshot responses do not set `X-Content-Type-Options: nosniff`. The assets are fully controlled at build time via `embed.FS` so MIME confusion is low risk, but defense-in-depth is cheap: setting `nosniff` on all admin responses prevents browsers from heuristically reinterpreting content (relevant if a future change introduces user-controlled response bodies, or for compliance audits).
**Fix:** Add `w.Header().Set("X-Content-Type-Options", "nosniff")` in each handler, or wrap the admin chi mux with a middleware that adds the header to every response.

### IN-04: Dead field `subscriber.ctx`

**File:** `internal/admin/tail.go:155-159, 199-216`
**Issue:** `subscriber` holds a `ctx context.Context` field set in `Subscribe` but read nowhere. The SSE handler uses `r.Context()` directly via the `sseLoop` parameter — the stored `ctx` never participates in lifecycle decisions. Either wire it (e.g., have `broadcast` skip subs whose `ctx.Err() != nil`) or remove the field.
**Fix:** Remove `ctx` from `subscriber`, and drop the `ctx context.Context` parameter from `Subscribe` if it has no other purpose.

### IN-05: Dead JS state `logConsecutiveReconnects`

**File:** `internal/admin/static/js/admin.js:358, 480, 489`
**Issue:** `logConsecutiveReconnects` is incremented in `onSSEError` and reset in `onSSEOpen` but read nowhere. Compare to `consecutiveFailures` (line 17), which IS read at line 327 (`if (consecutiveFailures >= 3) updateDot('is-red')`). Likely a partial implementation of "after N consecutive reconnect failures, show RED dot" that was never finished.

### IN-06: Dead `slot.id` fallback in `buildSlotLabel`

**File:** `internal/admin/static/js/admin.js:97`
**Issue:** `el.textContent = slot.label || ('Slot ' + slot.id)` — `SnapshotSlot` has no `id` field in the wire shape (`internal/admin/snapshot.go:34-39` defines only `label`, `alive`, `busy`, `current_session_id`). If `slot.label` is ever falsy, the fallback renders `Slot undefined`. In production, `pool.Detail()` always produces a label, but the fallback is misleading.
**Fix:** Either drop the fallback (`el.textContent = slot.label || '—'`) or extend `SnapshotSlot` JSON with an `id` field if a numeric position is genuinely useful.

### IN-07: Dead test code in `TestAdmin_AssetsFSContains`

**File:** `internal/admin/handlers_test.go:282-285`
**Issue:** The test declares `var configIsland struct { PollMs int }` then calls `_ = json.Unmarshal([]byte("{}"), &configIsland)`. The struct is never asserted on, and the input is a literal empty object so the unmarshal is a no-op. This appears to be a leftover from a deleted assertion.
**Fix:** Delete lines 282-285.

### IN-08: ReDoS via operator-supplied regex in `initGrepFilter`

**File:** `internal/admin/static/js/admin.js:553-557`
**Issue:** `new RegExp(val, 'i')` accepts any valid JS regex pattern. The operator can paste a catastrophic-backtracking pattern (e.g., `(a+)+$`) into the filter; the regex is then tested against every log line on append AND against every existing line on each input event. This can lock the browser tab. Severity is low because the operator is attacking their own browser (no remote vector), but it's a notable footgun.
**Fix:** Add a basic complexity check (reject patterns with nested quantifiers), and/or use `WebAssembly` regex engines that don't backtrack. Simpler: enforce a 1-second budget per `RegExp.test` call and surface a "filter too expensive" hint.

### IN-09: `subscriber` and `Subscribe` signature suggest unused-context API

**File:** `internal/admin/tail.go:199`
**Issue:** `Subscribe(ctx context.Context) *subscriber` accepts a `ctx` it ignores (it stores it on `sub.ctx` per IN-04 but reads nowhere). Public API of an unexported type is contained to package admin, but the signature is misleading — a future maintainer will reasonably assume that cancelling `ctx` triggers cleanup. It does not; `Unsubscribe(sub)` is the only cleanup path.
**Fix:** Either honor the contract (start a goroutine inside Subscribe that monitors ctx.Done and triggers Unsubscribe), or drop the ctx parameter and require the caller to defer Unsubscribe (which is the current convention — verified by every test site).
