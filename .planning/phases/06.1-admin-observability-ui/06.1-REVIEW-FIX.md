## Fixed Issues

### WR-01: Lone `\r` (carriage return) in log lines corrupts SSE frames

**Files modified:** `internal/admin/sse.go`, `internal/admin/sse_test.go`
**Commit:** `bc6a7b3`
**Applied fix:** Normalize `\r\n` -> `\n` first (so CRLF does not double-split),
then lone `\r` -> `\n` in `writeSSELine` before the existing `strings.Split`
on `\n`. All three EventSource line-terminator forms (`\n`, `\r`, `\r\n`)
now funnel through the same multi-line splitter and become well-formed
`data:` segments. Added three test cases to `TestAdmin_WriteSSELine_MultilinePayload`
asserting lone-CR, CRLF, and mixed-terminator inputs each produce the
correct SSE frame.

### WR-02: Partial log lines (no terminating `\n`) are silently dropped

**Files modified:** `internal/admin/tail.go`, `internal/admin/tail_test.go`
**Commit:** `0cc52a4`
**Applied fix:** Replaced `bufio.Scanner` (which was being recreated each
tick, discarding any buffered partial bytes) with a `bufio.Reader` +
`ReadString('\n')` loop plus an explicit `partialLine` carry-over string
held at tailer-state level. On EOF the trailing bytes (if any) are
preserved; the next tick prepends them so the line emits exactly once
when its terminator finally arrives. Rotation/reopen discards the carry
because it belongs to the prior inode. Per-line byte cap
(`TailerMaxLineBytes`) is enforced by truncating + emitting a marker
when the accumulator grows past the cap. Added
`TestAdmin_TailerPartialLinePersistsAcrossTicks` exercising the
write-partial → wait → complete-partial sequence and asserting both
that the partial is held back during interim ticks and that it emerges
exactly once after completion.

Also tidied an adjacent IN-02-flavored cleanup: replaced the
`nf.Seek(0, 2) // io.SeekEnd == 2` magic number with the named
`io.SeekEnd` constant since `io` was already being imported for the
new `errors.Is(err, io.EOF)` check. (IN-02 itself remains out of scope
for this iteration; this is the same line of code so the swap rode
along with the WR-02 imports for free.)

### WR-03 + WR-04: Reconnect-backfill dedup vs steady-state duplicate handling

**Files modified:** `internal/admin/static/js/admin.js`
**Commit:** `4017567`
**Applied fix:** Single commit covers both findings because they share the
same dedup code path. The prior implementation used a 20-entry FIFO ring
that was both (a) far too small to absorb the server's 500-line backfill
on reconnect (WR-03: operator saw up to 480 duplicate lines per
reconnect), and (b) always-on, so legitimate steady-state duplicates
(heartbeats, identical access logs) were silently swallowed (WR-04).

Inverted the dedup contract: dedup is now ACTIVE ONLY during a
post-open backfill window of `LOG_BACKFILL_WINDOW_MS` (2s), sized for
the full 500-line server replay (`LOG_BACKFILL_MAX = 500` matches
server `RingBufferLines`), using a `Set` + FIFO queue for O(1) lookup
(was O(n) `indexOf`). After the window closes every incoming line
passes through, so steady-state duplicates reach the operator
unfiltered. `logBackfillStart()` is wired into `onSSEOpen` so the
window re-arms on every browser auto-reconnect.

### WR-05: `pageHandler` and `snapshotHandler` commit `WriteHeader(200)` before rendering

**Files modified:** `internal/admin/admin.go`, `internal/admin/snapshot.go`
**Commit:** `76387f3`
**Applied fix:** Both handlers now render into a `bytes.Buffer` first.
On render failure (template execute error, JSON encoder error) nothing
has been committed to the wire yet, so a clean 500 envelope (via
`http.Error`) is still recoverable. On success, headers + 200 + buffer
contents are written in the usual order. Post-`WriteHeader` write
errors (client disconnected mid-write) drop to a debug log — there is
no recovery path at that point.

### WR-06: Level filter re-derivation loses null-level information

**Files modified:** `internal/admin/static/js/admin.js`
**Commit:** `00ee6e5`
**Applied fix:** `appendLine` now persists `parsed.level` (or `'unknown'`
for the null case) on the DOM element via `dataset.level`. Added a
`levelFromElement(el)` helper that returns `null` for `'unknown'` or
the level string otherwise. Both re-derive paths (`initLevelFilter`
dropdown handler, `initGrepFilter` debounce handler) now call
`levelFromElement` instead of inferring level from CSS class
membership. `matchesFilters` therefore sees identical `{ level }`
shapes from both initial render AND re-render paths — null-level
lines (raw subprocess panic stacks, non-slog format) are now
classified consistently regardless of which path runs.
