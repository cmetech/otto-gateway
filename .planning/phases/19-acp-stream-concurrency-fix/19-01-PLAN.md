<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Goroutine A (`close()`) ↔ Goroutine B (`Result()` caller, then caller-of-Result) | The data hazard. Goroutine A acquires `s.mu` AFTER `close(s.done)` unblocks Goroutine B's `<-s.done` wait. B can return its pointer-to-`s.result` and release `s.mu` while A is still racing to `s.mu.Lock()` + `StopReason` write at `stream.go:182`. B's caller then dereferences `fr.StopReason` with no lock held → `go test -race` data-race report. |

No external trust boundary changes (no auth, session, input-validation,
or crypto surface change). This is a pure intra-process synchronization
fix.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-01 | Tampering (under writer's own intent — silent stale read by reader goroutine) | `internal/acp/stream.go` — `Stream.Result()` returning `s.result` pointer that `close()` then mutates | **mitigate** | D-19-01: copy `*s.result` into a stack-local `cp` inside the existing `s.mu` critical section in `Result()` and return `&cp`. close()'s later mutations target `s.result`, not `cp`. Verification: `TestRegression_REL_ACP_01` positive assertion + `go test -race -count=60 ./internal/acp/ -run REL_ACP_01` (≥ 6,000 race trials per CI run). |
| T-19-02 | Information Disclosure (none) | n/a | **accept** | `FinalResult` carries no secrets — only `SessionID` (opaque routing key, set once at `newStream`), `ChunkCount` (counter), `StopReason` (enum). No PII / no credentials flow through this struct. |
| T-19-03 | Denial of Service (deadlock from re-acquiring `s.mu`) | `Result()` post-fix body | **accept** | `Result()` takes `s.mu` once and only reads — does NOT call any method that itself takes `s.mu`. No re-acquisition path. Verified by inspection of the 10-line post-fix body. |
| T-19-04 | TOCTOU on `if s.result == nil` then `cp := *s.result` | `Result()` post-fix body | **accept** | Both the nil-check and the struct-value copy happen inside the SAME `s.mu` critical section (single `s.mu.Lock()` / `defer s.mu.Unlock()` pair). No mutation window between check and use. |
