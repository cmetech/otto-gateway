# Phase 08.3.2: PII Smoke Test Methodology Fix — Research

**Researched:** 2026-06-03
**Domain:** Smoke-test methodology / deterministic test fixtures for LLM-mediated pipelines (PowerShell + Go test plumbing)
**Confidence:** HIGH for code-state facts (every claim verified against the working tree). MEDIUM-HIGH for the option recommendation (verified preconditions; outcome depends on which path planner picks).

## Summary

The Phase 8.3 deadlock fix shipped in v1.9.3 and was verified live on Windows on 2026-06-03. All three load-bearing assertions in `08.3-HUMAN-UAT.md` (engine.prompt.sent semantic shift, engine.prompt.completed shape, no ping/timeout failures) passed. The PII smoke test's "no ciphertext tokens leaked" check ALSO passed three times across all three surfaces — proving the encrypt → forward → decrypt pipeline is healthy. What failed is the `round-trip decrypt: '<value>' NOT in response` assertion, because Claude (via `kiro-cli`) refuses to verbatim-echo PII-shaped data on safety grounds, even when the gateway has tokenized the input. The gateway is correct; the smoke test bitrotted against a model policy change.

The test methodology assumes LLM cooperation. That assumption is now false against current Claude releases. The phase scope is to replace that dependency in `scripts/test-pii.ps1` (the `pii` scenario only) while keeping the test runnable on Windows and not regressing `make ci` or the v1.9.3 internal/acp / encrypt-decrypt code.

The good news for option 2a (preferred): `tests/e2e/cmd/fake-kiro-cli/main.go` already exists, already implements every ACP method the gateway uses (`initialize`, `session/new`, `session/set_model`, `session/prompt`, `session/cancel`, `ping`), and is pure-Go with no cgo. Its existing notification mechanism (`OTTO_FAKE_KIRO_NOTIFICATIONS_FILE` — a pre-built NDJSON file of `session/update` frames the fake replays during prompt) is sufficient: the smoke test writes a notifications file containing one `agent_message_chunk` per expected plaintext value, fires the gateway-tokenized prompt, and asserts those values round-trip. **No code changes to fake-kiro-cli itself are required for option 2a — only PowerShell-side wiring.** The fake exists today only because of build-tagged Go e2e tests; making the PowerShell script reach it cross-platform requires either a `go build` step in the script or a pre-baked binary location convention.

**Primary recommendation:** Implement option 2a (fake-kiro-cli worker mode) as the primary path. It is the only option that closes the underlying class of bitrot. Add option 2c (synthetic non-PII-shaped fixtures) as a fallback `-Mode live` path so an operator can still verify the live `kiro-cli` end-to-end pipeline against project codes when needed. Skip option 2b (reframed prompt) — it solves no class of problem, only one instance of it.

The critical insight: the fake's notifications are written by the smoke test BEFORE the gateway sees them, AFTER the operator has decided what plaintext to verify. ### Pattern 1: Operator-Mode Switch in a PowerShell Smoke Test
**What:** Add a `-Mode` (or `-Fake`) parameter that gates whether the script (a) launches a side gateway against `KIRO_CMD=<fake-binary>` or (b) just talks to the already-running gateway (today's behavior).

**When to use:** Whenever a smoke test has both a "live end-to-end" path AND a "deterministic-component-isolated" path. Standard test pyramid: keep both.

**Example shape (planner to refine):**
```powershell
[ValidateSet('live', 'fake')]
[string]$Mode = 'fake'  # safe default — the live path has known LLM-cooperation failure

# When -Mode fake: the script either (a) requires the operator to start the
# gateway with KIRO_CMD=fake-binary OR (b) launches its own short-lived gateway.
# Plan must decide which — preferred is (a) because it matches the existing
# "operator starts gateway, runs probes" workflow.
```

### Pattern 2: Notifications File for Deterministic Worker
**What:** Write a temp NDJSON file containing one JSON-RPC `session/update` notification per text chunk the fake should emit during `session/prompt`. The fake reads `OTTO_FAKE_KIRO_NOTIFICATIONS_FILE`, replays each line verbatim, then sends its `session/prompt` response.

**When to use:** Any time a test needs the gateway to see specific worker output without depending on real LLM behavior.

**Example (the exact frame shape, from `tests/e2e/tools_fixtures_test.go:212`):**
```json
{"jsonrpc":"2.0","method":"session/update","params":{"sessionId":"e2e-session-1","update":{"sessionUpdate":"agent_message_chunk","content":{"text":"Email: corey@cmetech.io\n"}}}}
```

`e2e-session-1` is the sessionID the fake hard-codes at `main.go:114`. The smoke test must use that exact ID (or extend the fake to accept the session-id from the test, but option 2a explicitly avoids extending the fake). [VERIFIED: `tests/e2e/cmd/fake-kiro-cli/main.go:113-122`]

### Anti-Patterns to Avoid
- **Editing fake-kiro-cli for one test:** The fake is shared across the entire e2e tools matrix. Any behavior change here ripples. Per `main.go` design notes: it's "controllable" via env vars *precisely* to avoid per-test forks. Keep that contract.
- **Hard-coding the fake binary path:** Use `go build -o $env:TEMP\fake-kiro-cli.exe` or a Makefile target. Different operators have different repo layouts.
- **Defaulting to -Mode live:** Default-on means the smoke test continues to fail out-of-the-box on a fresh operator box. Default to `fake` and document that `-Mode live` is for operators verifying end-to-end against a real kiro-cli.
- **Building the fake inside `test-pii.ps1` without a Go-presence check:** Operators without Go installed must get a clear error and the option to point at a pre-built binary via `$env:OTTO_FAKE_KIRO_BIN`.
- **Removing the existing PII-laden prompt entirely:** The shell sibling (`scripts/test-pii.sh:250`) and the PowerShell script (`scripts/test-pii.ps1:310-319`) both carry the same prompt. Plan must keep them in sync — drift between OS sibling scripts is a known bitrot vector in this project (see commit `f7ccd40` "UTF-8 BOM + ASCII-only chars for PowerShell 5.1" — a previous sibling-drift incident).

## Common Pitfalls

### Pitfall 1: PowerShell 5.1 + UTF-8 BOM
**What goes wrong:** PowerShell 5.1 (Windows 10 default) chokes on UTF-8 with BOM when piped to non-PS programs and on certain non-ASCII characters in here-strings.
**Why it happens:** Legacy Windows + PS5.1 default codepage.
**How to avoid:** Already fixed in commit `f7ccd40`. Stick to ASCII in script literals; if Unicode is needed in the prompt, escape via `\uXXXX` or fixture file. Notifications-file write should use `[System.IO.File]::WriteAllText($path, $body, [Text.UTF8Encoding]::new($false))` (no-BOM) — NOT `Out-File` (which writes BOM by default in 5.1).
**Warning signs:** Fake-kiro reads a corrupted first byte from notifications file, JSON parse fails silently in the fake's `emitNotifications` loop (see `tests/e2e/cmd/fake-kiro-cli/main.go:202`), no chunks emit, smoke test reports "no decryptable text in response."

### Pitfall 2: Fake-Kiro Notifications File NDJSON Strictness
**What goes wrong:** The fake's `emitNotifications` validates each line as parseable JSON and silently skips malformed lines (`main.go:202`). No log line tells you "I dropped your frame."
**Why it happens:** Belt-and-suspenders robustness intended for tests that compose frames programmatically.
**How to avoid:** Plan must include a verification step where the smoke test (or a setup task) parses each line of the notifications file with `ConvertFrom-Json` BEFORE handing it to the gateway. Catch malformed JSON at test-build time, not at runtime.
**Warning signs:** Test reports "no ciphertext leak" (good) but plaintext is also missing (bad) — fake emitted nothing because all your frames failed json.Unmarshal.

### Pitfall 3: Hard-coded session ID in the fake
**What goes wrong:** `tests/e2e/cmd/fake-kiro-cli/main.go:114` returns `"sessionId": "e2e-session-1"` for every `session/new` request. Any notification frame whose `params.sessionId` doesn't match WILL still be replayed (the fake doesn't filter), but the gateway's `internal/acp/client.go` Phase 8.3.1 demux work — IF it lands before 8.3.2 — will route mismatched-session updates to a "dropped" warning instead of the active stream.
**Why it happens:** Fake was designed before per-session demux.
**How to avoid:** Phase 8.3.1 is a parallel-track concern (not a dependency). Plan should make notification frames use `e2e-session-1` literally as the sessionId field so they're never dropped by demux. Document this brittleness in the plan as a Phase 8.3.1-coordination note.
**Warning signs:** After 8.3.1 lands, every `session/update` frame in the notifications file gets dropped with `acp: session/update for unknown session — dropped` in the log.

### Pitfall 4: `KIRO_CMD` resolved via PATH vs absolute
**What goes wrong:** Gateway resolves `KIRO_CMD` as written — if it's `fake-kiro-cli` (bare name), it runs `exec.LookPath` (Go stdlib). If `bin/fake-kiro-cli.exe`, that's a relative path resolved against the gateway's cwd, NOT the operator's invocation cwd.
**Why it happens:** `os/exec.CommandContext` standard behavior at `internal/acp/client.go:284`.
**How to avoid:** Smoke test MUST set `KIRO_CMD` to an absolute path. Use `(Resolve-Path $fakeBin).ProviderPath` in PowerShell to canonicalize.
**Warning signs:** Gateway fails to spawn worker, `/health/pool` shows `last_spawn_error` containing the fake binary path.

### Pitfall 5: PII encrypt mode boot validation requires `PII_ENCRYPT_KEY`
**What goes wrong:** Without `PII_ENCRYPT_KEY` set, the gateway boot fails OR (depending on slice 5 boot validation) falls back to plaintext.
**Why it happens:** `internal/plugin/pii/encrypt.go:DeriveKey` errors on empty value; `modes.go:ApplyMode` `encrypt` branch logs warn and returns plaintext if key is nil.
**How to avoid:** Smoke test invocation flow must ensure the gateway env at start time has both `PII_REDACTION_MODE=encrypt` AND `PII_ENCRYPT_KEY=<anything-non-empty>`. The default `.env.otto-gw.example` template already does both. No script change needed if operator follows install instructions; document explicitly in HUMAN-UAT addendum.
**Warning signs:** Gateway boots but `/health/hooks` shows `PIIRedactionHook.mode != "encrypt"` OR shows it but emits plaintext on the wire to the worker — smoke test catches this as a different failure shape (no ciphertext leak BUT also no decrypt to verify).

### Pitfall 6: `kiro-cli` PATH resolution on Windows under WSL
**What goes wrong:** If operator has `KIRO_CMD=wsl.exe` (a documented config), pointing at a fake binary may need to be `KIRO_CMD=C:\path\to\fake-kiro-cli.exe` directly. WSL+fake is a non-goal.
**Why it happens:** Documented Windows-WSL ACP-connection caveat (`internal/admin/templates/docs.html.tmpl:311`).
**How to avoid:** Plan should explicitly call out: option 2a smoke run uses a Windows-native fake binary, NOT a Linux-fake-via-WSL. The fake is pure-Go so it cross-compiles trivially per CLAUDE.md.
**Warning signs:** Hangs on session/new; the WSL stdio quirk re-surfaces against fake-kiro the same way it did against real kiro-cli.

### Building the notifications file (PowerShell)
```powershell
# Source: shape verified against tests/e2e/cmd/fake-kiro-cli/main.go:202
# (json.Unmarshal succeeds on each line) and tools_fixtures_test.go:212 (NotifText shape)
function New-NotifFrame {
    param([string]$SessionId, [string]$Text)
    $frame = [pscustomobject]@{
        jsonrpc = "2.0"
        method  = "session/update"
        params  = [pscustomobject]@{
            sessionId = $SessionId
            update    = [pscustomobject]@{
                sessionUpdate = "agent_message_chunk"
                content       = [pscustomobject]@{ text = $Text }
            }
        }
    }
    $frame | ConvertTo-Json -Compress -Depth 10
}

$lines = @(
    New-NotifFrame -SessionId "e2e-session-1" -Text "Email: corey@cmetech.io`n"
    New-NotifFrame -SessionId "e2e-session-1" -Text "IPv4: 192.168.1.42`n"
)
$notifsPath = Join-Path $env:TEMP "otto-pii-notifs.ndjson"
# Use no-BOM UTF-8 — see Pitfall 1.
[System.IO.File]::WriteAllText(
    $notifsPath,
    ($lines -join "`n") + "`n",
    [System.Text.UTF8Encoding]::new($false)
)
```
