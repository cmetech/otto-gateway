## Shared Patterns

### Pattern: `sessionId = "e2e-session-1"` literal

**Source:** `tests/e2e/cmd/fake-kiro-cli/main.go:114` (the fake hard-codes this on `session/new`).
**Apply to:** All notification frames the smoke test writes — PowerShell (`New-NotifFrame -SessionId "e2e-session-1"`), shell (`printf ... "sessionId":"e2e-session-1"`), and any Go test if added.
**Why:** Phase 08.3.1 per-session demux will route mismatched-session updates to the dropped-warning path (RESEARCH Pitfall 3). Using the literal is forward-compatible with that refactor landing before or after this phase.

### Pattern: No-BOM UTF-8 file write on Windows

**Source:** RESEARCH §Common Pitfalls Pitfall 1; commit `f7ccd40` (prior sibling-drift incident on this same file).
**Apply to:** Every PowerShell file-write that the fake binary will read (notifications NDJSON in particular).
```powershell
[System.IO.File]::WriteAllText($path, $body, [System.Text.UTF8Encoding]::new($false))
```
**Anti-pattern to avoid:** `Out-File`, `Set-Content`, and `>` redirection in PS5.1 all write a BOM by default. The fake's `json.Unmarshal` will silently skip the first frame.

### Pattern: Absolute `KIRO_CMD` paths

**Source:** RESEARCH Pitfall 4; `internal/acp/client.go:284` (`exec.CommandContext` standard PATH resolution).
**Apply to:** The operator step + the smoke-script error-message text that tells the operator how to set `KIRO_CMD`.
```powershell
$KiroCmdValue = (Resolve-Path $fakeBin).ProviderPath
```
**Anti-pattern:** `KIRO_CMD=bin/fake-kiro-cli.exe` (relative to gateway cwd, not invocation cwd).

### Pattern: Operator-mode flag default `fake` (not `live`)

**Source:** RESEARCH §Anti-Patterns; Open Question #2 recommendation; ROADMAP §08.3.2 SC2 preferred order (2a first).
**Apply to:** Both `scripts/test-pii.ps1` and `scripts/test-pii.sh`.
**Why:** The live path is known-broken on current Claude releases. Defaulting to `fake` means the script passes on a fresh operator box; `-Mode live` is opt-in for the operator who wants to verify against a real `kiro-cli`.

### Pattern: Help text + comment-based help update

**Source:** `scripts/test-pii.ps1:1-56` (existing PS comment-based help block).
**Apply to:** Both scripts when adding the new mode parameter.
**Why:** PS comment-based help is consumed by `Get-Help` and by the script's own `-h` / `-Help` aliases (`scripts/test-pii.ps1:76-83`). The new `.PARAMETER Mode` block must appear there for the help message to render the new flag.

### Pattern: PostHook decrypt invariant — preserved by both modes

**Source:** `internal/plugin/pii/modes.go:142-173` (ApplyMode encrypt branch) + RESEARCH §Architecture Diagram.
**Apply to:** No code change — but the plan MUST document that both `-Mode fake` and `-Mode live` exercise the encrypt PreHook + decrypt PostHook end-to-end. Only the worker's output content changes between modes; the hook chain is identical.
**Why:** The "no ciphertext tokens leaked" assertion is the load-bearing correctness signal for the encrypt/decrypt pipeline. Fake mode preserves it because the fake echoes plaintext, the gateway encrypts plaintext on the way in, the fake's echo-back text contains the plaintext (sourced from the NDJSON), AND any `[PII:…]` tokens passed through would still be decrypted by the PostHook — so a regression in decrypt would still surface as a leaked token.
