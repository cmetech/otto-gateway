# Phase 01.1 Plan 04: Notification Defensive Parsing + Permission RESPONSE Summary

**2. [Rule 1 — Bug] Dispatcher route logic was silently dropping server-initiated requests**

- **Found during:** Task 3 — `TestAutoGrantPermission` timed out after 2s waiting for the outbound response. The client received the permission request frame (id 42) but never invoked `handleNotification`.
- **Issue:** Phase 1's `dispatcher.route()` routed every frame-with-id to the pending map (looking it up against in-flight outbound requests we'd registered). kiro-cli sends `session/request_permission` as a JSON-RPC REQUEST (both `id` AND `method` populated), which means the dispatcher would silently drop the frame: no pending entry for id 42, so `route()` falls through. `handleNotification` is never called -> no `rpcResponse` is written -> kiro-cli deadlocks waiting for the response to its frame id. Plan 04's source change in `handleNotification` is correct in isolation but unreachable on the wire.
- **Fix:** Rewrote `dispatcher.route()` to dispatch by method-field presence per JSON-RPC 2.0:
  - `method != ""` -> `onNotif` (notification OR server-initiated request)
  - `method == "" && id != nil` -> pending-map lookup (response)
  - else -> drop silently (malformed)
- This is the JSON-RPC 2.0 wire-spec contract verbatim. Phase 1 conflated request/response routing by id-presence alone.
- Updated the three existing dispatcher tests to drop the spurious `Method:"…"` on response-frame fixtures.
- Added `TestDispatcherRoutes_ServerRequest_ToOnNotif` to lock the D-20 surface.
- **Files affected:** `internal/acp/dispatcher.go`, `internal/acp/dispatcher_test.go`.
- **Commit:** `2bab738`.
