# Phase 1.1: ACP Wire Alignment - Context

**Gathered:** 2026-05-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 1.1 brings the Phase 1 ACP JSON-RPC client (`internal/acp`) into
compliance with the live ACP spec and the working Node implementation
(`acp-ollama-server.js`, proven against `kiro-cli` 2.4.1) so Phase 2 can
invoke `session/prompt` against real `kiro-cli` and consume
`session/update` notifications without empty prompts, dropped chunks, or
permission-grant deadlocks. The phase ships a real-kiro `session/prompt`
round-trip integration test as the verification gate that unblocks
Phase 2.

**Deliverables (per ROADMAP.md Phase 1.1 success criteria):**

1. Spec-compliant `initialize` request: `params` includes
   `protocolVersion: 1`, `clientInfo`, and `clientCapabilities` (with
   `fs.{readTextFile,writeTextFile}` + `terminal` flags). Response's
   `agentCapabilities.promptCapabilities` is captured and accessible.
2. Spec-compliant `session/new`: params include `mcpServers: []`. Result
   reading handles both `sessionId` and `id` fallback.
   `result.models.availableModels[]` is extracted and surfaced.
3. Spec-compliant `session/prompt`: params field name is `prompt` (with
   `content` alias sent defensively). Block wire shape matches spec —
   text uses field `text`; `resource_link` includes required `name`;
   `image` block construction wired. Result's `stopReason` parsed and
   surfaced as `canonical.StopReason`.
4. Spec-compliant `session/update` consumption: accepts `session/update`
   OR `session/notification` OR `_kiro.dev/session/update` method names;
   unwraps `params.update` defensively; reads `sessionUpdate` (with
   `type` fallback); handles `agent_message_chunk`,
   `agent_thought_chunk`, `tool_call`, `tool_call_chunk`,
   `tool_call_update`, `plan` (plus CamelCase aliases); content
   extraction uses `body.content?.text ?? body.content ?? body.text`
   fallback chain.
5. `session/request_permission` handled as a REQUEST (responds to the
   original frame `id` with `{result:{optionId:"allow_always",
   granted:true}}`); the `session/grant_permission` request path from
   Phase 1 is removed.
6. New `internal/acp/integration_test.go` real-kiro test: gated on
   `kiro-cli` (D-17 pattern); spawns the subprocess, completes
   `Initialize → NewSession → Prompt("hi")`, drains `stream.Chunks`,
   asserts at least one `ChunkKindText` chunk arrives with non-empty
   content, asserts `Stream.Result()` returns with a non-error
   `StopReason` (typically `StopEndTurn`). `goleak.VerifyNone(t)`
   passes. **This is the Phase 2 unblock gate.**

**Requirements:** ACP-01..ACP-05 (re-validation against the live spec —
no NEW requirements; this is a correctness/alignment fix). One canonical
addition is load-bearing for the wire alignment: `ResourceLinkBlock`
gains a required `Name` field per ACP spec.

**Explicitly NOT in Phase 1.1:**

- Full D-08..D-10 canonical types (`ChatRequest`, `ChatResponse`,
  `Message`, `ContentPart`, `ToolCall`, `ToolSpec`, `Usage`, etc.) —
  Phase 2 lands these where they have consumers.
- Engine package (`internal/engine`), pool package (`internal/pool`) —
  Phase 2.
- Any HTTP adapter (Ollama, OpenAI, Anthropic) — Phase 2/3/3.1.
- `image` block content beyond the wire-shape struct field — Phase 2
  populates from Ollama `images: [b64]`.
- Full tool-call canonical chunk emission (Phase 1.1 keeps the Node
  reference's behavior: `tool_call` / `tool_call_update` updates are
  rendered as `ChunkKindThought` text — Phase 6 properly emits
  `canonical.ToolCallChunk` with `toolCallId`/`title`/`args`).
- Property tests via `pgregory.net/rapid` — Phase 6 (TRST-06 policy
  unchanged from Phase 1 D-10).
- Hosted CI — D-08 from Phase 1 still holds.

**Carrying forward from Phase 1 (decisions still in force):**

<decisions>
## Implementation Decisions

### Canonical type growth (`internal/canonical`)

- **D-01: Minimum-floor canonical additions only.** Phase 1.1 adds the
  smallest set of canonical types required by the wire-alignment work.
  `ChatRequest` / `ChatResponse` / `Message` / `ContentPart` / `ToolCall`
  / `ToolSpec` / `Usage` etc. (Phase 2's D-08..D-10 grab-bag) are
  explicitly deferred to Phase 2, where they have direct consumers
  (engine + adapter). Phase 1.1 ships:
  - `canonical.ResourceLinkBlock.Name string` (new field on existing
    struct) — required by ACP spec on `resource_link` content blocks.
  - `canonical.StopReason` enum (new) — ACP-faithful set, see D-02.
  - `canonical.ModelInfo struct { ID, Name string }` (new) — carries
    `availableModels[]` entries.
  - `canonical.PromptCapabilities struct { Image, Audio, EmbeddedContext bool }`
    (new) — carries `agentCapabilities.promptCapabilities` from
    `initialize` response.

- **D-02: `StopReason` is ACP-faithful + `StopUnknown`.** Enum values:
  `StopEndTurn`, `StopMaxTokens`, `StopMaxTurnRequests`, `StopRefusal`,
  `StopCancelled`, `StopUnknown`. One-to-one with ACP spec
  (`end_turn | max_tokens | max_turn_requests | refusal | cancelled`),
  plus `StopUnknown` for forward compatibility with values kiro-cli may
  emit that the spec doesn't (yet) enumerate. Wire-string parser maps
  unknown strings to `StopUnknown` rather than failing the response.
  Phase 2/3.1 may EXTEND this enum later (e.g., `StopSequenceMatch`,
  `StopToolUse`, `StopError`) when adapters need to render those for
  OpenAI/Anthropic responses sourced from non-ACP signals — that
  extension is not a Phase 1.1 concern.

- **D-03: `ModelInfo` and `PromptCapabilities` live in
  `internal/canonical`.** Keeps `internal/canonical` as the leaf
  type-only package (Phase 1 D-04 — canonical imports nothing under
  `internal/`). Phase 2's Ollama adapter will consume `ModelInfo` for
  `/api/tags` + `/api/show`; Phase 3.1's Anthropic adapter will consume
  `PromptCapabilities.Image` for image-block gating. File layout
  (planner's discretion): `canonical/model.go` and
  `canonical/capabilities.go` or grouped — pick one and be consistent.
  Per D-11 from Phase 1, no JSON tags on these types.

- **D-04: `ResourceLinkBlock` empty-Name policy — fall back to
  `path.Base(URI)` in `translate.go`.** When a producer hands a
  `canonical.Block{Kind: BlockKindResourceLink, ResourceLink: &{URI: "file:///foo/bar.txt"}}`
  with empty `Name`, `translate.go` populates the wire `name` field as
  `path.Base("/foo/bar.txt")` (with `file://` scheme stripped via
  `net/url.Parse`). Defensive at the seam — matches the
  `acp_wire_shapes.md` recommendation. Phase 2 cwd derivation reads
  `ResourceLink.URI` directly; this fallback doesn't affect cwd logic.

### API surface for ACP handshake state (`internal/acp`)

- **D-05: Accessor methods on `*Client`; existing signatures unchanged.**
  `Initialize(ctx) error` and `NewSession(ctx, cwd) (string, error)`
  keep their Phase 1 signatures (no breaking change for `main.go` or
  the future pool). Internally, `*Client` captures the response data
  during the RPC and exposes it via:
  - `func (c *Client) PromptCapabilities() canonical.PromptCapabilities`
    — returns the captured value (zero struct before `Initialize`
    succeeds; replaces on every successful `Initialize`).
  - `func (c *Client) AvailableModels() []canonical.ModelInfo` —
    returns a snapshot copy (nil before `NewSession` succeeds; replaces
    on every successful `NewSession`). Snapshot copy avoids the caller
    mutating shared state.
  Phase 5's pool will surface a `pool.Models()` that delegates to the
  first slot's `AvailableModels()` — same data per `kiro-cli` install.

- **D-06: Concurrency primitive — new `sync.RWMutex` separate from
  `streamMu`.** `*Client` gains a `stateMu sync.RWMutex` (distinct from
  the existing `streamMu` that guards `activeStream`) covering the two
  new fields:
  ```go
  type Client struct {
      // ... existing ...
      stateMu sync.RWMutex
      caps    canonical.PromptCapabilities
      models  []canonical.ModelInfo
  }
  ```
  `Initialize` / `NewSession` take `c.stateMu.Lock()` on the write side;
  the accessors take `c.stateMu.RLock()`. `streamMu` stays focused on
  the single-active-stream invariant; conflating them would extend
  critical sections unnecessarily.

- **D-07: `Stream.Result()` returns `StopReason`.** Parse
  `result.stopReason` from the `session/prompt` response, map the wire
  string to `canonical.StopReason` (unknown → `StopUnknown`), and
  surface it on `Stream.Result()` (currently
  `Result() (*FinalResult, error)`). `FinalResult` gains a
  `StopReason canonical.StopReason` field; or change the signature to
  `Result() (canonical.StopReason, error)` if `FinalResult` is empty
  today — planner's discretion based on the current `stream.go` shape.

### Wire-shape changes (`internal/acp/client.go`, `internal/acp/translate.go`)

- **D-08: Spec-compliant `initialize` params.** Rename existing
  `Capabilities capabilities` field to `ClientCapabilities clientCapabilities`
  on `initializeParams`. Add `ProtocolVersion int` (constant value `1`).
  `clientCapabilities` struct contains an `Fs fsCapabilities` field
  (`ReadTextFile, WriteTextFile bool`) and a `Terminal bool` field. For
  Phase 1.1 the gateway does NOT implement `fs/*` or `terminal/*`
  reverse handlers, so emit `fs: {readTextFile: false, writeTextFile: false}`
  and `terminal: false` — declares to kiro that we cannot service those
  reverse requests (matches the spirit of the brief: gateway is not a
  full ACP client). Re-evaluate if kiro-cli starts demanding `true` for
  the gateway to make progress. The Node implementation declares `true`
  for both because it relays those reverse calls; we don't (yet).

- **D-09: Capture `agentCapabilities.promptCapabilities` in
  `Initialize`.** After receiving the `initialize` response, parse
  `result.agentCapabilities.promptCapabilities` into the new
  `canonical.PromptCapabilities` value and store under `c.stateMu`.
  Defensive parsing: missing/empty `promptCapabilities` → zero struct
  (all false). Phase 3.1 will read this to gate image-block construction
  (and refuse Ollama→ACP image translation if `Image=false`).

- **D-10: Spec-compliant `session/new` params include
  `mcpServers: []`.** Add `MCPServers []mcpServer` to
  `sessionNewParams` (with `json:"mcpServers"` tag — emits explicit
  `[]` not `null` via `make([]mcpServer, 0)` or via a custom marshaler).
  For Phase 1.1 the gateway provides no MCP servers (Phase 8+ may
  surface this), so the slice stays empty. The Node implementation
  sends `mcpServers: []` defensively even though it's documented as
  optional — kiro-cli versions may treat missing as an error.

- **D-11: Read `result.sessionId` with `result.id` fallback.** New
  `sessionNewResult` shape captures both fields:
  ```go
  type sessionNewResult struct {
      SessionID string          `json:"sessionId"`
      ID        string          `json:"id"`
      Models    sessionNewModels `json:"models"`
  }
  ```
  `NewSession` returns `firstNonEmpty(r.SessionID, r.ID)`. If both are
  empty → return an error (the protocol contract is broken).

- **D-12: Extract `result.models.availableModels[]` in `NewSession` and
  store on `*Client`.** Parse each entry into
  `canonical.ModelInfo{ID: entry.ModelID, Name: entry.Name}` (the wire
  field is `modelId`, not `id` — see `acp_wire_shapes.md` §2). Store
  the resulting `[]canonical.ModelInfo` under `c.stateMu`. Replace on
  every successful `NewSession` (same kiro-cli install → same answer,
  so this is idempotent in practice).

- **D-13: Spec-compliant `session/prompt` params: rename `Blocks` →
  `Prompt`, add defensive `Content` alias.** New `promptParams` shape:
  ```go
  type promptParams struct {
      SessionID string      `json:"sessionId"`
      Prompt    []wireBlock `json:"prompt"`
      Content   []wireBlock `json:"content"` // defensive duplicate per Node ref
  }
  ```
  Both fields carry the same translated `[]wireBlock` slice — `Content`
  is sent defensively in case older kiro-cli versions still read the
  old name (Node code does this on line 296–303 of
  `acp-ollama-server.js`).

- **D-14: `wireBlock` shape — text uses `text` not `content`;
  resource_link adds required `name`; image construction wired.** New
  `wireBlock`:
  ```go
  type wireBlock struct {
      Type     string `json:"type"`
      Text     string `json:"text,omitempty"`      // type=="text"
      URI      string `json:"uri,omitempty"`       // type=="resource_link"
      Name     string `json:"name,omitempty"`      // type=="resource_link" (REQUIRED on wire; D-04 fallback fills)
      Title    string `json:"title,omitempty"`     // type=="resource_link"
      MIMEType string `json:"mimeType,omitempty"`  // type=="image"
      Data     string `json:"data,omitempty"`      // type=="image" (base64)
  }
  ```
  `translateBlock` switch grows an `BlockKindImage` arm (if Phase 1.1
  adds the canonical type — see D-15). The old wire-side `Content` field
  for text is renamed to `Text`. `Name` is REQUIRED on `resource_link`
  per spec; populate from `b.ResourceLink.Name` or fall back to
  `path.Base(URI)` per D-04.

- **D-15: `BlockKindImage` lives in Phase 2 with the Ollama adapter,
  not Phase 1.1.** Phase 1.1 sets up the `wireBlock` to carry image
  fields (so the wire encoder is complete) but doesn't add an
  `BlockKindImage` constant to `canonical.BlockKind` and doesn't add
  an `ImageBlock` struct. Rationale: the producer of image blocks is
  the Ollama adapter (translating `messages[].images: [b64]`), and the
  adapter+engine arrive together in Phase 2. Phase 1.1 keeping the
  block-shape work narrow avoids a struct in `canonical` with no
  consumer. The wire field tags exist so Phase 2 can add the canonical
  type and the translate.go arm with no schema churn.

  **Decided 2026-05-23 during write_context:** This pushes Phase 2's
  block scope slightly — Phase 2's `internal/canonical` work grows by
  one block kind. Captured here to align ROADMAP intent with this
  decision; Phase 2 CONTEXT will reflect.

### Notification handling — defensive parsing (`internal/acp/client.go`,
`internal/acp/translate.go`)

- **D-16: Three notification method names matched explicitly in
  `handleNotification`.** Switch case becomes:
  ```go
  case "session/update", "session/notification", "_kiro.dev/session/update":
  ```
  No normalization on method names (they're three exact strings); the
  normalization happens on the discriminator inside the body per D-19.

- **D-17: Body parsing — tolerant outer struct + `json.RawMessage` for
  the variant slot.** `sessionUpdateParams` shape:
  ```go
  type sessionUpdateParams struct {
      SessionID string          `json:"sessionId"`
      Update    json.RawMessage `json:"update"`  // present when params wraps in .update
      // The "flat" fields, populated when params is NOT wrapped:
      SessionUpdate string          `json:"sessionUpdate"`
      Type          string          `json:"type"`           // discriminator fallback
      Content       json.RawMessage `json:"content"`        // variant content
      Text          string          `json:"text"`           // last-resort text
      ToolCallID    string          `json:"toolCallId"`
      Title         string          `json:"title"`
      Args          map[string]any  `json:"args"`
      Output        string          `json:"output"`
      Entries       []planEntry     `json:"entries"`
  }
  ```
  After the first unmarshal, if `Update` is non-nil, re-unmarshal it
  into a `sessionUpdateBody` struct (same shape minus `SessionID` and
  `Update`); else treat the outer as the body. One small helper
  `firstNonEmpty(values ...string) string` covers each fallback chain
  inline. `map[string]any` is used ONLY where the shape is genuinely
  variant (e.g., `args` payload).

- **D-18: Content extraction follows `body.content?.text ?? body.content ?? body.text`.**
  `Content json.RawMessage` is decoded with a probe: try to unmarshal
  as `{text: string}` first; if it parses with a non-empty `text`, use
  that. Otherwise, try to unmarshal as a plain string; if that succeeds,
  use it. Otherwise, fall back to `body.Text` (the top-level `text`
  field). All three branches feed `translateUpdate` with a single
  `content string` argument.

- **D-19: Discriminator normalization — one helper, snake_case
  canonical form.** `normalizeUpdateType(s string) string` lowercases
  CamelCase to snake_case (e.g., `AgentMessageChunk` →
  `agent_message_chunk`). Behavior:
  - If `s` contains an underscore, return `strings.ToLower(s)`
    unchanged structurally (already snake-case; just normalize casing).
  - Else, insert `_` before every internal uppercase ASCII letter and
    lowercase the whole thing.
  - Empty → empty (caller treats as "unknown variant").
  After normalization, the switch has one case per variant:
  ```go
  switch normalizeUpdateType(firstNonEmpty(body.SessionUpdate, body.Type)) {
  case "agent_message_chunk": ...
  case "agent_thought_chunk": ...
  case "tool_call", "tool_call_chunk": ...  // Node renders both as thought
  case "tool_call_update": ...               // ditto
  case "plan": ...
  default: log Debug, drop (or fall back to text per current Phase 1 behavior)
  }
  ```
  Explicit table-driven unit test (`TestNormalizeUpdateType`) covers
  the normalization cases.

### `session/request_permission` handling

- **D-20: Respond on the original frame `id` — `session/grant_permission`
  request path is removed.** `handleNotification` for
  `session/request_permission` now sends an `rpcResponse{ID: <original
  frame id>, Result: {optionId: "allow_always", granted: true}}` over
  `writeCh`. The previous Phase 1 code path (which sent a new request
  with method `session/grant_permission` and a new id) is deleted in
  full — including:
  - The `session/grant_permission` `rpcRequest` send
  - The `grantParams` struct (`requestId` is no longer echoed back; the
    grant goes on the original RPC frame's id)
  - The unused `grantID := c.nextID.Add(1)` line
  `permissionParams` shape stays only if needed to debug-log the
  inbound request (planner's discretion).

  **Note:** `rpcResponse` is a new outbound envelope shape (`{jsonrpc,
  id, result}` — no `method`). Phase 1 didn't have this because the
  Phase 1 grant was sent as a new request. Add the shape inline to
  `client.go` or a small new file `rpc.go` — planner's choice.

### Testing strategy

- **D-21: Reframe fake-server tests to test DEFENSIVE PARSING variance.**
  The Phase 1 fake tests confirmed implementation-implementation
  symmetry (the fake emitted what our code expected, both using the
  same incorrect flat shape). Phase 1.1 changes the goal:
  - **Real-kiro test** (`integration_test.go`) is the spec-compliance
    proof — runs against `kiro-cli 2.4.1` ground truth.
  - **Fake-server tests** are repurposed to exercise the VARIANCE
    surface: multiple wire shapes that real-kiro can't easily emit (one
    real kiro-cli emits only its single wire variant).

- **D-22: Variance density lives in `translate_test.go` table-driven
  unit tests.** New `TestTranslateUpdate_VarianceMatrix` walks rows
  combining:
  - Notification method name (3 values)
  - Body wrap location (`params.update` vs flat)
  - Discriminator field (`sessionUpdate` vs `type`)
  - Discriminator casing (snake_case vs CamelCase)
  - Content shape (`{text: "x"}` vs `"x"` vs `body.text="x"`)
  - Update type (text / thought / tool_call / tool_call_chunk /
    tool_call_update / plan)
  Each row asserts the produced `canonical.Chunk` equals the expected
  value. Run as a Go table-driven test — no `rapid` dependency (TRST-06
  policy unchanged).

- **D-23: One end-to-end fake test that exercises mixed variants
  through the dispatcher.** Reduce the existing four
  `TestIntegration_FakeACP_*` tests down to ONE
  `TestIntegration_FakeACP_E2E_MixedVariants` (or two if the auto-grant
  scenario can't be cleanly folded in). The fake emits a sequence of
  notifications using DIFFERENT variants across the sequence (e.g.,
  first chunk via `session/update` + snake_case, second via
  `session/notification` + CamelCase, third via
  `_kiro.dev/session/update` + flat-content-string), and asserts the
  resulting `stream.Chunks` produces the expected canonical sequence.
  Confirms dispatcher routing handles all method names AND that the
  parser handles all body/discriminator/content variants WITHIN a
  single session. Existing `Ping`, `AutoGrant`, and
  `PromptChunkDelivery` happy-path assertions can be folded into this
  one test.

- **D-24: Real-kiro round-trip test is the Phase 2 unblock gate.**
  `internal/acp/integration_test.go` gets a new
  `TestIntegration_RealKiroCLI_PromptRoundTrip` (next to the existing
  `TestIntegration_RealKiroCLI_SmokeTest`). It:
  1. Resolves `kiro-cli` via D-17 (`LOOP24_KIRO_BIN` env var or PATH);
     skips if absent.
  2. Spawns the subprocess via `acp.New(cfg)`.
  3. Calls `Initialize(ctx)`.
  4. Asserts `client.PromptCapabilities()` is non-zero (at least one
     of Image/Audio/EmbeddedContext is true).
  5. Calls `NewSession(ctx, os.TempDir())`.
  6. Asserts `client.AvailableModels()` is non-empty.
  7. Calls `Prompt(ctx, sid, []canonical.Block{{Kind:BlockKindText,
     Text:&canonical.TextBlock{Content:"hi"}}})`.
  8. Drains `stream.Chunks`; asserts at least one `ChunkKindText` chunk
     arrived with non-empty `Text.Content`.
  9. Calls `stream.Result()`; asserts no error and that the returned
     `StopReason` is a non-error value (typically `StopEndTurn`).
  10. `defer goleak.VerifyNone(t)`.
  Timeout: 60 seconds (LLM responses can take time). Test is the Phase
  2 verification gate; Phase 2 cannot ship without it passing.

- **D-25: Test file layout unchanged.** Per Phase 1 D-18: whitebox unit
  tests in `framer_test.go`, `dispatcher_test.go`, `client_test.go`,
  `translate_test.go` (`package acp`); blackbox integration tests in
  `integration_test.go` and `fakeacp_test.go` (`package acp_test`);
  `testmain_test.go` keeps `goleak.VerifyTestMain` per TRST-05. Phase
  1.1 expands `translate_test.go` per D-22 and rewrites
  `fakeacp_test.go` + `integration_test.go` per D-21..D-24.

### Claude's Discretion

- Exact file layout inside `internal/canonical` for the new types —
  one new file (e.g., `types.go`) or split (`model.go` +
  `capabilities.go` + `stop_reason.go`). Pick one and be consistent

- Exact signature of `Stream.Result()` after the `StopReason` addition
  — extend the existing return shape or replace `*FinalResult` with
  `(canonical.StopReason, error)`. Pick based on the current Phase 1
  `stream.go` shape (see D-07).
- Where the `rpcResponse` envelope shape lives — inline in `client.go`
  next to `rpcRequest` / `rpcNotification`, or split into a small
  `rpc.go` file once there are 3+ envelope shapes. Pick one.
- Whether `Cwd` is still passed via the existing `sessionNewParams.CWD`
  field name or renamed to match the spec's `cwd` casing — the JSON
  tag is already `cwd`; the Go field name (`CWD` vs `Cwd`) is cosmetic.
- Exact error wording for unknown stop reasons / malformed responses
  — consistent with Phase 1's `acp:` prefix convention.
- Whether to log a `Debug` or `Warn` on unknown notification
  discriminator values after normalization — `Debug` is the kinder
  default (kiro-cli versions may add new types we don't recognize yet);
  `Warn` is louder.
- Sequential vs parallel execution of the seven new translate_test.go
  table rows — table-driven Go tests are sequential by default; if any
  row is slow (it shouldn't be — pure unit tests), planner can switch
  to `t.Parallel()`.

<specifics>
## Specific Ideas

- **`kiro-cli` 2.4.1 at `~/.local/bin/kiro-cli` is the ground-truth
  target.** Symlinked from `/Applications/Kiro CLI.app/Contents/MacOS/kiro-cli`.
  Where the live ACP spec and Node code differ on a wire shape, the
  Node code wins for Phase 1.1 because it's empirically validated
  against this binary.
- **The Node code's defensive triple-acceptance of notification
  methods, defensive `prompt`+`content` duplication, and three-step
  content extraction fallback chain are intentional.** kiro-cli
  versions vary in wire shape; defensive parsing absorbs the variance.
  Phase 1.1 preserves all of these patterns (D-13, D-16, D-17, D-18).
- **The fake-server integration test pattern was load-bearing wrong
  in Phase 1.** Phase 1.1 reframes the fakes to test the DEFENSIVE
  variance (D-21..D-23) while the real-kiro round-trip test (D-24) is
  the spec-compliance proof. This is the unblock gate for Phase 2.
- **Phase 1.1 doesn't touch the engine, pool, or any adapter.** Even
  though Phase 2's CONTEXT mentions `engine.New`, `pool.New`, and
  `adapter/ollama` extensively, those are Phase 2 work — Phase 1.1
  stays inside `internal/acp` + `internal/canonical`.
- **`session/request_permission` is the highest-priority deadlock
  fix.** The Phase 1 implementation sends a new request with a new id;
  kiro-cli is waiting for a response to the original id and blocks
  forever. Phase 2's first tool-using prompt would deadlock without
  D-20. Make sure the verification step asserts a tool-requesting
  prompt completes — or that the real-kiro test exercises a prompt
  that triggers `session/request_permission` (most non-trivial prompts
  do).
- **`promptCapabilities` is forward-looking but cheap to ship now.**
  Phase 1.1 only captures and exposes the value; Phase 3.1's Anthropic
  adapter will be the first consumer (gating image-block construction).
  Phase 2's Ollama adapter doesn't strictly need this gate — Ollama's
  `images: []` shape is base64-encoded inline, and kiro is allowed to
  reject if it doesn't support image — but Phase 1.1 captures it
  defensively.
