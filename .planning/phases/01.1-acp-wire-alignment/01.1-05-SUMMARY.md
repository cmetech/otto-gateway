  12. `result, err := stream.Result()`; `t.Fatalf` on err; nil result -> `t.Fatal`.
  13. `switch result.StopReason` over `StopEndTurn|StopMaxTokens|StopMaxTurnRequests` (acceptable) vs `StopUnknown` (parser regression — Errorf) vs default (refusal/cancelled — Errorf with value).

## Decisions Made

- **Drain loop style: `for !drained { select { ... } }`** rather than `goto drained:`. Both forms are explicitly allowed by Plan 05's `<action>` step 11. The bool-flag form was chosen because (a) there are no `goto` labels anywhere else in `integration_test.go`, so the bool keeps the file's style consistent, and (b) the loop body is short enough that the extra variable is not visual noise. Both compile to roughly the same machine code; pick is stylistic.

- **Test placement: BEFORE `TestIntegration_RealKiroCLI_SmokeTest`** in source order. The plan said "adjacent to" without specifying order. Putting the more-comprehensive test first matches Go convention (most-important / most-load-bearing test first). The SmokeTest's doc-comment was updated to say "above this" rather than "next to this" — one-word cosmetic edit, included in the same commit.

- **ErrClientClosed -> t.Skip guard preserved verbatim from SmokeTest.** Plan 05's `<action>` step 6 specified this pattern, and it is the only way the test can soft-skip when kiro-cli auth has expired (the binary is present but exits during initialize, surfacing as `acp.ErrClientClosed`). Phase 2 / Phase 5 may want to make this Errorf instead so CI flags broken-auth environments rather than skipping silently — punted as a future tightening.

- **No build tag on the test file.** The existing `integration_test.go` uses `package acp_test` with no build tag (it gates itself via `resolveKiroCLI(t)`'s skip-if-absent). The new test follows the same convention — no `//go:build integration` tag. Matches Plan 05's `<read_first>` note on the build-tag check.

- **Acceptable StopReason set kept tight.** Plan 05's `<action>` step 13 allows `StopEndTurn` / `StopMaxTokens` / `StopMaxTurnRequests`. Treating `StopRefusal` and `StopCancelled` as failures (rather than acceptable) is the right call because the prompt is the literal string `"hi"` — a refusal or cancellation would indicate either kiro-cli misbehaviour or a bug in the parser. `StopUnknown` is treated as a parser regression with a more pointed error message ("the wire string from kiro-cli was unrecognised or missing") since that's the most actionable failure mode for triage.
