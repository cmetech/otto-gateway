# Phase 1.1: Code Review Report

**Reviewed:** 2026-05-23
**Depth:** standard
**Files Reviewed:** 15
**Status:** issues_found

## Summary

Phase 1.1 brings the ACP wire shapes into compliance with the live spec and the proven Node implementation. The variance-absorbing translation layer (D-16..D-19) and the permission-response deadlock fix (D-20) are the highest-risk surfaces; both have credible coverage via the variance matrix and the consolidated E2E test. Canonical-purity (D-04, D-11) holds — `internal/canonical` does not import anything under `internal/` and carries no JSON tags. Concurrency primitives (`stateMu` distinct from `streamMu`) are correctly partitioned and I found no double-locking or held-lock-across-RPC patterns.

However, there is one **BLOCKER** in the `rpcFrame.ID` parse path: the type pins ID to `*uint64`, which silently drops any frame carrying a JSON-RPC string ID or a negative integer. Production kiro-cli emits numeric IDs today, so this is not an immediate runtime fault — but a `session/request_permission` frame with a string ID (allowed by JSON-RPC 2.0 §request.id) would unmarshal `ID=nil`, hit the "permission request without id — dropped" branch (`client.go:800`), and re-introduce the D-20 deadlock silently. Because the entire variance-tolerance posture of this phase is "absorb upstream wire variance defensively," accepting only numeric IDs is a known-shape gap on the same surface that D-20 is supposed to harden.

Several WARNINGs cluster around the readLoop/stream backpressure architecture: a slow consumer of `stream.Chunks` will block readLoop indefinitely, which will then prevent the `session/prompt` response from being read out of the pipe — a real deadlock if kiro emits more than 64 chunks before responding. The real-kiro test today calls `Prompt()` non-concurrently with its drain loop and depends on this not happening; this is fragile.

## Critical Issues

### CR-01: `rpcFrame.ID` typed as `*uint64` silently drops string/negative IDs — re-introduces D-20 deadlock

**File:** `internal/acp/dispatcher.go:11-17` (declaration); `internal/acp/client.go:791-810` (consumer)
**Issue:** The frame envelope declares `ID *uint64`. JSON-RPC 2.0 §4 (Request object) allows `id` to be a string, number, or null. Numeric IDs that are negative also won't unmarshal into `uint64`. When kiro-cli (or any future ACP agent) emits a `session/request_permission` REQUEST with a string ID, `json.Unmarshal` will leave `frame.ID = nil` while `frame.Method = "session/request_permission"`. The dispatcher routes correctly (method != "" arm wins), but `handleNotification`'s permission branch then takes the `frame.ID == nil` → "permission request without id — dropped" path (client.go:798-802). The subprocess waits forever for a response — the exact deadlock D-20 was supposed to fix. This is a load-bearing assumption for the entire phase's value proposition (Phase 2's first tool-using prompt will hang).

The phase's posture is explicitly "absorb wire-shape variance defensively." Accepting only numeric IDs contradicts that posture on the same surface (permission RPC) that D-20 is hardening. The Node implementation that this phase replicates does not have this restriction — JavaScript's `JSON.parse` produces a `string|number` naturally.

**Fix:** Type the ID as `json.RawMessage` (or `any`) and preserve it byte-for-byte on the outbound response. Concretely:

```go
// dispatcher.go
type rpcFrame struct {
    ID     json.RawMessage `json:"id,omitempty"`
    Method string          `json:"method,omitempty"`
    Result json.RawMessage `json:"result,omitempty"`
    Error  *rpcError       `json:"error,omitempty"`
    Params json.RawMessage `json:"params,omitempty"`
}

// Helper: a frame is a "response candidate" iff it has an id and no method.
func (f rpcFrame) hasID() bool { return len(f.ID) > 0 && string(f.ID) != "null" }

// dispatcher.route()
func (d *dispatcher) route(frame rpcFrame) {
    if frame.Method != "" {
        d.onNotif(frame)
        return
    }
    if !frame.hasID() {
        return
    }
    // For pending-map lookup we still need a comparable key — parse the
    // numeric ID here (where we control it), not at the wire boundary.
    var num uint64
    if err := json.Unmarshal(frame.ID, &num); err != nil {
        return // unknown ID shape for a response — drop
    }
    d.mu.Lock()
    ch, ok := d.pending[num]
    if ok { delete(d.pending, num) }
    d.mu.Unlock()
    if ok { ch <- frame }
}

// rpcResponse envelope — id is now json.RawMessage so we echo whatever shape arrived.
type rpcResponse struct {
    JSONRPC string          `json:"jsonrpc"`
    ID      json.RawMessage `json:"id"`
    Result  any             `json:"result"`
}

// handleNotification: permission branch
case "session/request_permission":
    if !frame.hasID() {
        c.cfg.Logger.Warn("acp: permission request without id — dropped")
        return
    }
    data, err := json.Marshal(rpcResponse{
        JSONRPC: "2.0",
        ID:      frame.ID, // echo verbatim, regardless of string/number/etc.
        Result:  map[string]any{"optionId": "allow_always", "granted": true},
    })
    // ... rest unchanged
```

Add a translate_test row asserting a string-ID `session/request_permission` frame produces an echoed-string-ID `rpcResponse`.

## Warnings

### WR-01: readLoop blocks on `stream.push` backpressure → prompt response can be buffered behind chunks → deadlock if buffer fills

**File:** `internal/acp/client.go:856-858` (push call site); `internal/acp/stream.go:63-73` (push body); `internal/acp/stream.go:50` (64-slot buffer)
**Issue:** `Stream.push` blocks on `s.chunks <- ch` until the consumer drains or `c.clientCtx` is cancelled. Push is called from `handleNotification`, which is called from `dispatcher.route`, which is called from `readLoop`. Therefore: a slow consumer of `stream.Chunks` blocks readLoop. While readLoop is blocked, no further frames are read from the subprocess pipe — including the `session/prompt` response that would close the stream.

The 64-slot buffer in `newStream` is the cushion. If kiro-cli emits more than 64 `session/update` chunks before the `session/prompt` response (entirely possible for long generations), the system deadlocks UNLESS the caller drains `Chunks` concurrently with calling `Prompt`. The integration test does this (`integration_test.go:136-208` runs Prompt on a goroutine). But `TestIntegration_RealKiroCLI_PromptRoundTrip` (`integration_test.go:352-379`) does NOT — it calls `Prompt` synchronously and drains afterward. It works today because real-kiro emits ~3 chunks for "hi" (per CONTEXT.md). Any future regression where kiro emits >64 chunks against this test will deadlock silently until the 90s test timeout fires.

This is also a documentation gap on the public API: `Prompt()`'s godoc doesn't tell callers they MUST consume `Chunks` from a separate goroutine. Phase 2's HTTP adapters need this contract spelled out.

**Fix:** Two complementary changes:
1. **Document the contract** on `Prompt()` and `Stream.Chunks`: callers must drain `Chunks` concurrently with the call that returns the stream. Add a worked example in the godoc.
2. **Rework the real-kiro test** to drain on a goroutine like the fake-server test does. Today's structure is a latent flake.

```go
// In integration_test.go:TestIntegration_RealKiroCLI_PromptRoundTrip — drain on goroutine:
chunksDone := make(chan struct{})
var textChunks int
var firstText string
go func() {
    defer close(chunksDone)
    for chunk := range stream.Chunks {
        if chunk.Kind == canonical.ChunkKindText && chunk.Text != nil && chunk.Text.Content != "" {
            textChunks++
            if firstText == "" { firstText = chunk.Text.Content }
        }
    }
}()
// Now Prompt-side response (which is the SAME goroutine that called Prompt,
// since Prompt is synchronous) can return — but in this test Prompt was already
// called above; restructure so Prompt runs on a goroutine like the fake test does.
```

A longer-term fix is to make `push` non-blocking-with-fallback (e.g., drop + log + flag on FinalResult.Truncated), but that changes the contract. Document-and-fix-the-test is the minimal change.

### WR-02: Permission response can stall readLoop indefinitely if `writeCh` is full

**File:** `internal/acp/client.go:827-831`
**Issue:** The permission-response select intentionally has no `default` arm (correct: dropping a permission response deadlocks kiro-cli). But the consequence is that a full `writeCh` (cap=16) will block the readLoop goroutine indefinitely. Under load with many concurrent Prompt calls in Phase 5's pool, this becomes a real risk: every Prompt enqueues a request via `writeCh`, and a permission response that arrives while 16 sends are buffered will block the readLoop until the writer drains.

The fundamental issue is that the readLoop is **both** the consumer of inbound frames AND the producer of one specific outbound frame (the permission response). This is OK at low concurrency but the architecture should be documented or refactored — perhaps the permission response should bypass writeCh and call `framer.writeFrame` directly under `framer.mu` (already there).

**Fix:** Either (a) document the invariant clearly ("permission responses serialize behind any pending writes — keep writeCh shallow OR ensure no Prompt RPC starvation") OR (b) write the permission response directly via the framer's locked writeFrame and skip writeCh entirely. Option (b) is structurally cleaner:

```go
// in handleNotification, permission branch:
if err := c.framer.writeFrame(json.RawMessage(data)); err != nil {
    c.cfg.Logger.Warn("acp: permission response write failed", "err", err)
}
```

This works because `framer.writeFrame` is mutex-protected (`framer.go:58-65`) — concurrent calls with the writer goroutine are already safe.

### WR-03: Misleading comment in `Close()` — code does not actually ignore "signal: killed"

**File:** `internal/acp/client.go:899-904`
**Issue:** The comment claims "Ignore 'signal: killed' — that's expected when context kills the process," but the code unconditionally wraps any error from `cmd.Wait()` into `firstErr`. Callers see a non-nil error from `Close()` on every successful subprocess termination via `cmd.cancel()`. The integration tests work around this by `t.Logf`-ing the error (`integration_test.go:299-301`, `:430-435`).

**Fix:** Either implement what the comment says, or update the comment. The cleanest fix is to actually filter:

```go
if err := c.cmd.Wait(); err != nil && firstErr == nil {
    // The subprocess was killed by clientCtx cancellation — that's expected.
    var exitErr *exec.ExitError
    if errors.As(err, &exitErr) && exitErr.ProcessState != nil {
        // Process was signaled (signal: killed) — expected on Close path.
        if !exitErr.ProcessState.Exited() {
            // signaled, not exited — that's our doing
        } else {
            firstErr = fmt.Errorf("acp: cmd wait: %w", err)
        }
    } else {
        firstErr = fmt.Errorf("acp: cmd wait: %w", err)
    }
}
```

Or simply log instead of returning, since `Close()` is on the teardown path:
```go
if err := c.cmd.Wait(); err != nil {
    c.cfg.Logger.Debug("acp: cmd exited (likely killed by context)", "err", err)
}
```

### WR-04: `extractContent` mishandles "explicit empty text" wire shape

**File:** `internal/acp/translate.go:251-265`
**Issue:** When the wire shape is `{"content":{"type":"text","text":""}}`, the first probe unmarshals successfully but `probe.Text != ""` is false, so it falls through. The second probe (`string`) fails because Content is an object, not a string. So it falls all the way through to `body.Text` (`return body.Text`). If `body.Text` is non-empty (some unrelated flat field), that value is surfaced instead of the explicit empty-text content.

The bug is the `probe.Text != ""` short-circuit on success. In the wrapped/`content`-object form, the presence of `content.text` (even empty) is the load-bearing signal — falling back to `body.Text` is wrong.

**Fix:** Distinguish "Content was an object that successfully parsed as `{text:...}`" from "Content was something else." Use a sentinel like:

```go
func extractContent(body sessionUpdateBody) string {
    if len(body.Content) > 0 {
        var probe struct{ Text *string `json:"text"` } // pointer signals presence
        if err := json.Unmarshal(body.Content, &probe); err == nil && probe.Text != nil {
            return *probe.Text // explicit empty text is still a real value
        }
        var s string
        if err := json.Unmarshal(body.Content, &s); err == nil {
            return s
        }
    }
    return body.Text
}
```

Low-probability in practice — kiro-cli is unlikely to emit empty `text` inside a populated `content` block — but the variance-absorption posture of this phase warrants closing this hole. Add a translate_test row:

```go
{
    name: "empty_content_text_does_not_leak_body_text",
    paramsJSON: `{"sessionId":"s1","sessionUpdate":"agent_message_chunk","content":{"type":"text","text":""},"text":"leaked"}`,
    want: canonical.Chunk{Kind: canonical.ChunkKindText, Text: &canonical.TextChunk{Content: ""}},
},
```

### WR-05: Inner-unmarshal failure in `translateUpdate` is silently dropped

**File:** `internal/acp/translate.go:172-193`
**Issue:** When the wrapped form arrives but the inner `update` payload is malformed JSON (e.g., a server bug or wire corruption), `json.Unmarshal(u.Update, &body)` errors are intentionally ignored (`_ = json.Unmarshal(...)`). The result is a zero-valued body, which falls through to the default-text arm producing an empty `ChunkKindText`. The user sees a phantom empty chunk; nothing is logged.

The comment acknowledges this ("the inner-unmarshal failure here is silently tolerated rather than reported separately") and points to handleNotification's outer-unmarshal Warn as the reporting channel. But that path only fires for the OUTER unmarshal; an inner-unmarshal failure is invisible.

**Fix:** Either propagate the error up to handleNotification (which has a logger) by returning `(Chunk, error)` and letting the caller decide whether to drop or log, or accept the loss and document it. Returning a sentinel from translateUpdate is cleaner:

```go
// In handleNotification, before push:
if chunk.Kind == 0 && chunk.Text == nil && chunk.Thought == nil && chunk.Plan == nil {
    // translateUpdate produced an empty result — log and drop.
    c.cfg.Logger.Debug("acp: session/update translated to empty chunk — dropped",
        "method", frame.Method, "params", string(frame.Params))
    return
}
```

This won't catch the "intentional empty default" case (unknown discriminator with empty content) cleanly. A better signal is to return `bool` from translateUpdate indicating "produced a valid chunk" — or, simplest, log the malformed wrap from inside translateUpdate via a package-level slog default. The current silent-drop is the worst of the options.

### WR-06: Magic number `-32099` repeated 5 times — no named constant

**File:** `internal/acp/client.go:536, 592, 645, 676, 737`; `internal/acp/dispatcher.go:112`
**Issue:** The sentinel JSON-RPC error code `-32099` is used in 5 places in `client.go` (one in each of Initialize, NewSession, Ping, SetModel, Prompt error paths) and once in `dispatcher.go:drainAll`. A typo in one of them silently swallows the close signal. This is exactly the kind of magic-number drift that Go's `const` is for.

**Fix:**
```go
// dispatcher.go (top of file, near rpcError):
// closeSentinelCode marks a frame synthesized by drainAll() to fail
// in-flight callers when the client is closing. Not part of the wire spec.
const closeSentinelCode = -32099
```

Then replace all 6 occurrences. The test at `dispatcher_test.go:195` also checks `-32099` directly — update it to use the constant.

### WR-07: Real-kiro test relies on chunk count fitting in 64-slot buffer

**File:** `internal/acp/integration_test.go:352-379`
**Issue:** Closely related to WR-01: this test calls `Prompt()` synchronously, then drains `stream.Chunks`. It depends on kiro-cli emitting ≤64 chunks before the prompt response — guaranteed for "hi" today, but a kiro behavioral change (e.g., emitting per-token instead of per-block chunks) would silently make this test deadlock until the 90s timeout fires. There's no assertion that catches "chunks were buffered" vs "chunks were truncated."

**Fix:** Restructure to drain concurrently (see WR-01 fix). Also consider asserting `result.ChunkCount > 0` to catch silent-truncation scenarios.
