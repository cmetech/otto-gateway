**Anthropic exception to the two-path rule (CRITICAL clarification):** Per CONTEXT D-07 (locked), Anthropic SSE streaming emits NATIVE `tool_use` content blocks for kiro-native `ChunkKindToolCall` — `content_block_start{tool_use, input:{}}` → `content_block_delta{input_json_delta}` → `content_block_stop`. This is INTENTIONAL and DIVERGES from the Ollama/OpenAI two-path rule (where kiro-native renders as `[tool: <name>]\n` thought-text only). The rationale:
- Anthropic's protocol has NO `[tool: <name>]` text convention. There is no text-narration equivalent.
- `@anthropic-ai/sdk` MessageStream parser expects native `tool_use` blocks; emitting text-only would silently lose tool-call semantics for loop24-client.
- D-07 was a deliberate, locked decision in CONTEXT.md predating REVIEWS.md.

Therefore, this slice DOES populate `Message.ToolCalls` and `Message.Content[ContentKindToolUse]` from kiro-native `ChunkKindToolCall` chunks — but via an ANTHROPIC-LOCAL aggregator (not via the generic Slice 1 collect.go path, which was revised to drop ChunkKindToolCall per the two-path rule). The local aggregator lives in `internal/adapter/anthropic/collect.go` (new file — wraps `engine.Run` directly and aggregates the chunk stream including `ChunkKindToolCall` into `Message.ToolCalls` + `ContentKindToolUse` parts, for the Anthropic surface only).
