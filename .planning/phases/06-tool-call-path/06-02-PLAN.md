<objective>
Ollama vertical slice for Phase 6. Closes TOOL-01 (Ollama plain-object args side per SC #2 — for COERCE-synthesized tool_calls only) and TOOL-02 (Ollama coerce side per D-01, BOTH non-streaming AND streaming per REVIEW HIGH #1) end-to-end through the Ollama surface: canonical → engine.CoerceToolCall hook-in (non-streaming + streaming end-of-stream) → render (non-streaming) → ndjson emitter (streaming) → unit + property tests. TOOL-03 (Ollama tools normalization) is already complete from Phase 2's forward seam (`internal/adapter/ollama/wire.go:321-330`) — this plan adds a regression test to lock it in.

**Two-path rule enforcement (REVIEW HIGH #2):** Per CONTEXT D-03/D-05, kiro-native `ChunkKindToolCall` renders ONLY as `[tool: <name>]\n` thought-text NDJSON narration — the done:true line carries `message.tool_calls[]` ONLY when populated by `engine.CoerceToolCall` (the coerce-from-text path). The previous plan's "kiro-native → final-line tool_calls aggregator" path is removed entirely.

**Streaming coerce path (REVIEW HIGH #1):** Ollama defaults to streaming (the `stream` field defaults to `true` and is commonly omitted). LangFlow/LangChain JSON-as-text compatibility — the load-bearing reason `coerceToolCall` exists — requires that the streaming path also run coerce. New flow:
1. The NDJSON emitter buffers assistant text deltas (`ChunkKindText`) into a `strings.Builder` AND emits them through the wire as normal.
2. On stream close, the emitter builds a synthetic `*canonical.ChatResponse{Message: {Content: [{Kind: ContentKindText, Text: buffered}]}}` and calls `engine.CoerceToolCall(req, syntheticResp)`.
3. If coerce returns `true` (text was a valid tool call), the synthetic resp now has `Message.ToolCalls[]` populated and `Content[].Text == ""`. The done:true line MUST carry `message.tool_calls[]` AND the wire MUST NOT contain the unconverted text deltas before it.
4. If coerce returns `false`, no tool_calls on the done line; the buffered text already streamed through is final.

**The text-replacement problem on streaming coerce:** because text deltas already wrote to the wire BEFORE the stream-end coerce decision, on a coerce-hit the client has a content+tool_calls inconsistency. Resolution per Node parity + REVIEW: when streaming + tools present + suspected JSON-shaped text starts arriving, the emitter BUFFERS the text instead of emitting it per-delta. The buffer is released as a single text NDJSON line ONLY at stream close AND ONLY IF coerce returned false. If coerce returned true, the buffered text is DISCARDED and only the done line with tool_calls is emitted. The heuristic for "looks like JSON" is `strings.HasPrefix(strings.TrimSpace(accumulated), "{") || strings.HasPrefix(strings.TrimSpace(accumulated), "```")` (matches what coerce will try to parse). Non-JSON-looking text streams through deltas normally (no buffering, no end-of-stream coerce).

Purpose: After this plan, an existing LangFlow flow pointing at `/api/chat` with `tools: [...]` receives correct tool_calls in Ollama's native wire shape for BOTH paths AND BOTH stream modes:
- Non-streaming: kiro-native renders inline as `[tool: <name>]\n` text (no tool_calls); coerce-from-text produces tool_calls.
- Streaming (default): same shape per-frame; coerce-from-text produces tool_calls on the done line via the buffered-text-then-end-of-stream-coerce path.

Output: Two new test additions (render_test.go extension, ndjson_test.go extension) plus one new handlers test (TestHandlers_DefaultStreamOmitted_GoesToStreaming). Four modified source files (wire.go, handlers.go, render.go, ndjson.go). Zero new dependencies. The slice runs in parallel with Slices 3 (OpenAI) and 4 (Anthropic) because all three share only the canonical pivot from Slice 1 — no file-level overlap.
