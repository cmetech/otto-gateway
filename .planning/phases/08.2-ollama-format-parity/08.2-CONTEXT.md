# Phase 08.2: Ollama `format` Parity - Context

**Gathered:** 2026-06-03
**Status:** Ready for planning
**Source:** Manual scoping conversation (Node-shim parity gap discovery)

<domain>
## Phase Boundary

Phase 08.2 closes a Node-shim parity gap discovered while reviewing fixes that
just landed in the JavaScript Ollama shim
(`../gitlab.rosetta.ericssondevops.com/loop_24/acp_server` →
`acp-ollama-server-win.js`). The Node shim is the production gateway for
LangFlow / `langchain-ollama` (`ChatOllama`) clients today. otto-gateway is
its replacement. Three fixes landed in Node in a single commit; an audit of
the equivalent otto-gateway code paths (see scoping notes below) showed:

- **Fix 3 (duplicate streaming output)** — already correct in Go
  (`internal/adapter/ollama/ndjson.go:599-609` + `render.go:32-34`). Final
  NDJSON frame's `message.content` is empty as Ollama spec requires.
- **Fix 2 (markdown fence stripping)** — already correct (conservative,
  whole-text-only) at `internal/engine/coerce.go:231-264`, but currently
  only wired on the tool-call coerce path — not on the JSON-format response
  path (because the JSON-format path itself doesn't exist).
- **Fix 1 (JSON-mode prompt steering)** — **N/A as a bug, but a feature
  gap**. The Ollama `format` field is parsed into a `json.RawMessage` at
  `internal/adapter/ollama/wire.go:25,119` and then **discarded**. There is
  no canonical-layer signal, no prompt steering, no schema awareness.

Net-new work in Phase 08.2 closes Fix 1's gap and wires up the existing
fence-stripper on that same path (Fix 2 reuse). Together this produces
behavioral parity with the Node shim for LangFlow flows that set
`format:"json"` or `format:<JSON schema>`.

**Why parity matters now (not later).** The project's load-bearing v1
invariant from `PROJECT.md`: *"a LangFlow flow pointing at `/api/chat` …
receives correct streamed responses."* A LangFlow flow author who enables
structured output on a `ChatOllama` node and points it at otto-gateway
today gets unfenced/unsteered output — the downstream JSON parser fails,
and from their perspective "the new gateway broke my flow." The Node shim
has been carrying this steering for production users; the replacement
must inherit it on day one of the cutover.

**Specifically, when `format` is present, the Node shim:**

1. Appends a verbatim **GEN_RULES** block to the system prompt:

   > Generate the COMPLETE result for EVERY item requested in this single
   > response — do not summarize, truncate, abbreviate, or omit items. Do
   > NOT add any prose, preamble, commentary, or follow-up questions, and
   > do NOT offer to export, save, or write to a file. Output the data
   > directly.

2. (If `format` is an object) embeds a schema-description line so the
   model sees the expected shape.

3. Conservatively strips a wrapping ```` ```json … ``` ```` fence from the
   final response text before returning it to the client.

Phase 08.2 reproduces this in Go, leveraging the existing canonical
seam (`canonical.ChatRequest.Format` at `chat.go:245`, currently dormant)
and the existing conservative `stripFences` helper (`engine/coerce.go:231`).
The steering itself lands as a new `PreHook` so it composes with the
Phase 8 hook chain — one place to enforce policy.

**Why a hook (not an adapter-local injection).** From `PROJECT.md`'s
Core Value: *"any guardrail (auth, rate-limit, content moderation, schema
validation, audit) defined once on the canonical request type applies
uniformly to all three [surfaces]."* JSON-format steering is precisely
that kind of policy. By living on the canonical request, the day the
OpenAI or Anthropic adapter populates `Format` (when their respective
`response_format` parameters get wired), the steering applies automatically.
Today only Ollama populates `Format`, but the architecture is
surface-agnostic from day one.

**Out of scope (locked):**

- **Streaming + `format:"json"` fence-stripping.** The Node shim's Fix 2
  does not strip mid-stream either — fences only land mid-delta in rare
  edge cases, and stripping requires buffered re-emission that breaks the
  streaming contract for clients that don't care about JSON. Almost all
  LangFlow `format:"json"` flows are non-streaming. Tracked in Deferred
  Ideas; revisit if a streaming consumer reports the issue.

- **OpenAI / Anthropic `response_format` wiring.** Both surfaces have
  native sampler-level JSON modes via Claude through `kiro-cli`, and
  their adapters do not currently populate `canonical.Format`. The hook
  is built surface-agnostic so the day those adapters wire it up,
  steering applies for free — but the wiring is a separate phase.

- **Sampler-level JSON constraint at the upstream.** `kiro-cli` (ACP)
  does not expose a `max_tokens` or JSON-mode sampler knob. Prompt-only
  steering is the v1 lever; this is the same constraint the Node shim
  operates under. We are not modifying `kiro-cli` or the ACP wire.

- **Schema validation of model output.** The steering is best-effort; we
  do not introspect the returned text against `Format.Schema` and reject
  mismatches. Downstream consumers (LangFlow components, application
  code) own validation. Adding gateway-side validation is a future hook.

- **New requirement IDs.** This is parity closure against the Node shim
  that otto-gateway is replacing — REQ-OLLAMA-01 is the existing
  "LangFlow zero-reconfig" requirement; Phase 08.2 strengthens that
  invariant rather than introducing new IDs.

- **Touching the existing `engine.coerce.go` tool-call coerce path.**
  That path handles "model emitted plain-JSON-as-text where a tool-call
  was expected" (Phase 6). It is unrelated to `format:"json"` requests
  and must remain byte-identical. Phase 08.2 reuses `stripFences` as a
  helper but does not modify the coerce logic itself.

<decisions>
## Implementation Decisions

### D-01: Reuse `canonical.ChatRequest.Format` — do not widen
`canonical.Format` already exists at `internal/canonical/chat.go:245` as
a Phase-3-era dormant forward-design seam (`Type string`, `Schema
map[string]any`). Phase 08.2 populates it from the Ollama wire and reads
it in the hook + render. **No new canonical fields.** No widening of
`Format` itself unless we discover the Schema map can't hold a value the
Node shim accepts — `interface{}`-typed map already accepts any decoded
JSON.

### D-02: Wire-decode in the Ollama adapter
The Ollama `format` field is `json.RawMessage` today at `wire.go:25` (chat)
and `wire.go:119` (generate). Decode logic lives in the Ollama wire-lift
helper (whatever produces the `canonical.ChatRequest` from the wire
struct). Both endpoints share the same logic — extract once, call from
both lift functions.

**Accepted shapes:**
- `"format": "json"` → `&Format{Type: "json"}`
- `"format": {<object>}` → `&Format{Type: "json_schema", Schema: <decoded map>}`
- `"format": ""` → `nil` (treat empty string as omitted)
- `"format": null` → `nil`
- Omitted → `nil`

**Rejected shapes** (return 400 at the wire layer with a clear error):
- `"format": "<anything other than 'json'>"` — Ollama spec only allows
  the bare string `"json"`. Other strings indicate client bug; failing
  fast beats silent acceptance.
- `"format": <non-object, non-string>` (e.g., number, array, bool).

### D-03: Steering text matches Node verbatim
The GEN_RULES block is copied byte-for-byte from the Node shim's fix so
LangFlow flows observe identical client-visible behavior. If we later
discover the rules can be tightened for kiro-cli specifically, that's a
follow-up phase — initial parity uses the proven text.

### D-04: Schema description format (Type == "json_schema")
When `Format.Type == "json_schema"` AND `Format.Schema != nil`, append
an additional line after GEN_RULES:
`The output must match this JSON schema: <compact-marshaled-JSON>`

We use `json.Marshal` (not `json.MarshalIndent`) for the schema so the
prompt stays compact. The schema map is already normalized JSON shape
(`map[string]any` from the wire decode), so marshaling round-trips
deterministically.

### D-05: Steering is a `PreHook`, not an adapter-local mutation
New package: `internal/plugin/jsonformat/` (mirrors the
`internal/plugin/pii/` layout from Phase 8). Single hook:
`JSONFormatSteeringHook` implementing `engine.PreHook`.

Behavior:
- `Pre(ctx, req)`:
  - If `req.Format == nil`: return `req, nil, nil` (pass-through, byte-identical).
  - Else: build the steering text, append to `req.System` with a single
    `\n\n` separator if `System` is non-empty (single newline otherwise),
    return the mutated `req`.

The hook does NOT short-circuit. It mutates and forwards.

**Why a separate hook and not folded into an existing one:** The Phase 8
hook architecture is composable on purpose; folding steering into
`PIIRedactionHook` or any other existing hook would couple two unrelated
policies and break the `GET /health/hooks` introspection contract.

### D-06: Env knob `JSON_FORMAT_STEERING_ENABLED` (default `true`)
Mirrors the `PII_REDACTION_ENABLED` env-knob shape from Phase 8 but with
the opposite default — PII redaction defaults off (opt-in privacy
feature); JSON-format steering defaults **on** because the v1 goal is
Node-shim parity and the Node shim does this unconditionally.

The hook reads the env at construction time (in `main.go`'s wiring),
not per-request. Hot-reload is out of scope.

When disabled, the hook is still registered (visible at
`/health/hooks`) but `Pre` becomes pass-through. This mirrors the
PII pattern and keeps the chain ordering stable across config
changes — operators see "JSON format steering: disabled" in the
hook introspection and know what behavior to expect.

### D-07: Hook chain order
Register the steering hook **after** `RequestIDHook` and `AuthHook`
(so auth short-circuit still wins) but **before** `PIIRedactionHook`
(so the steering text itself does not get accidentally redacted if it
ever contained something that looked like PII to a recognizer — defense
in depth, even though the current GEN_RULES text contains no PII
shapes). Specific position TBD by the planner reading
`cmd/otto-gateway/main.go` — must land between Auth and PII in
`ENABLED_HOOKS` registration.

### D-08: Fence-strip at the Ollama render layer, non-streaming only
Existing helper: `engine/coerce.go:231` `stripFences(text string) (string, bool)`.
The helper is package-private to `engine` today; expose it (or a thin
wrapper) so the Ollama render path can call it.

**Two options** for exposure, planner decides:
- (a) Rename `stripFences` → `StripFences` (export). Cheapest.
- (b) Add a new public helper `engine.StripJSONFences` that wraps the
  private one and documents the JSON-mode use case. Cleaner API surface
  but two names for the same behavior.

Default to (a) unless the planner finds a reason to prefer (b). Either
way, do NOT duplicate the fence-strip logic — the conservative
"entire-trimmed-body-is-one-fence" behavior is load-bearing and must
have a single implementation.

**Apply site:** Ollama non-streaming `/api/chat` and `/api/generate`
render paths in `internal/adapter/ollama/handlers.go`, immediately
before passing the assembled text to the wire renderer, gated on
`req.Format != nil`.

**Streaming path:** No fence-strip applied. See Out-of-Scope above.
The deferred-items doc will note this so a future operator who reports
"fences in streamed JSON" has the documented context.

### D-09: Surface-agnostic by intent, Ollama-only by current population
The hook fires on `req.Format != nil`. It does not inspect surface or
adapter identity. Today only the Ollama adapter populates `Format`, so
in practice the hook only affects Ollama requests. The day OpenAI's
`response_format` or Anthropic's `response_format` wires through, the
steering applies to those surfaces too without code change — but that
wiring itself is out of scope.

<specifics>
## Specific Ideas

**Verbatim GEN_RULES text** (D-03). Use exactly this, including the line
breaks shown:

```
Generate the COMPLETE result for EVERY item requested in this single response — do not summarize, truncate, abbreviate, or omit items.
Do NOT add any prose, preamble, commentary, or follow-up questions, and do NOT offer to export, save, or write to a file. Output the data directly.
```

(Single em-dash in the first line, matching the Node source. Two newlines
between this block and any prior `System` content; one newline within.)

**Schema description line format** (D-04). Compact form, no pretty-print:

```
The output must match this JSON schema: {"type":"object","properties":{...}}
```

**Hook package layout** (D-05). Mirror `internal/plugin/pii/`:

```
internal/plugin/jsonformat/
  jsonformat.go        — Hook struct, Pre method, New() constructor
  jsonformat_test.go   — unit tests: nil Format pass-through, json type, json_schema type, idempotent on repeated Pre, empty System, non-empty System with separator
  testmain_test.go     — goleak guard (matches project convention)
```

**Fence-strip apply point** (D-08). In `handlers.go` `handleChat` (line ~172
is where the canonical response text is assembled today for the non-streaming
path — confirm during planning). Same pattern in `handleGenerate`.

**Integration test fixture.** A fake-engine response of:

````
```json
[
  {"name": "Item 1", "id": 1},
  {"name": "Item 2", "id": 2}
]
```
````

through `/api/chat` with `"format": "json"` and `"stream": false` should
arrive at the wire as the unwrapped JSON array. A second test with no
`format` field on the same fake response should preserve the fences
(no strip applied).

<deferred>
## Deferred Ideas

- **Streaming + `format:"json"` fence-stripping.** Requires buffered
  delta re-emission. Revisit only if a streaming consumer reports
  fenced JSON in deltas as a real problem.

- **OpenAI `response_format` → `canonical.Format` wiring.** The OpenAI
  adapter currently does not populate `Format`. When OpenAI users want
  structured output through otto-gateway, wire the adapter and the
  steering applies for free.

- **Anthropic `response_format` → `canonical.Format` wiring.** Same
  rationale as OpenAI. Anthropic's tool-use JSON pattern is the
  current workaround.

- **Gateway-side schema validation.** Today the steering is best-effort;
  we don't validate the model's output against `Format.Schema` and
  reject. A future `JSONSchemaValidationHook` (Post) could do this with
  configurable strict/lenient modes.

- **Per-recognizer steering text overrides.** A future operator might
  want to inject domain-specific language into the steering block.
  Adding a config slot for "append custom system text when Format is
  set" is a small but real future enhancement.

- **`/health/hooks` surface for the new hook.** The Phase 8 endpoint
  reads the registered chain; the new hook will appear automatically
  with `name: "JSONFormatSteeringHook"`, `kind: "Pre"`, `enabled:
  <env-derived>`, `config: {default_on: true}`. No additional work
  needed here, but the planner should verify the introspection JSON
  comes out correctly.

</deferred>
