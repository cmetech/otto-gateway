<objective>
Fix four confirmed Medium reliability findings in the Pool/ACP subsystem:

P-4 (REL-POOL-04): A stalled chunk consumer blocks the readLoop goroutine by holding it in stream.push on c.clientCtx. The readLoop can't dispatch pings, which escalates to SIGKILL on a healthy kiro-cli worker. Fix: pass the per-request stream context (s.ctx) to stream.push in handleNotification so a stalled consumer fails only its own request.

P-5 (REL-POOL-05): Entry.LastUsed is written in entry_acp.go:MarkUsed without a mutex, while it is read concurrently at registry.go:358 also without holding r.mu. This is a data race that prevents go test -race from passing tree-wide. Fix: convert Entry.LastUsed to an atomic.Int64 (UnixNano) field named lastUsedNs with a LastUsed() time.Time accessor.

P-6 (REL-POOL-06): internal/acp/pool_pgid_windows.go:killProcessGroup returns nil unconditionally — kiro-cli children are orphaned on Windows gateway shutdown. Fix: implement killProcessGroup using taskkill /T /F with a 5-second context timeout.

O-1 (REL-CFG-04): The pool's acquire select parks silently when all slots are busy. Operators see "gateway silently stopped answering" with zero diagnostic signal. Fix: emit Warn("pool: waiting for free slot", "busy", ..., "size", ...) once per saturation episode using a p.warnOnce sync.Once field, reset in Pool.Close().

Additionally: Plan 16-01 owns plumbing pool.LastProgressAt() — an atomic.Int64 UnixNano field advanced on every streamed chunk, every ping ack, and every slot release — as the data source for Plan 16-02's D-05a degraded rule in health.go.

Purpose: Restores -race trust gate, closes Windows orphan-process gap, makes pool saturation diagnosable from the log alone, and separates per-request liveness from readLoop liveness.
Output: Four atomic commits (one per finding), each pairing the source fix with removal of the corresponding t.Skip in the Phase 14 regression test.
</objective>
