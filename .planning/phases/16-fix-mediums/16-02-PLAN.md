<objective>
Fix two confirmed Medium HTTP surface findings and plumb the D-05 cross-plan PoolStats.Status enum:

H-4 (REL-HTTP-04): Body-reading POST handlers (chat-body surfaces: /v1/chat/completions, /v1/messages, /v1/completions, /api/chat, /api/generate, /api/embed, /api/embeddings, /v1/embeddings) have no deadline on io.ReadAll / JSON decode. A stalled upload parks the handler goroutine indefinitely via TCP keepalive. Fix: add a new env var HTTP_BODY_READ_TIMEOUT_SEC (default 30, <= 0 is a boot error), wrap the body-read phase using time.AfterFunc + r.Body.Close() pattern so SSE response writes are unaffected.

H-5 (REL-HTTP-05): The admin tailer's line cap at tail.go:402 only fires when there is no trailing newline. A log line that is exactly at the cap and ends with \n escapes truncation and fans out uncapped through the ring buffer and SSE stream. Fix: enforce TailerMaxLineBytes on ALL lines regardless of newline termination.

D-05 cross-plan (for Plan 16-04 T-5): Extend PoolStats in internal/server/health.go with Status string (ok/degraded/exhausted). Status computation: "exhausted" when pool.IsExhausted() is true; "degraded" when Busy == Alive == Size AND time.Since(pool.LastProgressAt()) > 30s (using the LastProgressAt() method added by Plan 16-01); "ok" otherwise. Plan 16-04 depends on this field existing in the /health JSON.

Purpose: Bounds body-read hangs without touching SSE writes; fixes a silent memory-growth vector in the admin UI; publishes pool health enum for tray consumption.
Output: Three atomic commits (H-4, H-5, D-05 Status), each unskipping its Phase 14 regression test.
</objective>

|----------|-------------|
| client → POST /api/chat (and sibling endpoints) | Body content is untrusted; body-read deadline bounds time the goroutine is held by stalled upload |
| SSE stream → admin log-tail client | Log line content passes through tailer; cap prevents memory amplification from crafted large lines |
| GET /health → tray/operator | Pool status enum exposes operational state; values are computed internally from pool counters |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-02-01 | Denial of Service | server.go — body-read deadline via time.AfterFunc + r.Body.Close() | mitigate | HTTP_BODY_READ_TIMEOUT_SEC with <= 0 boot-error prevents zero-timeout DoS amplification; default 30s is conservative. Slow-reset attack: closing r.Body triggers a read error at the handler; connection is torn down normally — no amplification path |
| T-16-02-02 | Denial of Service | admin/tail.go — multi-MB line bypass | mitigate | TailerMaxLineBytes cap enforced on ALL lines post-fix; a crafted chat-trace line cannot fan out beyond the cap in memory or SSE stream |
| T-16-02-03 | Information Disclosure | health.go PoolStats.Status | accept | Status enum values (ok/degraded/exhausted) reflect pool operational state; no secrets, no PII, no path information |
