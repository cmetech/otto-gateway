Fix three confirmed Medium Config findings, update CLAUDE.md env-var documentation, and own the config-layer parsing for HTTP_BODY_READ_TIMEOUT_SEC (the H-4 env var whose server-side wrapper logic lives in Plan 16-02):

C-1 (REL-CFG-01): POOL_SIZE, SESSION_MAX, SESSION_TTL_MS, SESSION_TICK_INTERVAL_MS, CHAT_TRACE_MAX_AGE_DAYS are silently coerced when negative/zero by applyDefaults in pool.Config and session.Config. There is no named boot error. Fix: add sign checks in config.Load() immediately after each getEnvInt/getEnvDuration call, following the exact STREAM_IDLE_TIMEOUT_SEC pattern at config.go:362-368. POOL_SIZE also gets a sanity upper bound of 256. All checks accumulate into the existing errs slice so they land in the errors.Join at the end of config.Load().

C-2 (REL-CFG-02): PING_INTERVAL is parsed by getEnvDuration without a sign check. Negative/zero values pass through to time.NewTicker in acp/client.go:505, which panics with "non-positive interval for NewTicker". Fix: add a <= 0 sign check immediately after the PING_INTERVAL getEnvDuration call in config.Load(). No change to acp/client.go — if config.Load() guards correctly, the ticker never receives a bad value.

C-3 (REL-CFG-03): EMBEDDING_MODEL_DEFAULT is documented in CLAUDE.md but never read anywhere in the binary. Operators setting it receive no feedback. Fix (per D-03): emit one Warn in cmd/otto-gateway/main.go after config.Load() returns if the env var is set. Update CLAUDE.md to mark it "reserved, not yet implemented". Add HTTP_BODY_READ_TIMEOUT_SEC (net-new in v1.9) to CLAUDE.md env-var list.

H-4 config owner (REL-HTTP-04 partial): Plan 16-02 owns the server-side body-read wrapper in server.go. THIS plan owns the config.go side: parse HTTP_BODY_READ_TIMEOUT_SEC via getEnvInt with default 30, add a <= 0 sign check (boot error matching C-1/C-2 posture), set Config.BodyReadTimeout time.Duration = time.Duration(n) * time.Second. Wire in main.go: server.Config.BodyReadTimeout = cfg.BodyReadTimeout. This ensures config.go is committed here (Wave 1) before Plan 16-02 (Wave 2) references the pre-populated field on server.Config.

Purpose: Matches the fail-fast posture established by STREAM_IDLE_TIMEOUT_SEC; prevents a raw goroutine panic from PING_INTERVAL misconfiguration; removes a silent doc/code mismatch for embeddings; provides the HTTP_BODY_READ_TIMEOUT_SEC duration to Plan 16-02 via the Config struct.

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-05-01 | Denial of Service | config.go — fail-fast on bad knobs causes boot failure | accept | Intentional: misconfigured deployments fail loudly at boot rather than silently misbehaving at runtime; operator controls env vars |
| T-16-05-02 | Information Disclosure | main.go logger.Warn "value" field for EMBEDDING_MODEL_DEFAULT | accept | Value is the operator's own model name string (e.g., "all-MiniLM-L6-v2"); not user-supplied; logged at Warn level in structured log; no PII or secret exposed |

| T-16-05-04 | Denial of Service | config.go — HTTP_BODY_READ_TIMEOUT_SEC <= 0 boot error | accept | Intentional: zero or negative body-read timeout is a config error; fail-fast is correct; default 30s is conservative |
| T-16-05-SC | Tampering | No npm/pip/cargo installs in this plan | accept | Pure Go stdlib + doc changes; no new dependencies |
