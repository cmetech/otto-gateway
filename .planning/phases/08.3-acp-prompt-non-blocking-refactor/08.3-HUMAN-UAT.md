## Follow-up: Phase 08.3.2 — PII Smoke Test Methodology Fix (2026-06-04)

### Methodology change summary

The 2026-06-03 splunk-box HUMAN-UAT pass above confirmed Phase 8.3's load-bearing
contract (`engine.prompt.sent` semantic shift, three-surface 200-not-500 round
trips, no `WARN acp: ping failed`). However, the `scripts/test-pii.ps1 pii`
script's `round-trip decrypt: '<value>' present in response` assertion failed
NOT because of an encrypt/decrypt pipeline bug but because Claude (via
`kiro-cli`) refuses to verbatim-echo PII-shaped fixture data on safety grounds.
The "no ciphertext leak" assertion was passing all along — the encrypt → forward
→ decrypt pipeline is healthy. The failure was an LLM-cooperation dependency in
the smoke methodology, orthogonal to Phase 8.3.

Phase 08.3.2 replaces that LLM-cooperation dependency with a deterministic
`fake-kiro-cli` worker accessed via the existing `KIRO_CMD` env var. Both
`scripts/test-pii.ps1` and `scripts/test-pii.sh` now default to a new `-Mode fake`
/ `--mode fake` switch that writes JSON-RPC `session/update` agent_message_chunk
notifications to an NDJSON file the fake replays, eliminating the bitrot class.
The legacy LLM-cooperation path remains available under `-Mode live` /
`--mode live` with a deprecation banner.

### Operator verification steps (Windows)

1. Pull v1.9.4 on the Windows splunk box. Confirm `git log --oneline -1` shows the
   v1.9.4 tag or its head commit.
2. `make build-fake-kiro` (or unzip the v1.9.4 release if shipped). Produces
   `bin\fake-kiro-cli.exe`.
3. Edit `.otto-gw.overrides.env` to set both:
   - `KIRO_CMD=C:\otto-gateway\bin\fake-kiro-cli.exe` (absolute path)
   - `OTTO_FAKE_KIRO_NOTIFICATIONS_FILE=%TEMP%\otto-pii-notifs.ndjson`
4. Start gateway: `.\scripts\otto-gw.ps1 start`. Confirm `/health` returns 200.
5. Run: `.\scripts\test-pii.ps1 pii`. Default mode is now `fake` (no flag needed).
6. Verify the final line reads `0 check(s) failed` (or `all checks passed`)
   and the script exits 0.
7. Optional: run `.\scripts\test-pii.ps1 pii -Mode live` against a real `kiro-cli.exe`
   to observe the documented LLM-policy failure mode — expected output
   includes the deprecation banner plus `round-trip decrypt: '<value>' NOT in
   response` failures. This confirms the live path's known-failing state.

### Operator verification steps (POSIX)

1. From repo root on macOS or Linux: `make build-fake-kiro`. Produces
   `bin/fake-kiro-cli`.
2. Set the same two env vars in your gateway-launch environment:
   - `KIRO_CMD=$(pwd)/bin/fake-kiro-cli`
   - `OTTO_FAKE_KIRO_NOTIFICATIONS_FILE=$TMPDIR/otto-pii-notifs.ndjson`
3. Start gateway: `./scripts/otto-gw.sh start`. Confirm `/health` returns 200.
4. Run: `./scripts/test-pii.sh pii`. Default mode is `fake` (no flag needed).
5. Verify the final line reads `0 check(s) failed` (or `all checks passed`)
   and the script exits 0.
6. Optional: `./scripts/test-pii.sh pii --mode live` to observe the same
   live-LLM-policy failure mode the PowerShell sibling documents.

### `-Mode live` / `--mode live` known-failure note

The legacy live path remains supported but will fail today's
`round-trip decrypt: '<value>' NOT in response` assertion against current
Claude releases. Both scripts emit a deprecation banner pointing at this
phase on invocation. Operators who want to verify a real `kiro-cli` end-to-end
should use the standard Phase 8.3 smoke check (`engine.prompt.sent` +
`engine.prompt.completed` log markers + a non-PII chat round-trip) rather
than relying on PII verbatim-echo.
