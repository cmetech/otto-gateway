## Phase Goal

**As an** OTTO Gateway operator with kiro-cli workers behind the Anthropic / OpenAI / Ollama surfaces, **I want to** issue any prompt — including PII-encrypt mode round-trips that stream >64 ACP `session/update` chunks before the `session/prompt` response — **so that** the gateway returns a correct streamed answer instead of wedging until `STREAM_IDLE_TIMEOUT_SEC` fires a 500.

<objective>
Eliminate the chunk-buffer-overflow deadlock in `internal/acp/client.go:Prompt()` (current source: lines 696-803). Today `Prompt()` blocks on `respCh` until the `session/prompt` response arrives; `engine.Run` calls it synchronously and cannot drain `Stream.Chunks` until `Run` returns. The 64-slot buffer overflows on any non-trivial response, `push()` blocks the readLoop, the ping reply / response frame can't be parsed, and the request stays wedged until `STREAM_IDLE_TIMEOUT_SEC` (default 100s) fires a 500 — symptom reproduced live on Windows v1.9.2 against `scripts/test-pii.ps1` with `PII_REDACTION_MODE=encrypt`.

**Fix shape** (lift-and-shift; minimum-diff refactor):
1. `Prompt()` returns `(*Stream, error)` immediately after `c.send(...)` accepts the JSON-RPC payload.
2. A new `awaitPromptResult` per-prompt goroutine (registered with `c.wg`, named for goroutine-dump visibility) owns the existing two-arm select against `respCh` and `ctx.Done()`, parses `result.stopReason`, clears `c.activeStream` under `c.streamMu`, and finalizes via `stream.close(...)`.
3. `defer c.disp.cancel(id)` moves from `Prompt()` into the goroutine (Pattern P3); the explicit `c.disp.cancel(id)` on the send-failure path stays (Pitfall 4).
4. A new `engine.prompt.completed` DEBUG log line fires from inside the goroutine immediately before the happy-path `stream.close` — carries `session_id`, `chunks`, `stop_reason`.
5. `stream.go` docstring loses the "callers MUST drain concurrently" warning (Patterns P10, P11) — drainage MAY happen concurrently for throughput, but is no longer required for correctness.

**Why TDD:** the deadlock IS the assertion. The RED test (`TestIntegration_Prompt_OverflowsBuffer_DoesNotDeadlock`) drives the fake server to emit ≥128 `session/update` chunks BEFORE its `session/prompt` response and asserts (a) `Prompt()` returns within 100ms, (b) all 128 chunks drain, (c) `Stream.Result()` returns `StopEndTurn`, (d) `goleak.VerifyNone(t)` passes. Against the current synchronous `Prompt()` body at `client.go:696-803`, the test MUST fail — the load-bearing falsification surface that locks in the regression catch.

**Purpose:** Close ACP-02 / ACP-03 against the live ACP spec — re-validation, not new IDs. Remove the documented footgun ("Calling Prompt synchronously and draining Chunks afterward only works when the total chunk count fits in the 64-slot buffer") that the existing `client.go:701-725` docstring explicitly calls out.

**Output:** A single atomic vertical slice landed across three RED/GREEN/REFACTOR commits + a separate Windows operator regression sign-off checkpoint (commit-less).
</objective>
