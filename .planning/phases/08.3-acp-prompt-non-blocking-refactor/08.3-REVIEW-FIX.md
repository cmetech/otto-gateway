### WR-03: Sleep-based synchronization in two whitebox tests is flake-prone under CI race load

**Commit:** `756d8f9`
**Applied fix:** Added a whitebox test helper `waitForPendingRegistered(t *testing.T, c *Client, min int, timeout time.Duration)` that polls `len(c.disp.pending)` under `c.disp.mu` with a 1ms inner interval and a 5s deadline, then fails fast with a diagnostic on timeout. ## Skipped Issues

### WR-04: Send-failure path race between `c.streamMu` clear/reset and `handleNotification`

**File:** `internal/acp/client.go:730-752`
**Reason:** Deferred — pre-existing pattern, out of Phase 8.3 scope per phase-specific guidance. The REVIEW.md itself notes "This pre-dates Phase 8.3 — the same pattern is in the pre-refactor synchronous body — but Phase 8.3 had an opportunity to fix it and didn't." Fixing it here would change the `c.activeStream`-set ordering relative to `c.send`, expanding Phase 8.3's blast radius beyond the deadlock-removal contract that Phase 8.3 was scoped to deliver. The review also acknowledges "the race window is effectively empty" in practice: `c.send` only errors on `ctx.Done()` / `c.clientCtx.Done()`, and `handleNotification` requires a kiro-cli `session/update` notification to race the send, which the protocol does not emit before the `session/prompt` ACK. Recommend a follow-up housekeeping phase (or standalone issue) to (a) move `c.activeStream = stream` to AFTER `c.send` returns nil, OR (b) make `Stream.push()` non-blocking on send-failure. The first is preferable; the integration test's `fake.promptSeen` synchronization already proves the reordering is scheduling-safe.
**Original issue:** Between `c.activeStream = stream` (set on line 731, lock released line 732) and the `c.send` returning error and re-clearing on lines 749-751, `handleNotification` can read `c.activeStream`, call `s.push(...)`, and — if the caller has walked away from a `(nil, err)` `Prompt` return — fill the 64-slot buffer and wedge the readLoop until clientCtx fires.
