### `internal/acp/client.go` — `Prompt()` rewrite + new `awaitPromptResult` goroutine

**Analog (in-file):** `readLoop`/`writerLoop`/`pingLoop` for goroutine lifetime; current `Prompt()` body for the lift-and-shift block.

**Pattern P1 — Goroutine registration via `c.wg.Add(1)` BEFORE `go` statement**

Source: `internal/acp/client.go:328-331` (subprocess constructor) — every long-lived ACP goroutine is `wg`-registered the same way:

```go
c.wg.Add(3)
go c.readLoop(clientCtx)
go c.writerLoop(clientCtx)
go c.pingLoop(clientCtx)
```

Mirror for `awaitPromptResult` (per-prompt — `Add(1)` per call, NOT in the constructor). Per Anti-Pattern §Pattern 1: `c.wg.Add(1)` MUST happen-before the `go` statement, never inside the goroutine itself (Go memory-model requirement).

**Pattern P2 — `defer c.wg.Done()` as the very first statement**

Source: `internal/acp/client.go:371` (readLoop), `:415` (writerLoop), `:440` (pingLoop) — uniform shape:

```go
func (c *Client) readLoop(ctx context.Context) {
    defer c.wg.Done()
    // ... loop body
}
```

Mirror exactly in `awaitPromptResult`. Place `defer c.wg.Done()` FIRST, then `defer c.disp.cancel(id)` second so cancel runs before `wg.Done()` releases `Close()`'s wait.

**Pattern P3 — Dispatcher entry lifetime via `defer c.disp.cancel(id)` (lift into goroutine)**

Source: `internal/acp/client.go:728-729` (current synchronous Prompt) — the defer pattern that must move into the goroutine:

```go
respCh := c.disp.register(id)
defer c.disp.cancel(id)   // <— moves into awaitPromptResult
```

Operational notes from `dispatcher.go:80-84`:
```go
func (d *dispatcher) cancel(id uint64) {
    d.mu.Lock()
    delete(d.pending, id)
    d.mu.Unlock()
}
```

Idempotent on the map (delete-of-missing-key is benign). Per Pitfall 4: the send-failure path in `Prompt()` (current `client.go:749-754`) must still call `c.disp.cancel(id)` explicitly because no goroutine is spawned on send failure. Two paths, exactly one cancel each.

**Pattern P4 — Lift-and-shift the select block (verbatim) from `client.go:756-803`**

Source: `internal/acp/client.go:756-803` (current synchronous response wait — moves wholesale into the new goroutine). The two-arm select shape:

```go
select {
case <-ctx.Done():
    c.disp.cancel(id)
    c.streamMu.Lock()
    c.activeStream = nil
    c.streamMu.Unlock()
    // Best-effort cancel notification (no id = notification).
    c.sendNotification(rpcNotification{
        JSONRPC: "2.0",
        Method:  "session/cancel",
        Params:  cancelParams{SessionID: sessionID},
    })
    return nil, fmt.Errorf("acp: prompt: %w", ctx.Err())
case frame := <-respCh:
    if frame.Error != nil {
        c.streamMu.Lock()
        c.activeStream = nil
        c.streamMu.Unlock()
        if frame.Error.Code == closeSentinelCode {
            return nil, ErrClientClosed
        }
        return nil, fmt.Errorf("acp: prompt: rpc error %d: %s", frame.Error.Code, frame.Error.Message)
    }
    var r promptResult
    if len(frame.Result) > 0 {
        if err := json.Unmarshal(frame.Result, &r); err != nil {
            c.cfg.Logger.Warn("acp: prompt result parse failed", "err", err)
        }
    }
    stop := parseStopReason(r.StopReason)
    s := stream
    c.streamMu.Lock()
    c.activeStream = nil
    c.streamMu.Unlock()
    s.close(&FinalResult{StopReason: stop}, nil)
    return s, nil
}
```

**Translation in the goroutine:**
- All `return nil, err` statements become `stream.close(nil, err); return` (the stream is the only sync point now).
- All `return s, nil` statements become `stream.close(&FinalResult{StopReason: stop}, nil); return`.
- The explicit `c.disp.cancel(id)` on the ctx-cancel branch becomes redundant — the new `defer c.disp.cancel(id)` covers it. Per Pitfall 4 §benign double-cancel, keeping the explicit call is also safe.
- Insert `engine.prompt.completed` Debug log emission immediately BEFORE the happy-path `stream.close(...)` (see Pattern P7).

**Pattern P5 — `c.streamMu` guards every `c.activeStream` write**

Source: `internal/acp/client.go:734-736` (set) and `:386-389` (readLoop defer reads under same mutex):

```go
// Writer side (Prompt setting up):
c.streamMu.Lock()
c.activeStream = stream
c.streamMu.Unlock()

// Reader side (readLoop defer):
c.streamMu.Lock()
s := c.activeStream
c.activeStream = nil
c.streamMu.Unlock()
```

The new goroutine writes `c.activeStream = nil` at three exit points (ctx-cancel, rpc-error, happy path). Each write MUST be under `c.streamMu.Lock()/Unlock()`. Per Pitfall 2: the mutex serialises against the `readLoop` defer; one side wins (writes nil), the other sees either the live pointer or nil — both correct.

**Pattern P6 — Best-effort `session/cancel` notification (lift verbatim)**

Source: `internal/acp/client.go:763-767` (current synchronous Prompt ctx-cancel path):

```go
c.sendNotification(rpcNotification{
    JSONRPC: "2.0",
    Method:  "session/cancel",
    Params:  cancelParams{SessionID: sessionID},
})
```

Underlying primitive at `internal/acp/client.go:489-502`:

```go
func (c *Client) sendNotification(notif any) {
    data, err := json.Marshal(notif)
    if err != nil {
        c.cfg.Logger.Warn("acp: marshal notification failed", "err", err)
        return
    }
    select {
    case c.writeCh <- data:
    case <-c.clientCtx.Done():
        // Client closed — drop notification.
    default:
        // writeCh full — drop notification (best-effort for cancel/grant).
    }
}
```

This is fire-and-forget — `default` arm guarantees the goroutine never blocks on the cancel emission. Two-owner cancellation (CONTEXT.md decision) is safe because `ACPClient.Cancel(sid)` is documented idempotent.

**Pattern P9 — Reading `Stream.ChunkCount` under `s.mu` (in-package access)**

Source: `internal/acp/stream.go:60-63, 85-87` — `ChunkCount` is package-private under `s.mu`:

```go
type Stream struct {
    // ...
    mu     sync.Mutex
    result *FinalResult
    err    error
}

// push() increments under the same mutex (line 85-87):
func (s *Stream) push(ctx context.Context, ch canonical.Chunk) error {
    select {
    case s.chunks <- ch:
        s.mu.Lock()
        s.result.ChunkCount++
        s.mu.Unlock()
        return nil
    // ...
}
```

Per RESEARCH.md Pattern 5 Option C: inside the new goroutine (same package), take `s.mu.Lock()` and read `s.result.ChunkCount` directly. No new exported API. The goroutine reads AFTER the response frame is observed — by RESEARCH.md Pitfall 6 the readLoop has already finished all preceding `push()` calls (sequential frame processing).

Optional alternative if planner prefers an accessor: add a small `(*Stream).ChunkCount() int` method that locks `s.mu` and returns `s.result.ChunkCount` — keeps the call site in `client.go` clean without exposing the field.

**Pattern P10 — Remove deprecated "MUST drain concurrently" warning**

Source: `internal/acp/stream.go:32-45` (current docstring) — delete the warning paragraph and rewrite the contract:

Current text (to remove):
```
// Concurrency contract (Phase 1.1 WR-01): the readLoop goroutine is the sole
// producer on Chunks. push() is a BLOCKING send guarded only by the client
// lifetime context, so a slow consumer of Chunks back-pressures the readLoop
// — and while the readLoop is blocked, no further frames are read from the
// subprocess pipe, including the session/prompt response that would close the
// stream. The internal channel is buffered (64 slots) to absorb short bursts,
// but if kiro-cli emits more than 64 chunks before the prompt response a
// non-draining consumer will deadlock until the client context is cancelled.
//
// Required usage: callers MUST drain Chunks concurrently with whatever
// goroutine is waiting on Prompt() to return. The fakeacp integration test
// (TestIntegration_FakeACP_E2E_MixedVariants) and the real-kiro round-trip
// test both run Prompt on one goroutine and the Chunks drain on another.
// Phase 2's HTTP adapters MUST follow the same pattern.
```

Replacement contract (Phase 8.3): drainage MAY be concurrent for throughput, but is no longer required for correctness. Prompt returns non-blocking after `c.send` is accepted; the per-prompt goroutine drives `stream.close(...)` independently of chunk drainage. Update the "Phase 1.1 WR-01" reference to "Phase 8.3".

Also remove the IMPORTANT block at `stream.go:50-52` (same warning, abbreviated form).

**Pattern P12 — `closeOnce` guards double-close (Pitfall 1 defense — no change)**

Source: `internal/acp/stream.go:105-121`:

```go
func (s *Stream) close(result *FinalResult, err error) {
    s.closeOnce.Do(func() {
        s.mu.Lock()
        if s.result == nil {
            s.result = &FinalResult{}
        }
        if result != nil && result.StopReason != canonical.StopUnknown {
            s.result.StopReason = result.StopReason
        }
        s.err = err
        s.mu.Unlock()
        close(s.chunks)
        close(s.done)
    })
}
```

Defends Pitfall 1 (`readLoop` EOF defer racing the goroutine's response-path close). The first call wins; both paths supply the same `(nil, ErrClientClosed)` on Close, so the merged final state is identical.

## Race-Condition Defense Matrix

Mapping the four pre-flagged race cases (from the pattern_mapping_context) to the patterns above:

| Race | Defense Pattern | Notes |
|------|-----------------|-------|
| 1. `Close()` races in-flight prompt goroutine | P8 (closeSentinelCode routing) + P12 (closeOnce) + S3 (goleak per-test) | `Close()` step 2 `failPending` runs before step 4 `wg.Wait` — sentinel arrives via respCh, goroutine exits via case-`<-respCh` arm, `stream.close` is `sync.Once`-guarded. |
| 2. `ctx` fires after `Prompt()` returns but before adapter watchdog stops | P4 (ctx-cancel arm in lift) + P6 (best-effort sendNotification) + engine.go:215-225 watchdog (unchanged) | Two-owner pattern, decision-locked in CONTEXT.md. `ACPClient.Cancel` documented idempotent at `engine.go:152-158`. |
| 3. `c.activeStream` write race | P5 + S5 (every mutation under `c.streamMu`) | Pre-existing mutex invariant — preserve, do not weaken. |
| 4. `disp.cancel(id)` called twice | P3 (`defer c.disp.cancel(id)` once in goroutine) + dispatcher.go:80-84 (idempotent map delete) | Per Pitfall 4: send-failure path in Prompt() body still calls explicitly (no goroutine spawned); happy path defers inside the goroutine. Two paths, one cancel each — but extra calls are benign map-deletes. |
