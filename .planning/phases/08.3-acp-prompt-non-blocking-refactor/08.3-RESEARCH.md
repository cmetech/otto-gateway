## User Constraints (from CONTEXT.md)

### Locked Decisions

**Cancellation ownership — two-owner, double-cancel-safe.** `engine.Run` keeps its `context.AfterFunc` watchdog (`internal/engine/engine.go:215-225`) that calls `e.cfg.ACP.Cancel(sid)` on `ctx.Done()`. The new per-prompt goroutine inside `acp.Client.Prompt()` is **also** cancel-aware — observes `ctx.Done()`, sends its own best-effort `session/cancel` notification (matching the current synchronous path at `client.go:763-767`), clears `c.activeStream`, and finalizes the stream via `stream.close(nil, ctx.Err())`. Justification: `ACPClient.Cancel(sid)` is already idempotent (engine.go:152-158 documents this); a duplicate `session/cancel` notification is fire-and-forget on kiro-cli's side; the defensive layer protects against the race where ctx fires after `Prompt()` returns but before the adapter reaches its watchdog-stop call. Single-owner was rejected because it would enlarge the engine.go diff and couple cancel timing to `engine.Run`'s return path — defeating the whole point of returning early.

**`engine.prompt.sent` log signal — two log lines.** Keep `engine.prompt.sent` as the early-dispatch signal — fires immediately after `c.send(...)` accepts the request payload (which is millisecond-scoped). Its meaning shifts from "response landed" to "send accepted". **Add `engine.prompt.completed`** — a new DEBUG line emitted by the per-prompt goroutine immediately before `stream.close(...)` succeeds. Fields: `session_id`, `chunks` (final `ChunkCount`), `stop_reason` (parsed `canonical.StopReason` string). Renaming `engine.prompt.sent` was rejected (breaks external log greps + downstream ChatTrace tooling). Moving the emission into the goroutine without adding the new line was rejected (loses the new early signal entirely — the same diagnostic gap that masked today's deadlock would persist).

**Test strategy — add overflow test, preserve mixed-variants.** Add a new dedicated test, `TestIntegration_Prompt_OverflowsBuffer_DoesNotDeadlock` (likely whitebox in `internal/acp/`, possibly mirrored as an integration test using `fakeacp`). Fake emits ≥128 `session/update` chunks **before** its `session/prompt` response. Assertions: (1) `client.Prompt(ctx, sid, blocks)` returns within ~100 ms; (2) ranging over `stream.Chunks` drains all 128 chunks without losing any; (3) `stream.Result()` returns `canonical.StopEndTurn`; (4) `goleak.VerifyNone` passes. Keep `TestIntegration_FakeACP_E2E_MixedVariants` unchanged — it already runs "Prompt-on-one-goroutine, drain-on-another"; under the new contract this still works (it's no longer required for correctness), so it doubles as a regression artifact for callers following the historical recipe. Goleak coverage: the new per-prompt goroutine must be added to whatever `goleak.VerifyNone` invocations exist; verify by intentionally inducing a leak (skip the `stream.close(...)` on one termination path) and confirming the test catches it before reverting.

## Summary

Phase 8.3 eliminates a deadlock in `internal/acp/client.go:Prompt()` (lines 696-803). Today `Prompt()` blocks on `respCh` waiting for the `session/prompt` response. Its synchronous caller (`engine.Run` at `internal/engine/engine.go:203`) cannot drain `Stream.Chunks` until `Prompt()` returns; the 64-slot buffer (`stream.go:69`) overflows on any non-trivial response; `push()` blocks the readLoop; the readLoop blocks the response frame; the request stays wedged until `STREAM_IDLE_TIMEOUT_SEC` fires a 500. The fix: return the `*Stream` as soon as `c.send(...)` accepts the request payload, and move the response-wait into a per-prompt goroutine that finalizes via `stream.close(...)`. `Stream.Result()` becomes the single sync point — its existing `<-s.done` mechanic (`stream.go:125-130`) is already shaped for this with zero API changes.

The refactor is *narrow and localized*. The blocking `select { case <-ctx.Done(): ... case frame := <-respCh: ... }` block at `client.go:756-803` lifts verbatim into a goroutine; the only structural changes are (a) registering that goroutine with `c.wg`, (b) moving `defer c.disp.cancel(id)` from `Prompt()` to the goroutine, and (c) emitting `engine.prompt.completed` immediately before `stream.close(...)`. All three downstream `Prompt()` callers (`internal/engine/acp_adapter.go:51`, `internal/pool/pool.go:558`, `internal/session/entry_acp.go:60`) forward the return value opaquely — verified by grep. All `Result()` consumers (anthropic SSE at `internal/adapter/anthropic/sse.go:770`, anthropic Collect at `collect.go:182`, openai SSE at `internal/adapter/openai/sse.go:530`, ollama NDJSON at `internal/adapter/ollama/ndjson.go:527`) already block on `<-s.done` — they require no changes.

**Primary recommendation:** Refactor in a single atomic commit covering `client.go` (Prompt rewrite + new `awaitPromptResult` helper) + `stream.go` (docstring update only — no API change) + `engine.go` (add `engine.prompt.completed` log via the new ACP-side emission path — see Pattern 3 below). Land the new overflow test in the same commit so the deadlock-and-fix arrive together. Trust gates from `make ci` (gofumpt → vet → build → golangci-lint → test-race) must pass on first run; `goleak.VerifyTestMain` already covers all ACP goroutines via `internal/acp/testmain_test.go:18`.

**Critical decision point — where to emit `engine.prompt.completed`.** The new log line must observe the `session_id`, the final `ChunkCount`, and the parsed `StopReason`. All three are inside the goroutine's scope at the moment of `stream.close(...)`. Emitting from `engine.Run`'s caller path is impossible — `engine.Run` has already returned. Therefore the line is emitted via `c.cfg.Logger.Debug(...)` inside `internal/acp/client.go`'s new goroutine. To match the `engine.prompt.sent` namespace cleanly, the message string remains `"engine.prompt.completed"` even though the emitter package is `acp` (the message string is the grep key for ops, not the emitter package — consistent with `engine.new_session.ok` emitted from `engine.go:192`, which also uses an `engine.*` namespace).

### Pattern 1: Lift-and-shift goroutine — minimum-diff refactor

**What:** Move the existing blocking `select` block (`client.go:756-803`) verbatim into a new function `awaitPromptResult`, registered with `c.wg`, called via `go c.awaitPromptResult(...)`.

**When to use:** This is the load-bearing pattern for the refactor. It minimizes review surface and risk by keeping the existing error/cancel/stop-reason handling byte-identical to the synchronous path.

**Example (target shape):**
```go
// Source: derived from internal/acp/client.go:726-803 (current synchronous Prompt)
func (c *Client) Prompt(ctx context.Context, sessionID string, blocks []canonical.Block) (*Stream, error) {
    id := c.nextID.Add(1)
    respCh := c.disp.register(id)
    // NOTE: defer c.disp.cancel(id) moved into awaitPromptResult goroutine.

    stream := newStream(ctx, sessionID)

    c.streamMu.Lock()
    c.activeStream = stream
    c.streamMu.Unlock()

    wire := translateBlocks(blocks)
    if err := c.send(ctx, id, rpcRequest{
        JSONRPC: "2.0",
        ID:      id,
        Method:  "session/prompt",
        Params:  promptParams{SessionID: sessionID, Prompt: wire, Content: wire},
    }); err != nil {
        c.disp.cancel(id)                  // explicit because no goroutine spawned
        c.streamMu.Lock()
        c.activeStream = nil
        c.streamMu.Unlock()
        return nil, err
    }

    c.wg.Add(1)
    go c.awaitPromptResult(ctx, id, sessionID, stream, respCh)

    return stream, nil
}

// awaitPromptResult waits for the session/prompt response (or ctx cancel / client close)
// and finalizes stream via stream.close(). Lifetime: registered with c.wg.
func (c *Client) awaitPromptResult(
    ctx context.Context,
    id uint64,
    sessionID string,
    stream *Stream,
    respCh <-chan rpcFrame,
) {
    defer c.wg.Done()
    defer c.disp.cancel(id)

    select {
    case <-ctx.Done():
        c.streamMu.Lock()
        c.activeStream = nil
        c.streamMu.Unlock()
        // Best-effort cancel notification (matches client.go:763-767).
        c.sendNotification(rpcNotification{
            JSONRPC: "2.0",
            Method:  "session/cancel",
            Params:  cancelParams{SessionID: sessionID},
        })
        stream.close(nil, fmt.Errorf("acp: prompt: %w", ctx.Err()))
        return

    case frame := <-respCh:
        c.streamMu.Lock()
        c.activeStream = nil
        c.streamMu.Unlock()
        if frame.Error != nil {
            if frame.Error.Code == closeSentinelCode {
                stream.close(nil, ErrClientClosed)
                return
            }
            stream.close(nil, fmt.Errorf("acp: prompt: rpc error %d: %s", frame.Error.Code, frame.Error.Message))
            return
        }
        var r promptResult
        if len(frame.Result) > 0 {
            if err := json.Unmarshal(frame.Result, &r); err != nil {
                c.cfg.Logger.Warn("acp: prompt result parse failed", "err", err)
            }
        }
        stop := parseStopReason(r.StopReason)
        // NEW: engine.prompt.completed log emission (CONTEXT.md decision).
        c.cfg.Logger.Debug("engine.prompt.completed",
            "session_id", sessionID,
            "chunks", stream.ChunkCount(),    // see Pattern 5 below
            "stop_reason", stop.String(),     // canonical.StopReason has String() per Phase 1.1
        )
        stream.close(&FinalResult{StopReason: stop}, nil)
    }
}
```

### Pattern 2: Goroutine registration mirrors `readLoop`/`writerLoop`/`pingLoop`

**What:** The per-prompt goroutine is registered with the same `c.wg` `WaitGroup` that gates `Client.Close()`'s wg.Wait at `client.go:970`.

**When to use:** All ACP-package goroutines whose lifetimes must be observed by `Close()`.

**Example:**
```go
// Source: internal/acp/client.go:328-331 (subprocess constructor)
c.wg.Add(3)
go c.readLoop(clientCtx)
go c.writerLoop(clientCtx)
go c.pingLoop(clientCtx)
```

The per-prompt goroutine adds **one** to `c.wg` *per active Prompt call*. `c.wg.Add(1)` happens in `Prompt()` AFTER `c.send` succeeds, BEFORE the `go` statement (Go memory-model requirement — never `Add` from inside the goroutine itself).

### Pattern 3: Defensive `session/cancel` notification on ctx-cancel

**What:** When the per-prompt goroutine observes `ctx.Done()`, it sends `sendNotification(session/cancel)` itself. This duplicates the work `engine.Run`'s `context.AfterFunc` watchdog (`engine.go:215-225`) is *also* doing — and that duplication is intentional.

**When to use:** Whenever a goroutine has its own cancellation responsibility AND there is a parallel watchdog. Required here because `ACPClient.Cancel(sid)` is idempotent (engine.go:152-158 documents this) and the timing of the two cancels is unordered.

**Example:**
```go
// Source: internal/acp/client.go:763-767 (current synchronous path — lift verbatim)
c.sendNotification(rpcNotification{
    JSONRPC: "2.0",
    Method:  "session/cancel",
    Params:  cancelParams{SessionID: sessionID},
})
```

### Pattern 4: Stream lifetime sync via `<-s.done`

**What:** `Stream.Result()` blocks on `<-s.done` (`stream.go:125-130`). This is **already** the sync point for every chunk-loop site in the codebase.

**Verified sites that require zero changes:**
- `internal/adapter/anthropic/sse.go:770` — `final, rerr := run.Stream().Result()`
- `internal/adapter/anthropic/collect.go:182` — `final, rerr := run.Stream().Result()`
- `internal/adapter/openai/sse.go:530` — `final, rerr := run.Stream().Result()`
- `internal/adapter/ollama/ndjson.go:527` — `final, rerr := run.Stream().Result()`

All four sites already call `Result()` after the chunk channel closes. Under the new contract, the channel still closes via `stream.close(...)` from the goroutine, and `Result()` still returns the same shape. **No adapter changes required.**

### Pattern 5: `Stream.ChunkCount` accessor (decide during planning)

`engine.prompt.completed` needs the final chunk count. Two options:

| Option | How | Tradeoff |
|--------|-----|----------|
| A | `stream.result.ChunkCount` read under `s.mu.Lock` after the channel is closed | Requires exporting an accessor (e.g., `(*Stream).ChunkCount() int`) or reading from a `FinalResult` returned by an internal helper. Stream's `mu` is already used at `stream.go:107`. |
| B | The goroutine maintains its own counter by ranging over `stream.Chunks` itself | **Rejected** — the adapter is the consumer; the goroutine reads `respCh` only. |
| C | Pull `ChunkCount` from `stream.result` directly inside the goroutine (same package, can read the unexported field under `s.mu`) | Cleanest. No new exported surface. The goroutine lives in package `acp` and `stream.result` is package-private. Pattern: take `s.mu.Lock()` around the field read. |

**Planner recommendation:** Option C. Inside the new goroutine (package `acp`), read `stream.result.ChunkCount` under `stream.mu.Lock()` immediately before calling `stream.close(...)`. No new exported API.

### Anti-Patterns to Avoid

- **Sending on a closed channel.** `stream.close` is guarded by `sync.Once` (`stream.go:106`) — calling it from both the goroutine AND from `readLoop`'s defer (`client.go:383-393`) is safe but produces a race on which one "wins". The `closeOnce.Do` ensures only one set of `(result, err)` is recorded. **Test must verify both orderings.**
- **Forgetting `defer c.disp.cancel(id)`.** A bare `c.disp.register(id)` without a matching cancel leaks the pending-map entry on every Prompt call. The defer pattern at `client.go:729` is the existing safety net — preserve it inside the goroutine.
- **Adding `c.wg.Add(1)` inside the goroutine.** Go memory model violation — `Add` must happen-before the goroutine starts running. Always `c.wg.Add(1)` immediately before the `go` statement in `Prompt()`.
- **Renaming `engine.prompt.sent`.** Breaks downstream ops greps and ChatTrace tooling. CONTEXT.md explicitly forbids this.
- **Removing the engine-side watchdog.** Decision-locked. The watchdog handles deliberate request cancellation from streaming adapters (e.g., when the HTTP request ctx is canceled) and provides defense-in-depth alongside the goroutine's own cancel handling.
- **Treating `defer c.disp.cancel(id)` as redundant after the goroutine reads `respCh`.** When `route()` delivers a frame to `respCh`, it has already done `delete(d.pending, num)` under `d.mu` (`dispatcher.go:128`). The `defer cancel(id)` is then a no-op in the happy path — but is required in the `<-ctx.Done()` and close-sentinel paths to drain the pending entry. The cost of the extra `delete` on a non-existent key is one map lookup. Keep the defer.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-goroutine lifetime tracking | A `chan struct{}` "done" pattern + manual select | Existing `c.wg sync.WaitGroup` | Already covers `readLoop`/`writerLoop`/`pingLoop`. Adding a new mechanism splits the goroutine-leak surface across two patterns. |
| Idempotent stream finalization | Manual "closed?" boolean + double-close guard | Existing `s.closeOnce sync.Once` at `stream.go:106` | Already correct. Multiple `stream.close(...)` calls collapse to one. |
| ctx-cancellation watchdog | `time.AfterFunc` + manual goroutine | `context.AfterFunc` (Go 1.21+) at `engine.go:215` | Already used and decision-locked. |
| Idempotent cancel | "did we already cancel?" boolean | `ACPClient.Cancel(sid)` (already idempotent per `engine.go:152-158`) | Documented contract; tested. |
| RPC-id dispatching | Manual map walks | Existing `dispatcher.go` | Owns `register` + `cancel` + `route` + `drainAll` + the `closeSentinelCode` sentinel. The refactor uses these unchanged. |

**Key insight:** The refactor's risk lies entirely in **timing**, not in new primitives. Every primitive needed already exists in `internal/acp` and is verified by existing tests. The job is to *recompose* them — not to introduce new ones.

## Common Pitfalls

### Pitfall 1: Stream-close race between goroutine and readLoop EOF defer

**What goes wrong:** If kiro-cli exits (subprocess crash, Close() called) WHILE the goroutine is selecting on `respCh`, two close paths race:

1. `readLoop`'s defer at `client.go:383-393` reads `c.activeStream` and calls `s.close(nil, ErrClientClosed)`.
2. The goroutine receives a close-sentinel frame from `failPending` via `respCh` and calls `stream.close(nil, ErrClientClosed)`.

**Why it happens:** `failPending(ErrClientClosed)` runs from both the `readLoop` EOF path (`client.go:398`) and `Close()` step 2 (`client.go:953`). Both deliver to every pending caller's `respCh`. The goroutine and the readLoop-defer can both observe the close.

**Why it's safe:** `stream.close` is guarded by `s.closeOnce` (`stream.go:106`). Only the first wins. Both paths supply the same `(nil, ErrClientClosed)` arguments so the final state is identical.

**How to avoid:** Don't fight the race — document it. **Add a test** that calls `Close()` while a Prompt is in-flight and asserts `goleak.VerifyNone` + `stream.Result()` returns `ErrClientClosed`.

**Warning signs:** Flaky test where `stream.Result()` returns sometimes `nil` and sometimes `ErrClientClosed` — would indicate the merge semantics in `s.close` (`stream.go:113-115`) silently replaced one with the other. Look for the `closeOnce.Do` callback being entered twice (impossible by `sync.Once` contract; if it appears, the bug is elsewhere).

### Pitfall 2: `c.activeStream` clear race between goroutine and Close()

**What goes wrong:** The goroutine writes `c.activeStream = nil` under `c.streamMu` at three exit points. The `readLoop` defer at `client.go:386-389` reads `c.activeStream` under the same mutex. If they run concurrently, one clears, the other reads nil — and that's correct behavior.

**Why it happens:** Race between subprocess exit (drives `readLoop` defer) and the goroutine's normal response path.

**Why it's safe:** Both sites take `c.streamMu`. The mutex serializes the read+write. The reader gets either the live stream pointer (calls `s.close(nil, ErrClientClosed)`) or nil (no-op). The writer always sets to nil. The stream itself is closed exactly once via `closeOnce`.

**How to avoid:** Preserve the existing `c.streamMu.Lock()/Unlock()` pattern verbatim around all `c.activeStream` writes in the goroutine. Three writes total: ctx-cancel path, rpc-error path, happy path.

**Warning signs:** `go test -race` flagging `c.activeStream` in any test that drives Close-during-Prompt.

### Pitfall 3: `c.wg.Wait()` deadlock on Close() when goroutine is stuck on respCh

**What goes wrong:** `Close()` step 4 calls `c.wg.Wait()` (`client.go:970`). If the goroutine is selecting on `respCh` and never gets a frame, `Wait` hangs forever.

**Why it happens:** Forgetting that `Close()` step 2 calls `failPending(ErrClientClosed)` (`client.go:953`), which `drainAll`'s the dispatcher. The goroutine WILL receive a sentinel frame on `respCh` and exit.

**Why it's safe:** Provided `failPending` is called BEFORE `wg.Wait`. Verify the order at `client.go:944-970`:
1. `c.cancel()` — cancels `clientCtx`.
2. `c.failPending(ErrClientClosed)` — drains pending, delivers sentinel to every respCh.
3. `c.stdin.Close()` / `c.rwc.Close()` — unblocks `readLoop`.
4. `c.wg.Wait()` — now safe because all goroutines have received their exit signal.

Step 2 happens BEFORE step 4. The goroutine sees the sentinel and exits cleanly.

**How to avoid:** Do not reorder `Close()`'s shutdown sequence. The order is decision-locked in Phase 1 and preserved here.

**Warning signs:** `goleak.VerifyTestMain` failing with the new goroutine still alive — would indicate the goroutine missed an exit signal. Verify it selects on BOTH `ctx.Done()` AND `respCh`. (Note: `clientCtx` cancellation does NOT directly fire the goroutine's `ctx.Done()` arm — `ctx` is the *caller's* ctx, not the client's. The client's exit signal arrives via `respCh` from `failPending`.)

### Pitfall 4: Double-cancel of dispatcher entry

**What goes wrong:** `c.disp.cancel(id)` is called twice — once by the goroutine's `defer`, and a second time if the planner adds a "defensive cleanup" elsewhere.

**Why it happens:** Pattern temptation. A reviewer sees `defer c.disp.cancel(id)` inside a goroutine and adds another `c.disp.cancel(id)` "just in case" inside `Prompt()` after `c.send` fails.

**Why it's safe (operationally):** `dispatcher.cancel(id)` at `dispatcher.go:80-84` does `delete(d.pending, id)` under `d.mu`. Calling it twice on the same id is two map-deletes on a key that was already gone — totally benign.

**How to avoid:** The send-failure path in `Prompt()` (current code at `client.go:749-754`) **must** call `c.disp.cancel(id)` explicitly because the goroutine is NOT spawned in that path. The happy path (goroutine spawned) defers `c.disp.cancel(id)` inside the goroutine. Two paths, exactly one cancel each.

**Warning signs:** None observable — but verify with code review that the send-failure path explicitly cancels and the happy path defers.

### Pitfall 5: `engine.prompt.sent` semantics drift undocumented

**What goes wrong:** Ops engineers who relied on `engine.prompt.sent` to mean "response landed" now see it firing immediately after send-accept. Dashboards measuring `new_session.ok → prompt.sent` interval suddenly read 1ms instead of N seconds.

**Why it happens:** Log semantics shifted but the string is preserved.

**How to avoid:** Phase commit message MUST call out the semantic shift and direct ops to migrate to `engine.prompt.completed` for the response-landed metric. Add a one-line entry in `docs/operating.md` Logging section. Include both lines' field shapes there.

**Warning signs:** Ops Slack post-deploy: "why is prompt.sent always 1ms now?" Document proactively.

### Pitfall 6: `ChunkCount` read race with `push()`

**What goes wrong:** The goroutine reads `stream.result.ChunkCount` under `s.mu` immediately before calling `stream.close(...)`. If the readLoop is still calling `push()` (which takes `s.mu` to increment ChunkCount at `stream.go:85-87`), the goroutine reads a stale count.

**Why it happens:** The response frame can arrive at the dispatcher before the readLoop has finished `push()`-ing all preceding `session/update` frames. The readLoop processes frames sequentially per `readLoop` at `client.go:394`, so this is actually impossible — the response frame is a frame, and it's processed after all preceding frames in stdout order. The readLoop's `push()` calls complete before `route()` delivers the response frame.

**Why it's safe (operationally):** Sequential frame processing by the single readLoop goroutine guarantees all `push()` calls for a turn complete before the response frame is dispatched.

**How to avoid:** Still take `s.mu` when reading `ChunkCount` — defense in depth and matches the existing pattern.

**Warning signs:** Test asserting `ChunkCount == 128` reports 127. If this ever happens, the readLoop ordering invariant is violated and the bug is upstream.
