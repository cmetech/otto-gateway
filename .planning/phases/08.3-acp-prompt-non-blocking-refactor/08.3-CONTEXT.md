# Phase 08.3: ACP Prompt() Non-Blocking Refactor — Context

**Gathered:** 2026-06-03
**Status:** Ready for planning
**Source:** Live diagnosis from Windows v1.9.2 PII smoke-test hang; design decisions captured via `/gsd-discuss-phase`.

<domain>
## Phase Boundary

Phase 8.3 fixes a chunk-buffer-overflow deadlock in `internal/acp/client.go:Prompt()` that wedges the gateway whenever kiro-cli emits more than 64 `session/update` chunks before its `session/prompt` response. Today's `Prompt()` blocks until the final response arrives, but its primary caller (`internal/engine/engine.go:Run` line 203) calls it synchronously and **cannot** start draining `Stream.Chunks` until `Run` returns — and `Run` can't return until `Prompt` returns. The 64-slot `Stream.Chunks` channel (`internal/acp/stream.go:69`) overflows on any non-trivial response, `push()` blocks, the readLoop blocks, the ping reply can't be parsed, the `session/prompt` response can't be parsed, and the request stays wedged until `STREAM_IDLE_TIMEOUT_SEC` (100s in the reproduced incident) fires a 500.

The existing `internal/acp/client.go:701-725` docstring acknowledges this verbatim: *"Calling Prompt synchronously and draining Chunks afterward only works when the total chunk count fits in the 64-slot buffer."* Phase 8.3 removes the footgun.

### Reproduced symptom (Windows v1.9.2, native kiro-cli.exe, POOL_SIZE=4)

```
21:26.43  request POST /v1/messages arrives (PII redact mode = encrypt)
21:26.44  engine.new_session.ok           <- session/new returned
          (silence 34s — Prompt() blocked, readLoop blocked, ping reply blocked)
22:00.10  WARN acp: ping failed
22:23.06  request POST /v1/messages 500 100000ms  <- STREAM_IDLE_TIMEOUT
```

PID-1 of the 4 kiro-cli workers gained ~22 MB and burned CPU during the hang — kiro-cli was actively streaming into a pipe that was no longer being read.

### Fix shape

- `Prompt(ctx, sid, blocks)` returns the `*Stream` as soon as `c.send(...)` accepts the JSON-RPC request payload. It does **not** wait for the `session/prompt` response.
- A per-prompt goroutine owns the response-wait. It selects on `respCh`, `ctx.Done()`, and the client lifetime context; parses `result.stopReason` via `parseStopReason`; clears `c.activeStream` under `c.streamMu`; and finalizes via `stream.close(&FinalResult{StopReason: stop}, nil)`. RPC errors propagate via `stream.close(nil, err)`. The `closeSentinelCode` path surfaces `ErrClientClosed`.
- `Stream.Result()` becomes the single sync point for callers that need the final `StopReason` / terminal error. The existing `<-s.done` mechanic in `internal/acp/stream.go:125-130` is already shaped exactly for this — no Stream-API changes.
- Callers consume `Stream.Chunks` directly (they already do); the new contract is that `Prompt()` never deadlocks regardless of chunk count.

</domain>

<decisions>
## Implementation Decisions

### Cancellation ownership — two-owner, double-cancel-safe

**Decision.** `engine.Run` keeps its `context.AfterFunc` watchdog (`internal/engine/engine.go:215-225`) that calls `e.cfg.ACP.Cancel(sid)` on `ctx.Done()`. The new per-prompt goroutine inside `acp.Client.Prompt()` is also cancel-aware — it observes `ctx.Done()`, sends its own best-effort `session/cancel` notification (matching the current synchronous path at `client.go:763-767`), clears `c.activeStream`, and finalizes the stream via `stream.close(nil, ctx.Err())`.

**Why two owners is fine.** `ACPClient.Cancel(sid)` is already idempotent — the engine's pre-existing comment at `engine.go:152-158` documents this ("Cancel is idempotent"). A duplicate `session/cancel` notification is a best-effort fire-and-forget that costs nothing on kiro-cli's side; the spec treats repeats as no-ops. The defensive layer also protects against the race where ctx fires after `Prompt()` returns but before the adapter reaches its watchdog-stop call.

**Why not single-owner.** Removing the engine-side watchdog enlarges the engine.go diff (the watchdog also handles deliberate request cancellation from streaming adapters), and a watchdog-only design would couple the cancel timing to engine.Run's return path — defeating the whole point of returning early.

### `engine.prompt.sent` log signal — two log lines

**Decision.** Keep `engine.prompt.sent` as the early-dispatch signal — it fires immediately after `c.send(...)` accepts the request payload (i.e., what `Prompt()` now blocks on, which is millisecond-scoped). Its meaning shifts from "response landed" to "send accepted".

**Add `engine.prompt.completed`** — a new DEBUG line emitted by the per-prompt goroutine immediately before `stream.close(...)` succeeds. Carries `session_id`, `chunks`, and `stop_reason`. Ops dashboards that previously timed `new_session.ok → prompt.sent` to measure end-to-end response latency now time `new_session.ok → prompt.completed` for the same number; the new `prompt.sent → prompt.completed` interval separates queue-to-write latency from response latency, which is exactly the signal that would have made today's deadlock obvious-on-sight in the original incident log.

**Why not rename.** Renaming `engine.prompt.sent` breaks external log greps and downstream ChatTrace tooling — not worth it for cosmetic clarity when an additional line gives strictly more information.

**Why not keep semantics by moving the emission into the goroutine.** Loses the new early signal entirely. The whole reason we noticed the deadlock was that `engine.prompt.sent` was missing from the log — a moved-into-goroutine emission would have stayed missing in exactly the same way, providing zero diagnostic improvement over the status quo.

### Test strategy — add overflow test, preserve mixed-variants

**Decision.** Add a new dedicated test (likely whitebox in `internal/acp/`, possibly mirrored as an integration test using `fakeacp`): `TestIntegration_Prompt_OverflowsBuffer_DoesNotDeadlock`. The fake emits ≥128 `session/update` chunks **before** its `session/prompt` response. Assertions:
1. `client.Prompt(ctx, sid, blocks)` returns within a tight deadline (e.g. 100 ms).
2. Ranging over `stream.Chunks` drains all 128 chunks without losing any.
3. `stream.Result()` returns the expected `StopReason` (`canonical.StopEndTurn`).
4. `goleak.VerifyNone` passes — the per-prompt goroutine exits cleanly.

**Keep existing `TestIntegration_FakeACP_E2E_MixedVariants` unchanged.** It already runs the "Prompt-on-one-goroutine, drain-on-another" pattern; under the new contract this pattern still works (it's just no longer required for correctness). The test thus doubles as a regression artifact proving backward compatibility for any external caller that still follows the historical recipe.

**Goleak coverage.** The new per-prompt goroutine must be added to whatever `goleak.VerifyNone` invocations already exist in the ACP package's test setup. Verify by intentionally inducing a leak (e.g., skip the `stream.close(...)` on one termination path) and confirming the test catches it before reverting.

### Claude's Discretion (defaults locked, no further input needed)

- **Goroutine registration with `c.wg`.** The per-prompt goroutine is registered with the client's existing `c.wg` `WaitGroup` so `Client.Close()` waits for in-flight finalization, matching the existing pattern used by `readLoop`, the writer goroutine, and `pingLoop` (`client.go:331`/`362`). This keeps lifetime semantics uniform across the client's goroutines.
- **Dispatcher entry ownership.** `c.disp.register(id)` stays in `Prompt()` (before `c.send` — must precede send for race-free response routing). The matching `c.disp.cancel(id)` moves into the per-prompt goroutine's `defer` so the entry outlives `Prompt()` return. On `Client.Close()`, `failPending` (`client.go:506`) drains the dispatcher's pending map exactly as today; the goroutine receives the close-sentinel error via `respCh` and finalizes the stream cleanly.
- **Other `Prompt()` callers.** `internal/engine/acp_adapter.go:51`, `internal/pool/pool.go:558`, and `internal/session/entry_acp.go:60` all return / forward the `*Stream` opaquely. The signature `(*Stream, error)` does not change; only the timing of return changes. No call-site updates are required at those three sites; verify during planning that no caller assumes "Prompt returned ⇒ response complete" (none should, given the docstring's explicit warning, but worth a grep).
- **Watchdog stop coordination.** Streaming adapters that hold the `Run` handle call `runHandle.StopWatchdog()` after natural completion (`internal/adapter/anthropic/handlers.go:194-197` is the canonical example). No change needed — the watchdog still runs against ctx cancellation, just now in parallel with the goroutine's own cancel handling rather than in series with `Prompt()`'s synchronous wait.
- **Stream docstring updates.** The `// Required usage: callers MUST drain Chunks concurrently with whatever goroutine is waiting on Prompt() to return.` warning in `internal/acp/stream.go:41-45` is removed (no longer required for correctness, only for throughput on truly massive responses). The Prompt godoc's concurrency contract section is rewritten to describe the new behavior. The "Phase 1.1 WR-01" reference is updated to cite Phase 8.3.

</decisions>

<specifics>
## Specific Ideas

- **Goroutine name for logs/diagnostics.** When the per-prompt goroutine logs or appears in stack traces, it should be identifiable. Suggest a wrapper function `awaitPromptResult(ctx, id, sessionID, stream, respCh)` (or similar) rather than an anonymous `go func() { ... }()` — easier to spot in goroutine dumps and in goleak failure output.
- **Log fields.** `engine.prompt.completed` should carry the same `session_id` field as `engine.prompt.sent` plus `chunks` (final `ChunkCount`) and `stop_reason` (parsed `canonical.StopReason` string). This matches the field shape of `engine.new_session.ok` for grep-consistency.
- **Test fake budget.** The new overflow test should emit a chunk count comfortably above 64 — suggest 128 as the round number. Chunks should be cheap to construct (single-character `ChunkKindText` frames are fine); the test is about the deadlock, not about realistic content shape.
- **Reproduction verification.** Plan should include a manual verification step that the Windows `test-pii.ps1 pii` scenario succeeds end-to-end on a fresh install once the fix is merged. The `EVAL` cipher round-trip is the original regression — confirm decrypt fires and plaintext lands in the response.

</specifics>

<deferred>
## Deferred Ideas

- **OpenAI stream=false hang BEFORE `engine.new_session.ok`.** The v1.9.2 wire-test run showed a separate hang on the 4th request (POST /v1/chat/completions with stream=false) where the gateway never logged `engine.new_session.ok` — i.e., the worker didn't respond to `session/new` itself. This is upstream of Prompt() and explicitly out of scope for Phase 8.3. File as a follow-up worker-state diagnostic phase (probably worker leak / stale session in kiro-cli from a prior request) once 8.3 ships.
- **Buffer-size tuning.** With the deadlock removed, the 64-slot channel becomes a pure throughput knob. May or may not warrant tuning — separate optimization phase, not this one.
- **Backpressure observability.** A `pool.acquire.wait_ms` style metric on `stream.push()` block duration would let ops detect slow consumers without inferring it from latency. Nice-to-have, not blocking.

</deferred>
