## Summary

The Phase 8.3 refactor is **correct on the load-bearing concurrency contract**: `Prompt()` returns `(*Stream, nil)` as soon as `c.send` accepts the payload, the per-prompt `awaitPromptResult` goroutine is registered with `c.wg` BEFORE `go` (memory-model-safe), the defer order in the goroutine is `c.wg.Done()` registered first → runs last (LIFO contract documented inline at client.go:770-773), and the three termination arms (ctx.Done, RPC error / close-sentinel, happy-path response) each finalize the stream via exactly one `stream.close(...)` call. The `s.closeOnce` guard collapses the readLoop-defer ↔ goroutine race on close, and `c.streamMu` is taken-and-released without nested calls on either path.

The four cross-file invariants enforced by the planner all hold:
- **D-02 enforcement:** `git diff` confirms `internal/engine/engine.go` is untouched; `engine.prompt.sent` at engine.go:208 unchanged; new `engine.prompt.completed` emitted from `internal/acp/client.go:854-858` with `session_id` / `chunks` / `stop_reason` field names matching the test assertion at client_test.go:1234-1241.
- **D-01 two-owner cancel:** the goroutine's ctx-cancel arm emits its own `session/cancel` notification matching `c.Cancel(sid)`'s shape; the engine watchdog at engine.go:215-225 fires the same notification idempotently.
- **D-03 dispatcher.cancel defer move:** `defer c.disp.cancel(id)` lives INSIDE `awaitPromptResult` (line 798), not in `Prompt()`; the send-failure path at client.go:748 has the EXPLICIT non-deferred cancel as required.
- **D-04 c.wg pattern:** `c.wg.Add(1)` at line 758 is on its own line before `go` at line 759, with the `defer c.wg.Done()` as the first defer in the goroutine.

The whitebox tests cover all three documented termination arms (close-during-prompt → ErrClientClosed; ctx-cancel + wire-frame assertion for session/cancel; happy-path log shape). The integration test fires the 100ms non-blocking gate before emitting 128 chunks and asserts no chunk loss. The `drivePromptStream` helper in `internal/engine/acp_adapter_test.go` correctly identifies and fixes the Rule-1 induced-bug from the non-blocking refactor (`ctx.Background()` for `Prompt`, bounded ctx for `Initialize`/`NewSession`).

The findings below are operational/diagnostic asymmetries and one missing operator-facing documentation surface — none are correctness bugs.

## Warnings

### WR-01: `engine.prompt.completed` is not emitted on error or ctx-cancel termination arms — log asymmetry breaks "sent → completed" pairing

**File:** `internal/acp/client.go:800-860`
**Issue:** The Debug log line `engine.prompt.completed` is emitted ONLY on the happy path (line 854, inside the `frame.Error == nil` branch). The ctx-cancel arm (line 801-816) and the RPC-error / close-sentinel arms (line 822-828) both call `stream.close(...)` and return without emitting any companion log line.

This breaks the operator-facing pairing the SUMMARY claims is load-bearing (CONTEXT.md D-02: "engine.prompt.completed carries the response-landed signal that ops dashboards previously timed via engine.prompt.sent"). A prompt that is cancelled by the client mid-flight, or that fails because kiro-cli closed the pipe, produces an `engine.prompt.sent` line with no matching terminal line — so any dashboard counting completions or measuring latency will silently undercount or hang in "in-flight" state. The Windows v1.9.2 PII regression (which this phase exists to close) is exactly the failure mode where Cancel-on-deadline would be common, and where operators most need the diagnostic.

**Fix:** Emit a terminal log line on every exit arm. Either reuse `engine.prompt.completed` with a `stop_reason` reflecting termination cause, or add a parallel `engine.prompt.terminated` line. Concrete patch:

```go
// ctx-cancel arm — replace the bare stream.close call with:
c.cfg.Logger.Debug("engine.prompt.completed",
    "session_id", sessionID,
    "chunks", chunkCountUnderMu(stream),
    "stop_reason", "ctx_cancelled",
)
stream.close(nil, fmt.Errorf("acp: prompt: %w", ctx.Err()))

// frame.Error != nil arm — same shape with stop_reason "rpc_error" or "client_closed".
```

Reading `chunkCount` requires the same `stream.mu.Lock()` dance as the happy path; factor into a small helper to avoid repeating four lines.

### WR-02: `engine.prompt.sent` semantic shift is not surfaced in `docs/operating.md` or any CHANGELOG/release-notes artifact

**File:** Repository-wide — no doc artifact in the Phase 8.3 commit chain references the semantic change.
**Issue:** Phase 8.3 silently changes the meaning of `engine.prompt.sent` (engine.go:208) from "kiro-cli accepted the request AND returned a session/prompt response" (pre-refactor, because Prompt was synchronous) to "kiro-cli accepted the request payload on the writer channel" (post-refactor). RESEARCH.md Pitfall 5 and CONTEXT.md D-02 both flagged this as load-bearing for ops, and the operator-verification Task H instructs the operator to specifically watch for `engine.prompt.sent` firing "within milliseconds of new_session.ok (NOT after a 30+ second pause)" — confirming the team knows the semantic moved.

`docs/operating.md` exists in the repo and is the canonical operator surface; `grep -n "engine.prompt" docs/operating.md` returns empty. No CHANGELOG entry, release-note, or DEVELOPERS.md note is in the Phase 8.3 file list. Any operator running a dashboard that timed prompt latency via the gap between `engine.prompt.sent` and a downstream "response landed" signal will see latencies collapse to milliseconds post-deploy and may mis-diagnose the change as a kiro-cli behavioural shift.

**Fix:** Add a short operator-facing note. Minimum viable surface:

```markdown
## v1.9.3 (Phase 8.3): ACP Prompt() non-blocking refactor — log-line semantics

`engine.prompt.sent`
  Now fires when the gateway writer goroutine accepts the
  `session/prompt` payload onto the channel — typically within
  milliseconds of `engine.new_session.ok`. Previously this line
  fired only AFTER kiro-cli's `session/prompt` response landed,
  which gated on the agent's full response (often 5–30s for an
  LLM turn). Dashboards that measured "prompt latency" via the
  gap between this line and a downstream marker should switch to
  the new `engine.prompt.completed` line.

`engine.prompt.completed` (NEW)
  Fires when the per-prompt goroutine observes the
  `session/prompt` response (or the close-sentinel / ctx-cancel
  terminal arm — see WR-01). Carries `session_id`, `chunks`,
  `stop_reason` (raw kiro-cli wire string).
```

Drop into `docs/operating.md` under whatever log-line section already exists; if none exists, this is its own subsection.

### WR-03: Sleep-based synchronization in `TestClient_PromptCtxCancel_FinalizesStreamWithCtxErr` and `TestClient_CloseDuringInFlightPrompt_FinalizesStreamCleanly` is flake-prone under CI load

**File:** `internal/acp/client_test.go:981`, `internal/acp/client_test.go:1064`
**Issue:** Both new whitebox tests use `time.Sleep(50 * time.Millisecond)` to give `awaitPromptResult` time to register in the dispatcher pending map before the test invokes `Close()` or `cancel()`. On a slow CI runner under `-race` (which slows scheduling by 5–10x) this 50ms can be insufficient — the test then cancels BEFORE the goroutine has reached the select, and the close-sentinel / ctx.Done arms may interact with an unregistered respCh in ways the test was not designed to exercise.

The existing pattern `TestPendingRequestsFailedOnClose` at client_test.go:209 uses the same 50ms sleep, but it predates Phase 8.3 and isn't the load-bearing close-race test the planner asked Phase 8.3 to provide. Replicating a known-fragile pattern propagates the fragility.

**Fix:** Replace the sleep with a deterministic synchronization on the dispatcher's pending map. A whitebox test can take `c.disp.mu` and poll for `len(c.disp.pending) > 0`:

```go
// Wait deterministically for awaitPromptResult to register, instead of
// time.Sleep(50ms). Bounded poll with a 2s ceiling.
deadline := time.Now().Add(2 * time.Second)
for time.Now().Before(deadline) {
    c.disp.mu.Lock()
    n := len(c.disp.pending)
    c.disp.mu.Unlock()
    if n > 0 {
        break
    }
    time.Sleep(time.Millisecond)
}
```

This is whitebox so the access is legitimate; it removes the only timing-based race in the new tests and converts the 50ms blind wait into a sub-millisecond observed signal.

### WR-04: Send-failure path at `client.go:748` re-acquires `c.streamMu` to clear `activeStream` immediately after releasing it at line 732 — narrow but real race window with `handleNotification`

**File:** `internal/acp/client.go:730-752`
**Issue:** The flow is:
1. Take streamMu, write `c.activeStream = stream`, release (lines 730-732).
2. Call `c.send(...)`.
3. On send error: cancel dispatcher entry, take streamMu again, set `c.activeStream = nil`, release (lines 748-751).

Between (1) and (3), `handleNotification` can grab `c.activeStream` (line 960-962), drop the lock, and then call `s.push(c.clientCtx, chunk)`. Because the caller will receive `(nil, err)` from Prompt and never drain the channel, `push` will succeed up to 64 chunks (buffer size) and then block forever on the 65th — the readLoop goroutine wedges until clientCtx fires.

In practice this window is very narrow: `c.send` only errors on `ctx.Done()` or `c.clientCtx.Done()`, both of which usually fire after no kiro-cli traffic was ever requested. The realistic trigger is "caller cancelled ctx before send accepted" — at which point no upstream notification chain has started — so the race window is effectively empty. But the code path EXISTS and a future agent that pre-emits chunks (or a fakeacp server that races a `session/update` ahead of the `session/prompt` ACK) would expose it.

This pre-dates Phase 8.3 — the same pattern is in the pre-refactor synchronous body — but Phase 8.3 had an opportunity to fix it (the cleanup is now adjacent to the send-error return) and didn't.

**Fix:** Drop the "set activeStream BEFORE send" invariant. Instead, set `activeStream` only AFTER `c.send` returns nil. The race-window cost is "a session/update that arrives between send-accept and activeStream-set lands in the unknown-session drop" — but the integration_test's `fake.promptSeen` synchronization already proves that scheduling is safe in practice. Concrete sketch:

```go
wire := translateBlocks(blocks)
if err := c.send(ctx, id, rpcRequest{...}); err != nil {
    c.disp.cancel(id)
    return nil, err
}
c.streamMu.Lock()
c.activeStream = stream
c.streamMu.Unlock()
c.wg.Add(1)
go c.awaitPromptResult(ctx, id, sessionID, stream, respCh)
return stream, nil
```

Alternatively: keep the current ordering but make `push()` non-blocking on send-failure (drop chunks rather than block on the buffer). The former is preferable — it removes the race-window entirely.

### IN-01: `chunkCount` value snapshot at `client.go:846` is documented as "defense-in-depth" but the docstring overstates the guarantee

**File:** `internal/acp/client.go:840-847`
**Issue:** The comment claims `the readLoop's sequential frame processing already guarantees all push() calls for this turn have completed before this response frame was dispatched`. This is true for kiro-cli's protocol (the agent emits every `session/update` for a turn before the `session/prompt` response), but the comment elides that a slow Chunks consumer can still leave `push()` blocked on the 64-slot buffer at the moment the response frame is dispatched. In that case `ChunkCount` reflects only the pushed-and-accepted count, not the count of frames the agent emitted. The Stream consumer is responsible for draining; pending pushes resolve before `stream.close` runs (the buffer is drained by the consumer after close).

The Lock/Unlock is harmlessly correct; the docstring is just slightly misleading about what's being guaranteed. Suggest tightening:

**Fix:** Adjust the comment to reflect what's actually true:

```go
// Phase 8.3 Pitfall 6 defense-in-depth: read ChunkCount under
// stream.mu before close — the value is the count of push() calls
// the readLoop has successfully posted to the buffered channel.
// If a slow consumer left push() blocked at the moment of the
// response frame, chunks still in flight do not appear here;
// they'll land in Stream.Chunks before the buffer drains and the
// channel closes. For the diagnostic log line below, this snapshot
// is "chunks delivered into the channel so far", not "chunks the
// agent emitted".
```

### IN-03: Stream godoc at `stream.go:32-48` mixes "MAY be concurrent for throughput" with a buffered-channel implementation detail that callers shouldn't depend on

**File:** `internal/acp/stream.go:43-48`
**Issue:** The new Phase 8.3 docstring is correct but bleeds the "64 slots" buffer constant into the public contract. Callers reading this docstring may build assumptions on "I can let 64 chunks queue before I must drain" — but the buffer size is an implementation detail at `newStream` (line 68) and may change. The buffer's purpose (absorb bursts so readLoop doesn't block on momentary consumer hiccups) is the contract; the literal "64" is not.

**Fix:** Reword to describe the property, not the constant:

```go
// The internal channel is modestly buffered to absorb short bursts;
// if a consumer is slower than the producer the readLoop blocks on
// push() until the consumer catches up, but the session/prompt
// response continues to flow through awaitPromptResult and
// Stream.Result() will still return cleanly once the buffer drains.
```
