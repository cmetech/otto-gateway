## How It Connects

- **Adapter-level layer:** each new `*_shortcircuit_test.go` file declares a tiny `shortCircuitFakeEngine` (5 lines) that returns a pre-built `*fakeRunHandle` with `scResp = newAuthErrorResp()`. The handler under test calls `eng.Run(streamCtx, req)`, gets the fake handle, then hits the Plan 01 short-circuit guard which inspects `runHandle.ShortCircuitResponse()`. Because `scResp != nil`, the guard fires and `writeError(w, 401, ...)` runs before the emitter is reached. The negative byte-marker assertions then prove the emitter never ran.
- **E2E layer:** the three new rows go against the real otto-gateway binary booted by `bootGateway`. The bearer is `Bearer wrongtoken`, AUTH_TOKEN is `validtoken`, so the canonical AuthHook short-circuits and produces a `canonical.ChatResponse` with `StopReason==StopError`. The streaming branch in each handler catches that via `runHandle.ShortCircuitResponse()` and renders the per-surface native envelope at 401 before SSE/NDJSON headers are written. The real-binary path also passes through middleware (RequestIDHook, LoggingHook, etc.) so the test is a true end-to-end pin.
- **Goleak gating:** each adapter package already has `goleak.VerifyTestMain` in `testmain_test.go`. The new whitebox files inherit the gate automatically â€” no per-file setup needed.

## Decisions Made

- Two atomic commits (one per task) per plan Â§<verification> â€” Slice A's atomic-commit rule applied only to Plan 01's interface/handler change. Plan 02 is test-only and benefits from per-task atomicity (Task 1 = adapter tests; Task 2 = E2E rows).
- Used Pattern F's recommended `(t, body, ct)` harness signature widening for the E2E table rather than the wider `(t, resp, body)` alternative. The streaming rows need Content-Type and nothing else from the response.
- Each new test file declares its own minimal `shortCircuitFakeEngine` rather than complicating each package's existing `fakeEngine`. The existing fakeEngines wire `runChunks`/`runFinal` and never accept an injected RunHandle; adding that field per-surface would have touched three more files for no gain â€” the new test file already owns the surface-specific shape it needs.
- Asserted OpenAI Content-Type via `strings.HasPrefix(ct, "application/json")` (not exact equality) because httptest's `Server` proxies through `http.Server` which may append `; charset=utf-8` on some Go versions. Ollama and Anthropic use exact equality because their `writeError` helpers set the exact string and the assertion fires at the `httptest.ResponseRecorder` boundary (no `http.Server` proxy).

## Deviations from Plan

**None for the core flow â€” plan executed as written.**

Two minor adjustments documented for transparency:

- **shortCircuitFakeEngine type (one per new test file).** The plan's read_first guidance ("reuse existing helpers from handlers_test.go or ndjson_test.go") suggested I could reuse each package's existing `fakeEngine`. But the existing fakeEngines in ollama (handlers_test.go:49-55) and openai (integration_test.go:144-160) construct the RunHandle internally from `runChunks`/`runFinal` and never accept an injected handle. Rather than mutate three existing fakeEngines (which would have touched files outside the plan's `files_modified` set in a meaningful way and required care to keep existing tests green), I introduced a tiny 5-line `shortCircuitFakeEngine` per new test file. This kept the diff to existing files minimal (just the fakeRunHandle struct + the ShortCircuitResponse stub body) and made the new test file fully self-contained.
- **OpenAI Content-Type assertion uses `strings.HasPrefix` instead of exact equality.** httptest may surface `application/json; charset=utf-8` through the round-trip; matching the prefix is the robust check. Ollama and Anthropic adapter tests use exact equality because they POST directly to `a.ProtectedRouter()` and the `writeError` helper sets the header to the exact string.
