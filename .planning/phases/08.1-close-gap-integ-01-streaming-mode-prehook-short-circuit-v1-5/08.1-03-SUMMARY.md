# Phase 08.1 Plan 03: Brief §3.12 trust-gate sequence — explicit in Makefile + CI Summary

WARNING-01 close-out via documentation-of-existing-discipline: brief §3.12's `gofumpt -d → go vet → go build → golangci-lint → govulncheck → go test -race → go test -run Example → cross-compile` sequence is now visible in both `Makefile` (new explicit targets + rewired `ci:` chain) and `.github/workflows/ci.yml` (new explicit steps interleaved with existing make-delegated steps), per Phase 08.1 D-16.

## What Changed

### Makefile

Three new targets, one updated `ci:` chain, one updated `.PHONY` line.

**New: `fmt-check`** (brief §3.12 step 1, read-only formatting verification):

```make
fmt-check: ## Verify gofumpt formatting (brief §3.12 step 1 — fails on diff)
	@if command -v gofumpt >/dev/null 2>&1; then \
		diff="$$(gofumpt -d . 2>&1)"; \
		tool="gofumpt"; \
	else \
		diff="$$(gofmt -d . 2>&1)"; \
		tool="gofmt"; \
	fi; \
	if [ -n "$$diff" ]; then \
		echo "FAIL: $$tool reports formatting diffs (brief §3.12):"; \
		echo "$$diff"; \
		exit 1; \
	fi
```

The existing `fmt:` target (write-mode, `gofumpt -w .`) is unchanged — `fmt-check` is the CI-safe sibling. Falls back to `gofmt -d` if gofumpt is unavailable, per RESEARCH §"Environment Availability".

**New: `vet`** (brief §3.12 step 2):

```make
vet: ## Run go vet (brief §3.12 step 2 — explicit even though govet linter covers it)
	go vet ./...
```

Duplicates what golangci-lint's `govet` linter already runs; D-16 calls for the explicit step so future contributors don't ask "where's vet?"

**New: `examples`** (brief §3.12 step 7):

```make
examples: ## Run go Example tests (brief §3.12 step 7)
	go test -run Example ./...
```

Gates Go's `Example_*` test convention separately from the regular suite.

**Existing `build:`** target was already present (line 24) and is reused as the brief §3.12 step 3 dependency.

**Rewritten `ci:` chain** — was `lint test-race arch-lint`; now matches the brief §3.12 canonical order:

```make
ci: fmt-check vet build lint test-race arch-lint examples ## Full CI gate (brief §3.12 canonical sequence)
	$(shell go env GOPATH)/bin/govulncheck ./...
```

The govulncheck call stays in the recipe body (existing pattern; no separate target).

**`.PHONY` line** updated to declare `fmt-check vet examples`.

`make -n ci` dry-run confirms the canonical order:

```
gofumpt -d  (via fmt-check recipe)
go vet ./...
go build -ldflags=... -o bin/otto-gateway ./cmd/otto-gateway
golangci-lint run ./...
go test -race ./...
go-arch-lint check --project-path .
go test -run Example ./...
govulncheck ./...
```

### .github/workflows/ci.yml

Four new steps inserted into the existing `lint-test-arch` job. The `cross-compile-smoke` job is unchanged.

**Inserted BEFORE the existing `golangci-lint` step:**

```yaml
- name: Install gofumpt
  run: go install mvdan.cc/gofumpt@latest

- name: Format check (gofumpt -d)
  run: |
    set -euo pipefail
    diff="$(gofumpt -d .)"
    if [ -n "$diff" ]; then
      echo "FAIL: gofumpt reports formatting diffs (brief §3.12 step 1):"
      echo "$diff"
      exit 1
    fi

- name: Vet
  run: go vet ./...

- name: Build (host)
  run: go build ./...
```

**Inserted AFTER the existing `make test-race` step:**

```yaml
- name: Example tests
  run: go test -run Example ./...
```

Each new step inherits the job's existing checkout + Go-setup. Step names use sentence case to match existing style. The leading comment block above the new steps documents the delegation pattern:

> Phase 08.1 WARNING-01: explicit trust-gate sequence per docs/briefs/go_port_brief.md §3.12. The local Makefile holds the canonical sequence (`make ci`); CI duplicates the leading steps (gofumpt -d → go vet → go build) for in-log readability before delegating golangci-lint + arch-lint + test-race to the action / make targets below. Keeping both sides explicit prevents drift — see Phase 08.1 D-16.

**Final step order in `lint-test-arch`** (from `awk` extract on the committed file):

1. Checkout
2. Set up Go (from go.mod)
3. Install gofumpt
4. Format check (gofumpt -d)
5. Vet
6. Build (host)
7. golangci-lint
8. Install go-arch-lint
9. Install govulncheck
10. Test (-race)
11. Example tests
12. Architecture boundaries
13. Vulnerability scan

This matches brief §3.12 (gofumpt → vet → build → lint → govulncheck-install → test-race → examples → arch-lint → vulnerability-scan) with one re-order: govulncheck *installation* is grouped near the arch-lint install (DRY), while the actual `govulncheck` *run* happens at the end as the final §3.12 step. This is the pre-existing structure and unchanged by Plan 03.
