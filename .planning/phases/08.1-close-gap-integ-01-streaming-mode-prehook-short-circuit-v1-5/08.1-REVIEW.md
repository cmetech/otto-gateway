## Warnings

### WR-01: CI workflow pins golangci-lint and go-arch-lint but not govulncheck

**File:** `.github/workflows/ci.yml:38`
**Issue:** The workflow header (lines 11-15) explicitly states the version-pin policy: "golangci-lint, go-arch-lint, and govulncheck are pinned to specific versions so a release in any of them does not break CI overnight." However, the actual env block declares `GOVULNCHECK_VERSION: latest`. This contradicts the stated policy and reintroduces the exact failure mode the policy is meant to prevent — a govulncheck release that flags a vulnerability OTTO has not yet patched could break CI without an explicit human decision to upgrade. Phase 08.1's WARNING-01 fix is specifically about the trust-gate sequence being reproducible; an unpinned scanner is the weakest link in that gate. (Note: golangci-lint-action also uses `@v6` major-pin rather than a SHA, and `mvdan.cc/gofumpt@latest` at line 63 has the same drift hazard, but govulncheck is the one explicitly called out as pinned in the workflow's own preamble.)
**Fix:**
```yaml
# Pinned tool versions. Bump together with the local toolchain.
GOLANGCI_LINT_VERSION: v1.62.2
GO_ARCH_LINT_VERSION:  v1.15.0
GOVULNCHECK_VERSION:   v1.1.4   # was: latest — pin to match the policy stated above
```
Consider similarly pinning `mvdan.cc/gofumpt@latest` on line 63 to a tag, since `gofumpt` issuing a formatting-rule change can break the `gofumpt -d` step overnight without a code change.

### WR-02: OpenAI streaming short-circuit test uses `HasPrefix` while Ollama/Anthropic siblings require exact match — weaker header-leak detector

**File:** `internal/adapter/openai/handlers_shortcircuit_test.go:175`
**Issue:** The Ollama short-circuit test (`handlers_shortcircuit_test.go:128`) and Anthropic short-circuit test (`handlers_shortcircuit_test.go:135`) both require `ct != "application/json"` (exact match). The OpenAI sibling uses `!strings.HasPrefix(ct, "application/json")`. The asymmetry is dangerous because the test's stated purpose (line 173 "the load-bearing wire-shape canary against header-leak regressions") is to catch a regression where `text/event-stream; charset=utf-8` leaks from `runSSEEmitter`. The handler-level `writeError` writes exactly `application/json` (no charset), and the SSE emitter writes `text/event-stream` — a regression that wrote `application/json; charset=utf-8` would pass this test but indicate the handler-side helper had drifted (e.g., someone added charset detection at the wrong layer). The Anthropic test (same surface category) uses exact-match and is the better template. Note: the openai test uses `httptest.NewServer` while the Ollama/Anthropic tests use `httptest.NewRecorder` directly — the round-trip through `net/http`'s response wire may legitimately introduce header normalization, so an exact-match assertion here may have driven the choice. Even so, accepting any `application/json*` prefix is broader than needed; tightening to `==` matches the sibling tests' rigor without a real failure risk.
**Fix:**
```go
// Was:
if !strings.HasPrefix(ct, "application/json") {
    t.Errorf("Content-Type: got %q, want application/json prefix ...", ct)
}
// Should be:
if ct != "application/json" {
    t.Errorf("Content-Type: got %q, want exactly %q (Pitfall 3 / T-08.1-HEADER-LEAK)", ct, "application/json")
}
```
If the round-trip introduces unavoidable normalization (e.g., charset appended by httptest.Server's writer), document why and assert against both literal forms.

## Info

### IN-01: Test-fake `StopWatchdog()` returns are inconsistent across packages

**File:** `internal/adapter/ollama/ndjson_test.go:54`, `internal/adapter/openai/sse_golden_test.go:124`, `internal/adapter/anthropic/handlers_test.go:169`
**Issue:** The three `fakeRunHandle.StopWatchdog()` implementations differ:
- Ollama returns `func() bool { return true }` (always-success stop)
- OpenAI returns `nil` (explicit nil)
- Anthropic returns `nil`

Two of three return nil — which is the production short-circuit path's actual behavior — but Ollama returns a real function. This means the nil-guard in `ollama/handlers.go:202` (`if stop := run.StopWatchdog(); stop != nil { stop() }`) is exercised by Ollama tests with `stop != nil`, while the OpenAI and Anthropic tests exercise the `stop == nil` branch. Both branches of the nil-guard get coverage across the three packages, but the divergence is incidental rather than intentional; future test authors copying one of the three patterns risk locking in a sub-pattern. None of this is a bug — the production `*engine.Run.StopWatchdog()` always returns nil on a PreHook short-circuit per `engine.go:150` documentation.
**Fix:** Normalize all three fakes to `return nil` (matches production short-circuit behavior). Add a one-line comment naming Phase 08.1 Pitfall 4 so the next test author understands the guard rationale.

### IN-02: PII Summary allocated on every request — never collected when AuthHook short-circuits

**File:** `internal/adapter/ollama/handlers.go:100`, `internal/adapter/openai/handlers.go:98`, `internal/adapter/anthropic/handlers.go:151`
**Issue:** All three handlers call `stampPluginCtx(ctx, r)` before `eng.Run`, which allocates a fresh `*pii.Summary` via `pii.NewSummary()`. When AuthHook short-circuits (the new Phase 08.1 path), no PIIRedactionHook or LoggingHook runs to consume the Summary — it is allocated and immediately discarded. On a high-rate bad-bearer attack this is one small heap allocation per request plus the ULID generation. Not a correctness defect; flagging as a minor efficiency note. If a future phase adds rate-limit-by-failed-auth (per the Deferred Ideas: 429 short-circuit), the allocation cost matters more.
**Fix:** No action recommended for v1.5. If revisited, consider lazy Summary allocation behind a hook-side getter so only the PIIRedactionHook pays the cost.

### IN-04: Tools like `gofumpt` and `golangci-lint-action` use floating refs in CI

**File:** `.github/workflows/ci.yml:63`, `.github/workflows/ci.yml:86`
**Issue:** Line 63: `go install mvdan.cc/gofumpt@latest` resolves to whatever the latest tag is at run time — a `gofumpt` rule change between runs will flip the green/red status without a repo commit. Line 86 uses `golangci/golangci-lint-action@v6` (major-version tag, not a commit SHA). Both contradict the stated "pinned … so a release does not break CI overnight" intent in the workflow preamble. GitHub Actions best practice (and Anthropic's own pattern in many recent repos) is to pin third-party actions to commit SHAs. This is an upstream-supply-chain hardening item rather than a Phase 08.1 regression.
**Fix:**
```yaml
- name: Install gofumpt
  run: go install mvdan.cc/gofumpt@v0.7.0  # pin to a tag

- name: golangci-lint
  uses: golangci/golangci-lint-action@a4f60bb28d35aeee14e6880718e0c85ff1882e64  # v6.1.1
```

## Notes on Architectural Choices (NOT findings)

1. **Hard-coded `http.StatusUnauthorized`** in all four short-circuit blocks (D-04). AuthHook is the only v1 PreHook short-circuit producer; status derivation is deliberately deferred until a non-401 producer (rate-limit → 429, content-mod → 403) lands. Documented in Deferred Ideas.

3. **`/admin` and Ollama list-mode stubs bypass AuthHook** — accepted v1 risk per WARNING-02 + WARNING-04, documented (not fixed) in Phase 08.1 Slice B. Not part of the code-review scope here.
4. **`/health/hooks` is auth-exempt** by design (operator owns the boundary). Out of scope.
