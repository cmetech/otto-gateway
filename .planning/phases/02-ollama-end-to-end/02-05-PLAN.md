<threat_model>
## Trust Boundaries

The pool spawns subprocesses (`*acp.Client` per slot) and routes engine calls
through them. The subprocess spawn surface is the highest-risk perimeter in
Phase 2 (gosec G204 territory). Operator-controlled KIRO_CMD / KIRO_ARGS env
vars feed into exec.CommandContext via acp.Config (acp.New).

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-22 | Tampering / Elevation of Privilege | Operator sets KIRO_CMD to a malicious binary; pool spawns it | accept | Operator-controlled trust boundary. PROJECT.md confirms env-var-write access is equivalent to OS access. acp.Config + acp.New use exec.CommandContext (not shell interpolation); gosec G204 is suppressed at the acp.client.go:162 site with explicit justification (Phase 1). No new spawn surface in Plan 05. |
| T-02-23 | Denial of Service | Subprocess hang during Initialize() blocks warmup forever | mitigate | Warmup propagates the caller's context. main.go (Plan 06) constructs warmupCtx with a deadline (planner discretion in Plan 06). Phase 1's acp.Initialize observes ctx cancellation. |
| T-02-24 | Information Disclosure | Slot leak via missing release in error path | mitigate | Pool.NewSession releases the slot on NewSession error before returning. Pool.Prompt's stream wrapper uses sync.Once-guarded release so double-release is impossible AND release happens exactly once on Result(). TestPool_NewSession_RequiresWarmup verifies the acquire path; integration smoke (Plan 06) exercises the full happy path. |
