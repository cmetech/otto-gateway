<threat_model>
## Trust Boundaries

Plan 02 introduces the HTTP-layer auth + IP allowlist surface. Trust boundary is
the inbound HTTP request crossing into the chi handler chain. Untrusted inputs:
`Authorization` header, `X-Forwarded-For` header, `RemoteAddr` (when run behind
no proxy, untrusted IPs can connect; when behind a proxy, XFF becomes the
trustworthy source — Loop24 laptop deployment has no proxy per Assumption A3).

The actual mounting of these middlewares into the chi sub-router happens in
Plan 06 (server wiring). The middlewares themselves are pure functions over
(cfg, request) → response; their threat surface is fully contained in this plan.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Information Disclosure | Bearer token timing side-channel (byte-by-byte recovery via response timing) | mitigate | bearer.go uses `subtle.ConstantTimeCompare` (Pattern 3). TestBearer_ValidToken_PassesThrough + TestBearer_InvalidToken_Rejects + TestBearer_MultiToken_ValidatesAny exercise the comparison surface. |
| T-02-06 | Spoofing | X-Forwarded-For spoofing bypasses IP allowlist when no proxy is in front (laptop deployment) | mitigate | Codex H-7 fix: XFF is NOT trusted by default. extractClientIP uses r.RemoteAddr unconditionally unless cfg.TrustXForwardedFor is true (operator opt-in via AUTH_TRUST_XFF env var added in Plan 03). TestIPAllowlist_XFFNotTrustedByDefault asserts a spoofed `X-Forwarded-For: 127.0.0.1` from a non-allowlisted RemoteAddr is denied. TestIPAllowlist_XFFRespectedWhenEnabled proves the opt-in path still works for proxy-fronted deployments.
