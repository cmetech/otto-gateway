<objective>
Create the `internal/engine` package: the orchestrator that converts a
canonical.ChatRequest into ACP block flattening, session lifecycle management,
per-request cwd derivation, and chunk aggregation into a canonical.ChatResponse.
The package also declares the Phase 8 hook-chain seam (PreHook/PostHook
interfaces with empty slice fields in Config) and the consumer-defined
ACPClient + Stream interfaces that decouple engine from `internal/acp` directly.

Purpose: this is the canonical-engine layer per the adapter-over-canonical
architecture (brief §3.13, CONTEXT.md D-01..D-05). All three Phase 2/3/3.1
surface adapters call into the engine via a thin Engine consumer interface;
the engine itself depends only on canonical types and the ACPClient interface.

Output: nine new files in `internal/engine/`. Heavy plan — broken into THREE
tasks: (T1) skeleton/interfaces/hooks/engine.go, (T2) pickCwd + buildBlocks
pure-function pair with property/golden tests, (T3) Collect + Run with fake-ACP
test harness.

Wave 2 — depends on Plan 01 (canonical chat types). Cannot run in parallel
with Plan 05 (pool) because Plan 05 implements the ACPClient interface this
plan declares.

Key insight: *acp.Stream exposes Chunks as a FIELD (not a method). engine.Stream
declares Chunks() as a METHOD. A small shim adapter is required at the engine
boundary so *acp.Stream satisfies engine.Stream — or, equivalently, pool's
Prompt wrapper returns a struct that has a Chunks() method delegating to the
field. Planner: place the shim in internal/engine/acp_stream.go (a tiny adapter
file) OR fold it into Plan 05's pool.Prompt return type. THIS PLAN takes the
former approach: declare a `wrapACPStream(*acp.Stream) engine.Stream` helper
in internal/engine/engine.go so the engine package owns its own boundary
adapter.
