<objective>
Create the `internal/engine` package: the orchestrator that converts a
canonical.ChatRequest into ACP block flattening, session lifecycle management,
per-request cwd derivation, and chunk aggregation into a canonical.ChatResponse.
The package also declares the Phase 8 hook-chain seam (PreHook/PostHook
interfaces with empty slice fields in Config) and the consumer-defined
ACPClient + Stream interfaces that decouple engine from `internal/acp` directly.

Purpose: this is the canonical-engine layer per the adapter-over-canonical
architecture (brief §3.13, CONTEXT.md D-01..D-05). All three Phase 2/3/3.1
surface adapters call into the engine via a thin Engine consumer interface;
the engine itself depends only on canonical types and the ACPClient interface.

Output: nine new files in `internal/engine/`. Heavy plan — broken into THREE
tasks: (T1) skeleton/interfaces/hooks/engine.go, (T2) pickCwd + buildBlocks
pure-function pair with property/golden tests, (T3) Collect + Run with fake-ACP
test harness.

Wave 2 — depends on Plan 01 (canonical chat types). Cannot run in parallel
with Plan 05 (pool) because Plan 05 implements the ACPClient interface this
plan declares.
</objective>

Key insight: *acp.Stream exposes Chunks as a FIELD (not a method). engine.Stream
declares Chunks() as a METHOD. A small shim adapter is required at the engine
boundary so *acp.Stream satisfies engine.Stream — or, equivalently, pool's
Prompt wrapper returns a struct that has a Chunks() method delegating to the
field. Planner: place the shim in internal/engine/acp_stream.go (a tiny adapter
file) OR fold it into Plan 05's pool.Prompt return type. THIS PLAN takes the
former approach: declare a `wrapACPStream(*acp.Stream) engine.Stream` helper
in internal/engine/engine.go so the engine package owns its own boundary
adapter.

<threat_model>
## Trust Boundaries

The engine sits below the HTTP layer (no inbound network) and above the ACP
layer (which spawns subprocesses). Engine's input is `*canonical.ChatRequest`
constructed by an adapter — already validated at the HTTP layer (8 MB cap,
JSON parse, schema). The engine's primary trust boundary is the
canonical→ACP block flattening (buildBlocks) and the cwd derivation (pickCwd),
both of which influence subprocess behavior.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-15 | Tampering | Crafted X-Working-Dir header sets cwd to attacker-chosen path (e.g., /etc) | accept | Per RESEARCH.md Security Domain V12: the value reaches exec.Cmd.Dir, which is a working-directory hint, not a file path being opened. Worst case: kiro-cli starts in an unfamiliar directory and likely fails. Operator-deployment trust boundary; auth middleware (Plan 02 + Plan 06) gates inbound requests when AUTH_TOKEN is set. |
| T-02-16 | Tampering | resource_link URI with shell metacharacters reaches cwd via longest-common-parent | mitigate | longestCommonParent operates only on file:// URIs after url.Parse; non-file URIs are silently dropped. The url.Parse layer rejects malformed URIs. filepath.Clean is implicit via filepath.FromSlash → no path-traversal semantics escape (cwd is a directory hint, not an open()d path). |
| T-02-17 | Information Disclosure | Engine logs request text including PII | accept | Engine does not log request content. Adapter (Plan 06) emits accessLog (path + status) but not body. Future deployments needing PII redaction can hook Phase 8's LoggingHook. |
| T-02-18 | Denial of Service | Pathologically long Messages slice causing buildBlocks to allocate huge string | accept (cross-plan) | Bounded at the HTTP ingress via `http.MaxBytesReader(8 MB)` (Plan 06). Once the body is parsed and bounded, the strings.Builder allocation is bounded by extension. |
| T-02-19 | Denial of Service | Slow client reading Collect's response, holding the ACP slot indefinitely | mitigate (cross-plan) | ctx is propagated end-to-end; HTTP request cancellation cancels ctx, which Run/Prompt observes. Phase 4 adds explicit session/cancel propagation; Phase 2's Collect respects ctx through the underlying acp.Client. |
| T-02-20 | Repudiation | Engine error after NewSession leaks session resources in kiro-cli | mitigate | RESEARCH.md Pitfall 6: Run calls e.cfg.ACP.Cancel(sid) on every error path after NewSession succeeds. TestEngineRun_PromptError_CancelsSession + TestEngineRun_SetModelError_CancelsSession assert this contract. |
| T-02-21 | Elevation of Privilege | PreHook implementation returning attacker-controlled response bypassing engine | accept | PreHook implementations are in-process Go code registered at boot — operator-controlled trust boundary. Phase 8 will register only stdlib-equivalent hooks (RequestID, Auth, Logging). |

Residual exposure: cwd derivation from operator-influenced inputs (T-02-15)
is accepted because the cwd is a directory hint, not an open()d file. Phase 5
operator may add an os.Stat sanity check at the pool layer if desired.
