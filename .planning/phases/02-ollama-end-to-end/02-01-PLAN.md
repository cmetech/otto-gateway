<threat_model>
## Trust Boundaries

This plan introduces no new trust boundary: canonical is a leaf package of pure
data types with no I/O. The only inbound surface is the Go compiler. Downstream
plans (engine, pool, adapter/ollama) will be the layers where untrusted client
data flows into canonical structs and where validation must live.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | ChatRequest field shape drift | mitigate | Discriminator coverage test (Test 2, Test 3) locks the iota values so adapters cannot silently misread wire data after a refactor. |
| T-02-02 | Tampering | Accidental JSON-tag addition to canonical | mitigate | TestNoJSONTags reflective sweep (Test 5) blocks any future PR that re-introduces JSON tags into canonical (D-11 discipline). |
| T-02-03 | Information Disclosure | Sensitive content (system prompt, tool args) leaking via debug logs | accept | No logging in canonical types; downstream plans (engine, adapter) own the redaction posture. No PII handled at this layer. |
| T-02-04 | Denial of Service | Unbounded Metadata map[string]any allowing oversized client payloads | accept | Bound enforcement lives at the adapter HTTP layer (Plan 06 applies `http.MaxBytesReader(8MB)` per Node parity). Canonical accepts any in-memory map; bounding belongs at the wire ingress. |

Residual exposure: none at this layer (no auth, no I/O). The "protected by Plan
06 middleware" disposition for T-02-04 is explicit.
</threat_model>
