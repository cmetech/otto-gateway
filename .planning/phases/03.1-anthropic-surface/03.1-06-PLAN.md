<objective>
Close VERIFICATION.md Gap 2 / REVIEW CR-02: extractToken in internal/auth/bearer.go uses
strings.HasPrefix(authHeader, "Bearer ") — an exact-case match — which violates RFC 7235 §2.1
and RFC 6750 case-insensitive scheme requirements. A client sending `Authorization: bearer
s3cret` (lowercase) silently falls through to the x-api-key branch. This causes two real
defects: (a) valid lowercase-Bearer credentials are rejected with 401 when no x-api-key is
also supplied, and (b) the documented D-15 "Authorization wins to defeat downgrade attack"
guard is BROKEN for lowercase clients — `bearer wrong + x-api-key good` auths as good.

Purpose: Restore RFC 7235/6750 compliance and the D-15 downgrade-attack guard for all case
variants of the Bearer scheme. The fix is a single substitution in extractToken with a length
guard. Two new whitebox tests in auth_internal_test.go pin the contract: the lowercase scheme
must be accepted, AND the lowercase scheme with a wrong token must still beat the x-api-key
fallback (preserves D-15 precedence semantics).

Output: bearer.go's extractToken uses `strings.EqualFold` with a `len(authHeader) >=
len(bearerPrefix)` guard; auth_internal_test.go gains TestExtractToken_LowercaseBearerSchemeAccepted
and TestExtractToken_LowercaseBearer_DoesNotFallThroughToXAPIKey; `go test ./internal/auth/
-race -count=1` is fully green; no regression in TestBearer_DualHeader or the broader auth
package suite.

From internal/auth/bearer.go (lines 77-83) — the function under repair:
```go
func extractToken(r *http.Request) string {
    const bearerPrefix = "Bearer "
    if authHeader := r.Header.Get("Authorization"); strings.HasPrefix(authHeader, bearerPrefix) {
        return authHeader[len(bearerPrefix):]
    }
    return r.Header.Get("x-api-key")
}
```

From the doc-comment on extractToken (lines 62-76) — the D-15 contract this MUST preserve:
- "try `Authorization: Bearer <token>` first, fall back to `x-api-key: <token>` only when
  Authorization is absent or non-Bearer"
- "Precedence is deliberately Authorization-wins: when BOTH headers are present the Bearer
  value is used unconditionally. This blocks a downgrade attack where the attacker supplies
  a bad Bearer alongside a stolen x-api-key"

The fix MUST honor BOTH the case-insensitivity requirement AND the Authorization-wins
precedence. The chosen pattern (per VERIFICATION.md gap 2 and REVIEW CR-02):
```go
if len(authHeader) >= len(bearerPrefix) &&
    strings.EqualFold(authHeader[:len(bearerPrefix)], bearerPrefix) {
    return authHeader[len(bearerPrefix):]
}
```
The `len(authHeader) >= len(bearerPrefix)` guard prevents an out-of-bounds panic when the
Authorization header is shorter than 7 bytes (or absent — `r.Header.Get` returns ""). The
slice `authHeader[:len(bearerPrefix)]` is then exactly the scheme + space; strings.EqualFold
compares it case-insensitively to "Bearer ".
