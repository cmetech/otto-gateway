<objective>
Close VERIFICATION.md Gap 1 / REVIEW CR-01: tool_use.input field is silently dropped from the
Anthropic wire when a canonical ToolUsePart has nil or empty-map Input, violating the Anthropic
Messages spec's "tool_use.input is required (always a JSON object, even {})" contract. The
@anthropic-ai/sdk Zod parser throws on a missing input field, so any no-arg tool call surfaces
as a parse exception in loop24-client.

Purpose: Restore wire compliance for the load-bearing ANTH-03 tool-call round-trip requirement
so a tool_use block with no arguments still serializes as `"input":{}`. The existing populated-
Input test (TestChatResponseToMessage_ToolUseInputIsObject) happens to pass {q:..,limit:5},
which masks the defect — this plan introduces the missing nil-Input regression test FIRST
(RED), then applies the minimal render-time coercion (GREEN) so both old and new tests pass.

Output: render.go is patched so chatResponseToMessage coerces nil/empty ToolUse.Input to
map[string]any{} before constructing the contentBlock; render_test.go gains
TestChatResponseToMessage_ToolUseEmptyInput_RendersEmptyObject asserting the wire carries
`"input":{}` for a nil-Input ToolUsePart; `go test ./internal/adapter/anthropic/ -race
-count=1` is green; the existing TestChatResponseToMessage_ToolUseInputIsObject continues to
pass unchanged.
</objective>

    if part.ToolUse != nil {
        // Anthropic spec requires tool_use.input as a JSON object even when
        // empty. Coerce nil/empty to map[string]any{} so json.Marshal emits
        // "input":{} rather than dropping the field via omitempty (CR-01 fix).

            Name:  part.ToolUse.Name,
            Input: input,
        })
