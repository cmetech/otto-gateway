# Phase 3.1: Anthropic Surface - Context

**Gathered:** 2026-05-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 3.1 brings the **second adapter** (`internal/adapter/anthropic`)
online, sharing the same canonical engine as Phase 2's Ollama adapter.
`POST /v1/messages` serves loop24-client (GSD Pi CLI, `@anthropic-ai/sdk`
v0.90.x) end-to-end against `kiro-cli` — **non-streaming AND
`messages.stream()` SSE on day one** (the streaming path is non-negotiable
per the 2026-05-23 Key Decision, because `messages.stream()` is the
dominant call path in loop24-client and Anthropic's SSE event sequence is
structurally different from OpenAI's and cannot be deferred to Phase 4).

This phase proves the adapter-over-canonical layout supports a second
surface (Ollama + Anthropic). Phase 3 will subsequently add OpenAI as
the third adapter under the same `/v1` prefix (SURF-08 disambiguates by
endpoint: `/v1/messages` vs `/v1/chat/completions`).

**Deliverables (per ROADMAP.md Phase 3.1 success criteria):**

1. `curl -X POST http://localhost:11434/v1/messages -H 'x-api-key: …' -H 'anthropic-version: 2023-06-01' -d '{"model":"auto","max_tokens":256,"messages":[{"role":"user","content":"hi"}]}'`
   returns an Anthropic-compatible JSON response with top-level `id`,
   `type:"message"`, `role:"assistant"`, `content:[{type:"text",text:"…"}]`,
   `stop_reason`, `usage` — sourced from the same canonical engine that
   serves `/api/chat`.
2. loop24-client configured with `ANTHROPIC_BASE_URL=http://localhost:11434`
   and a valid `ANTHROPIC_API_KEY` completes BOTH a non-streaming
   `messages.create({stream:false})` AND a streaming `messages.stream({...})`
   round-trip, with zero changes to loop24-client code.
3. Streaming emits the full Anthropic SSE event sequence: `message_start`
   → (`content_block_start` → `content_block_delta`+ → `content_block_stop`)+
   → `message_delta` → `message_stop`, with `event:` and `data:` framing
   and `ping` keepalives every ~15s, all sourced from the same
   `<-chan canonical.Chunk` produced by `engine.Run(ctx, req)`.
4. Tool-call requests round-trip: `tool_use` content blocks emitted in
   canonical shape and rendered with `input` as an object (Anthropic
   style — OpenAI's JSON-string rendering is Phase 3); `tool_result`
   blocks accepted on the inbound side.
5. `ENABLED_SURFACES` env var is introduced with default `ollama,anthropic`
   enabling both surfaces. Setting `ENABLED_SURFACES=ollama` disables the
   Anthropic adapter at boot; Phase 3 will extend the default to include
   `openai`. `internal/adapter/anthropic` imports only `internal/canonical`
   + `internal/plugin`; the `.go-arch-lint.yml` boundary check passes.
6. Header contract enforced: missing `anthropic-version` returns a
   canonical `invalid_request_error` rendered in Anthropic's
   `{"type":"error","error":{"type":"…","message":"…"}}` shape; the
   gateway accepts both `x-api-key` and `Authorization: Bearer …` auth
   modes (loop24-client uses both depending on provider).

**Requirements covered:** ANTH-01..07, SURF-08.

**Explicitly NOT in Phase 3.1:**

- OpenAI surface (`/v1/chat/completions`, `/v1/completions`, `/v1/models`)
  — Phase 3.
- Real `session/cancel` on client-disconnect — Phase 4 (STRM-04). Phase
  3.1 relies on `r.Context()` propagation only; the underlying
  `ACPClient.Prompt` ctx unwinds naturally.
- Tool dispatch / execution (`coerceToolCall`, OpenAI tool-call rendering)
  — Phase 6 (TOOL-01..03). Phase 3.1 routes `tool_use` and `tool_result`
  shapes faithfully but does NOT execute tools or coerce JSON-as-text.
- Embeddings (`/v1/embeddings`) — Phase 7.
- Hook chain implementations (RequestIDHook, AuthHook, LoggingHook) —
  Phase 8. Phase 2's empty `PreHook`/`PostHook` slice seam carries
  forward unchanged.
- Real warm pool (`POOL_SIZE > 1` default), dead-slot detection, stateful
  sessions (`X-Session-Id`) — Phase 5.
- Cross-compile CI matrix gating merges — Phase 9.
- Anthropic-via-Vertex (`@anthropic-ai/vertex-sdk`) / Anthropic-via-Bedrock
  (`@aws-sdk/client-bedrock-runtime`) variants — SURF-V2-03 (deferred).
- `POST /v1/messages/count_tokens` — out of scope (planner decides
  between 404 and a passthrough stub; Anthropic SDK calls it for token
  pre-flight but loop24-client does not require it).
- `redacted_thinking` content blocks on the outbound side — kiro-cli has
  no equivalent; Phase 3.1 never emits this variant.
- Token-accurate `usage` reporting — kiro-cli's `session/prompt` response
  does not report token counts; Phase 3.1 returns honest zeros (see D-12).

<decisions>
## Implementation Decisions

### Engine / canonical surface (cross-cutting)

- **D-01: Anthropic adapter uses `engine.Run()` + ranges `Chunks()` directly.**
  No new engine helper. The streaming SSE handler calls
  `engine.Run(ctx, req)`, gets back `*engine.Run`, and ranges
  `run.Stream().Chunks()` itself. Non-streaming uses the existing
  `engine.Collect`. Matches the Phase 2 D-01 engine surface intent —
  `Run` is the streaming path, `Collect` is the non-streaming sugar.
  Zero engine-package signature churn.

- **D-02: `engine.Collect` STOPS dropping `ChunkKindThought`; aggregates
  thoughts parallel to text into a `ContentPart{Kind: ContentKindThinking}`
  on the assembled `Message.Content`.** Current `collect.go:50-58` has an
  explicit "intentionally drop" comment naming Phase 3.1 — this phase
  cashes that promise. The change adds a second `strings.Builder` for
  thoughts; when non-empty after the stream ends, `assembleChatResponse`
  appends a `ContentPart{Kind: ContentKindThinking, Text: thoughts}` to
  `Message.Content` AFTER the text part. Phase 2 Ollama's `render.go`
  is updated trivially to either (a) drop the thinking part (current
  behavior — `ollamaChatResponseMessage.Thinking` stays empty) or (b)
  fill `ollamaChatResponseMessage.Thinking` (already in the struct from
  Phase 2's forward design). Planner picks (a) or (b) based on Node
  reference parity check.

- **D-03: Any `ChunkKind` change opens a new SSE block in the
  Anthropic adapter's streaming state machine.** Block 0 = text, Block 1
  = thinking (when `Thought` arrives after `Text`), Block 2 = text (if
  text resumes), etc. Tool-call chunks (Phase 6) slot in naturally as
  the next block kind. Matches Anthropic's wire model directly — every
  kind transition is a `content_block_stop` + index-bump + new
  `content_block_start`. No special-casing of thinking ordering at this
  layer.

### Anthropic adapter (`internal/adapter/anthropic`)

- **D-04: Adapter-local block-index state machine.** Per-request struct:
  ```go
  type sseEmitter struct {
      w           http.ResponseWriter
      flusher     http.Flusher
      blockIndex  int
      currentKind canonical.ChunkKind
      blockOpen   bool
      messageID   string
  }
  ```
  On each chunk: if `blockOpen && chunk.Kind != currentKind`, emit
  `content_block_stop` + bump `blockIndex` + emit `content_block_start`
  for the new kind + emit `content_block_delta`. If `!blockOpen`, emit
  `content_block_start` first. Writer is touched from exactly one
  goroutine (D-05). Canonical chunks stay simple (no `BlockIndex` field
  added to `canonical.Chunk` — that would violate D-11 from Phase 2).

- **D-05: Single select-loop in the SSE handler goroutine handles
  ping ticker + chunk channel + ctx.Done.** One goroutine, one
  `select { case c := <-chunks: writeEvent(...); case <-ticker.C:
  writeEvent("ping", {}); case <-ctx.Done(): return }`. Writer touched
  from exactly one goroutine — no mutex, no `sync.Map`, no surprise
  interleaving. `http.ResponseWriter`/`http.Flusher` is the writer
  pair; Flush is called after each event write. Testing is trivial via
  a controlled chunks channel + a fake ticker channel injected via the
  emitter struct.

- **D-06: Phase 3.1 relies on `ctx` propagation for client-disconnect;
  no explicit `engine.Cancel(sid)` is added.** When `r.Context()` is
  canceled the SSE select-loop's `ctx.Done` case fires and the handler
  returns. `engine.Run` was called with that ctx, so the underlying
  `ACPClient.Prompt` ctx unwinds and the stream naturally closes.
  STRM-04's explicit `session/cancel` over JSON-RPC is Phase 4 scope.
  Phase 3.1 SHOULD include a debug log line on `ctx.Done` (option 3
  from the discussion) — pure observability, costs nothing.

- **D-07: `anthropic-version` header is checked per-handler in
  `adapter/anthropic/handlers.go`.** First line of `handleMessages`:
  if `r.Header.Get("anthropic-version") == ""` → return 400
  `invalid_request_error`. Any non-empty value is accepted — gateway
  is shape-only, doesn't switch behavior on api version. If a future
  endpoint (`count_tokens`) lands, lift this into a chi middleware on
  the anthropic protected router.

- **D-08: System block normalization — string verbatim, array joined
  with `\n\n`.** Adapter's `wireToChatRequest` checks if `system` is a
  JSON string or array. String → `req.System = s`. Array → walk
  text-typed entries, concatenate `.text` values with `\n\n` separator.
  Non-text blocks and `cache_control` are dropped. `req.System` stays
  string-typed — zero canonical churn. Phase 7+ may revisit if
  prompt-caching becomes load-bearing.

- **D-09: `tool_result.content` normalization — string verbatim, array
  joined and images dropped with debug log.** wire.go normalizes both
  shapes to a flat string for `canonical.ToolResultPart.Content`. Array
  form: walk text-typed blocks, join with `\n`. Image blocks inside
  `tool_result` get dropped with a debug log entry (Phase 3.1 doesn't
  dispatch tools — the tool_result is read-only history forwarded to
  kiro-cli). Phase 6 may widen `ToolResultPart.Content` to `[]ContentPart`
  when tools actually execute.

- **D-10: `anthropic-beta` headers and `cache_control` fields are
  accepted-and-ignored; logged at debug.** Adapter reads them, logs
  them at debug level for observability, then discards. Never
  persisted to canonical. Matches Phase 2's `KeepAlive`/`Options`
  accepted-and-ignored pattern. "Passed through" in ANTH-04 means
  "doesn't 400", not "forwarded to backend" — kiro-cli has no
  equivalent.

- **D-11: Inbound `thinking` blocks preserve as
  `ContentPart{Kind: ContentKindThinking, Text: <thinking text>}` in
  canonical.** wire.go's content-block decoder recognizes
  `{type:"thinking",thinking:"..."}` and maps to the existing
  `ContentKindThinking` dormant seam from Phase 2 D-09. Engine's
  `buildBlocks` gains a small extension (planner's discretion):
  either a new `[Reasoning]` bracketed section, or fold the thinking
  text into `[Assistant]` alongside the response text. Either way the
  thinking content reaches kiro-cli as context for multi-turn
  conversations.

- **D-12: `usage` block returns honest zeros — `input_tokens: 0,
  output_tokens: 0`.** kiro-cli's `session/prompt` response doesn't
  report token counts and Phase 3.1 does NOT add a local tiktoken-style
  counter. `cache_creation_input_tokens` and `cache_read_input_tokens`
  stay omitted (zero-value, `omitempty` JSON tag). Documented limitation;
  loop24-client tolerates zeros (SDK doesn't crash). Phase 7+ may add
  real counting if kiro-cli exposes them or a local tokenizer is
  acceptable.

- **D-13: `redacted_thinking` content blocks are NEVER emitted on the
  outbound side in Phase 3.1.** kiro-cli has no equivalent. The adapter
  only emits `{type:"thinking",thinking:"..."}` blocks for
  `ChunkKindThought` data. Inbound `redacted_thinking` blocks in
  conversation history are passed through transparently (or dropped at
  the wire-decode layer — planner picks based on whether kiro-cli
  preserves them in subsequent turns).

- **D-14: Per-request `cwd` derivation reuses the Phase 2 pipeline.**
  Anthropic content blocks include `resource_link` (per Anthropic
  Messages API spec). wire.go populates
  `canonical.ChatRequest.ResourceLinks` from every `resource_link`
  block in the inbound `messages[].content` array. `engine.pickCwd`
  (Phase 2 D-16) already consumes `ResourceLinks` via the
  longest-common-parent step — zero engine change. The
  `X-Working-Dir` header override still applies as the
  highest-priority cwd source.

### Auth + routing (`internal/auth`, `internal/server`, `internal/config`)

- **D-15: `auth.Bearer` middleware accepts BOTH `Authorization: Bearer
  <token>` AND `x-api-key: <token>` headers globally.** Modify
  `internal/auth/bearer.go`: try `Authorization: Bearer` first, fall
  back to `x-api-key` if absent. Same constant-time-compare path; same
  `AUTH_TOKEN` env var. All three surfaces accept both headers — Ollama
  clients never send `x-api-key` (no impact); Pi-SDK never sends
  `x-api-key` (no impact); loop24-client uses both (works). One
  middleware, one mental model.

- **D-16: `ENABLED_SURFACES` env var introduced in Phase 3.1 with
  default `ollama,anthropic`.** Loaded by `config.Load` via
  `getEnvStrSliceComma("ENABLED_SURFACES", []string{"ollama","anthropic"})`.
  Semantics:
  - Unset or empty string → use default (`["ollama","anthropic"]`).
  - Comma-separated list → explicit enable list (e.g.,
    `ENABLED_SURFACES=ollama` disables anthropic; `ENABLED_SURFACES=anthropic`
    disables ollama).
  - **Unknown surface name (e.g., `olama` typo) → `config.Load` returns
    an error; binary exits non-zero with a clear message.** Fail-fast on
    misconfig aligns with trust-gate philosophy.
  - Phase 3 extends the default to `ollama,anthropic,openai` and adds
    `openai` to the allowed-names set.

- **D-17: `server.Config` grows parallel `AnthropicPath` +
  `AnthropicProtectedRouter` fields (mirrors Phase 2's `OllamaPath` +
  `OllamaProtectedRouter`).** `NewFromConfig` calls
  `r.Route(cfg.AnthropicPath, ...)` with the same auth middleware
  chain (`auth.Bearer` + `auth.IPAllowlist` + `AuthTrustXFF` per
  Codex H-7) and mounts `cfg.AnthropicProtectedRouter` under it.
  `cmd/main.go`'s `newApp` wires the anthropic adapter only when
  `cfg.EnabledSurfaces` contains `"anthropic"`. Phase 3.1 does NOT
  solve multi-mount-on-`/v1` yet — anthropic owns `/v1` solo;
  Phase 3 will deal with the OpenAI co-mount when it lands (likely
  via a refactor into a generic `Surfaces []SurfaceMount` list, but
  that's Phase 3's call).

- **D-18: `/v1/messages` is the ONLY `/v1` endpoint Phase 3.1 mounts.**
  `count_tokens` is out of scope. The anthropic protected router has
  exactly one route: `POST /messages` (mounted under
  `cfg.AnthropicPath`, default `/v1`). No version-equivalent exempt
  route is needed on the outer router — Anthropic has no `/version`
  surface in its public API; the existing `/api/version` (Ollama
  shape, exempt) and `/health` (gateway shape, exempt) are
  sufficient.

- **D-19: `ANTHROPIC_PATH_PREFIX` env var defaults to `/v1`.** Loaded
  alongside `OLLAMA_PATH_PREFIX` and `OPENAI_PATH_PREFIX`. Both
  `OPENAI_PATH_PREFIX` and `ANTHROPIC_PATH_PREFIX` default to `/v1`
  (SURF-08 — endpoint-level disambiguation). Setting
  `ANTHROPIC_PATH_PREFIX=/anthropic/v1` moves Anthropic to its own
  prefix without breaking Phase 3's OpenAI mount.

### Error handling

- **D-20: Anthropic error envelope `{"type":"error","error":{"type":"…","message":"…"}}`
  is emitted by an adapter-local helper `writeError(w, status, errorType,
  message)`.** Mirrors Phase 2 Ollama's `writeError` helper. The 8
  `error_type` values + HTTP status mapping (per ANTH-06):
  - `invalid_request_error` → 400 (bad JSON, missing `anthropic-version`,
    missing `model`, missing `max_tokens`, empty `messages`, bad
    content-block shape).
  - `authentication_error` → 401 (bad or missing api key when
    `AUTH_TOKEN` is set).
  - `permission_error` → 403 (IP allowlist denies; planner decides
    whether to render this as `permission_error` or `authentication_error`
    — Anthropic's spec is ambiguous about the difference for
    proxy-gateway 403s).
  - `not_found_error` → 404 (unknown endpoint under `/v1`).
  - `request_too_large` → 413 (body cap exceeded — same 4 MiB cap as
    Phase 2 chat body).
  - `rate_limit_error` → 429 (Phase 3.1 doesn't rate-limit; this
    mapping is forward-design for the Phase 8 budget hook).
  - `api_error` → 500 (engine errors, pool errors).
  - `overloaded_error` → 529 (pool empty / Acquire blocked beyond a
    threshold — Phase 5 territory; Phase 3.1 may never emit this).

<specifics>
## Specific Ideas

- **loop24-client (`~/Projects/repos/local/loop24-client`) is the
  SURF-08 acceptance bar for Phase 3.1.** When in doubt about
  Anthropic-shape fidelity (e.g., `usage` field omitempty semantics,
  `content_block_delta` payload shape, `signature_delta` handling),
  run loop24-client against the real Anthropic API (with valid
  `ANTHROPIC_API_KEY`) first and capture the wire shapes, then verify
  our Go version emits the same shapes. The user has loop24-client
  installed and reachable.

- **`@anthropic-ai/sdk@^0.90.0` is the ground-truth shape target.**
  Per `~/Projects/repos/local/loop24-client/package.json`. Shape drift
  vs this version is a Phase 3.1 regression risk. The SDK's
  `MessageStream` parser is strict about event-name spelling and
  `event:`/`data:` framing — Phase 3.1's SSE writer must emit byte-
  for-byte compatible framing.

- **Anthropic's SSE event sequence is structurally different from
  OpenAI's.** OpenAI uses `data: <chunk>\n\n` lines + a terminal
  `data: [DONE]\n\n` frame. Anthropic uses `event: <name>\ndata:
  <json>\n\n` for EVERY frame, with a defined event-name vocabulary
  (`message_start`, `content_block_start`, `content_block_delta`,
  `content_block_stop`, `message_delta`, `message_stop`, `ping`,
  `error`). This is WHY Phase 3.1 cannot defer streaming to Phase 4
  — the canonical chunk channel has to fan out to two structurally
  different SSE shapes (Anthropic now + OpenAI Phase 4).

- **The `ping` event is structural, not just keepalive theatre.**
  Anthropic's SDK starts a 60-second connection idle timer; without
  the `ping` every ~15s the SDK may abort the stream. Phase 3.1's
  ticker is load-bearing for correctness, not just nice-to-have.

- **Phase 3.1 owns `/v1` solo; Phase 3 deals with co-mount.** This is
  the only phase where `/v1` has a single surface. Phase 3 adds OpenAI
  under `/v1` with disambiguation by endpoint (`/v1/messages` vs
  `/v1/chat/completions`) per SURF-08. Phase 3 is responsible for the
  router-composition refactor (potentially folding into a generic
  `Surfaces []SurfaceMount` list per D-17 deferred decision).

- **No tool dispatch in Phase 3.1.** Tool `input` is rendered as an
  object (Anthropic style) when assistant emits `tool_use`; tool_result
  is accepted on inbound and forwarded into the [Tool result] block of
  the ACP wire. But Phase 3.1 does NOT execute tools, does NOT call
  `coerceToolCall` (that's Phase 6), and does NOT need the canonical
  ToolSpec → engine tool-list translation working (Phase 6).

- **The `interleaved-thinking-2025-05-14` beta is debug-only in Phase
  3.1.** ANTH-04 says we accept `anthropic-beta` headers and pass
  through. D-10 interprets this as "accept-and-ignore + debug log".
  When kiro-cli or a future Phase activates real beta semantics, the
  header value can be lifted into request metadata.
