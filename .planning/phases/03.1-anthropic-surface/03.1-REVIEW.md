# Phase 3.1: Code Review Report

**Reviewed:** 2026-05-24
**Depth:** standard
**Files Reviewed:** 38
**Status:** issues_found

## Summary

Phase 3.1 introduces the Anthropic Messages surface as a parallel adapter
mounted under `/v1` alongside the existing Ollama `/api` surface, sharing
the auth.Bearer + auth.IPAllowlist chain. The implementation is structured
cleanly along the TRST-04 boundary (consumer-defined Engine interface in
the anthropic package; concrete *engine.Engine bridged via main.go), and
the SSE state machine in `sse.go` uses a single-goroutine select-loop that
respects the D-05 "one writer" invariant. Golden-fixture tests pin the
byte-exact wire shape against three representative chunk sequences; the
integration test enforces strict SSE byte-framing (`event:` → `data:` →
blank) against real kiro-cli output. Auth dual-header precedence is
covered by both whitebox and middleware-level tests with explicit
downgrade-attack cases.

The defects below are concentrated on three axes: (1) **Anthropic spec
compliance** — `tool_use.input` may be silently dropped from the wire when
the canonical `Input` map is nil/empty, which violates the SDK's required-
object contract; (2) **Auth interop** — case-sensitive `"Bearer "` prefix
match falls through to the x-api-key path on lowercase scheme tokens,
contrary to RFC 7235 §2.1 (and silently breaks clients that emit lowercase
schemes); (3) **Observability** — SSE post-header errors are logged at
Debug, so real writeEvent failures vanish from operator-visible logs even
though they indicate a degraded stream.

The remainder are quality issues: stale comments contradicting the code,
a fixed message-id on crypto/rand failure that allows ID collision across
concurrent requests, and a dead `errors.Is` import kept alive purely to
silence the linter.

## Critical Issues

### CR-01: tool_use.input dropped from wire when canonical map is nil/empty

**File:** `internal/adapter/anthropic/render.go:42-49`
**Issue:** The `contentBlock` struct tags `Input map[string]any` with
`json:"input,omitempty"`. When a tool call carries no arguments
(`Input == nil` or `Input == map[string]any{}`), `encoding/json` drops the
field entirely from the marshaled output. The Anthropic Messages spec
defines `tool_use.input` as **required** — it must always be present as a
JSON object, even when empty. `@anthropic-ai/sdk` parses the response
through a Zod schema that throws on a missing `input` field, so an
assistant turn that emits a no-arg tool call will surface as a parse
exception in loop24-client (and any other SDK consumer) rather than as a
valid `ContentBlock`. Note that the chatResponseToMessage code path
(line 104-112) passes `part.ToolUse.Input` straight through without
forcing a non-nil empty map, so the bug is reachable from any engine that
produces a `ToolUsePart` with nil Input. The existing
`TestChatResponseToMessage_ToolUseInputIsObject` test happens to pass a
populated map (`{"q":"ducks","limit":5}`) which masks the defect.

**Fix:**
```go
// render.go contentBlock — drop omitempty on Input. The Anthropic
// spec requires the field to be a JSON object even when empty.
type contentBlock struct {
    Type     string         `json:"type"`
    Text     string         `json:"text,omitempty"`
    Thinking string         `json:"thinking,omitempty"`
    ID       string         `json:"id,omitempty"`
    Name     string         `json:"name,omitempty"`
    Input    map[string]any `json:"input,omitempty"` // see chatResponseToMessage below
}

// render.go chatResponseToMessage — coerce nil/empty to an empty map
// for tool_use blocks so the wire always carries "input": {}.
case canonical.ContentKindToolUse:
    if part.ToolUse != nil {
        input := part.ToolUse.Input
        if input == nil {
            input = map[string]any{}
        }
        out.Content = append(out.Content, contentBlock{
            Type:  "tool_use",
            ID:    part.ToolUse.ID,
            Name:  part.ToolUse.Name,
            Input: input,
        })
    }
```
Add a regression test:
```go
func TestChatResponseToMessage_ToolUseEmptyInput_RendersEmptyObject(t *testing.T) {
    resp := &canonical.ChatResponse{
        Message: canonical.Message{
            Role: canonical.RoleAssistant,
            Content: []canonical.ContentPart{{
                Kind: canonical.ContentKindToolUse,
                ToolUse: &canonical.ToolUsePart{
                    ID: "toolu_1", Name: "ping", Input: nil, // explicit nil
                },
            }},
        },
    }
    out := chatResponseToMessage(resp, "auto")
    wire, _ := json.Marshal(out)
    if !bytes.Contains(wire, []byte(`"input":{}`)) {
        t.Errorf("wire missing required input:{}; got %s", wire)
    }
}
```

### CR-02: Authorization scheme match is case-sensitive — silently routes lowercase "bearer" to x-api-key fallback

**File:** `internal/auth/bearer.go:77-83`
**Issue:** `extractToken` does `strings.HasPrefix(authHeader, bearerPrefix)`
with `bearerPrefix = "Bearer "` (exact case). RFC 7235 §2.1 mandates that
the auth-scheme token (here, "Bearer") is **case-insensitive**, and RFC
6750 explicitly states implementations MUST treat the scheme as case-
insensitive. A client sending `Authorization: bearer s3cret` (lowercase
scheme) will silently fall through to the x-api-key branch — which means:
(a) when AUTH_TOKEN is set and the client does NOT also send x-api-key,
the request is rejected with 401 even though the credential is valid;
(b) when the client sends BOTH a lowercase `Bearer` AND a (stolen or
weaker) x-api-key, the lowercase Bearer is ignored and the request is
auth'd by x-api-key — i.e., the D-15 "Authorization wins to defeat
downgrade attack" contract documented in the bearer.go header comment
is BROKEN for lowercase clients. The downstream impact is at least an
interop bug (lowercase clients silently fail) and at most a downgrade
vector. The Phase 3.1 D-15 dual-header tests
(`TestBearer_DualHeader`) all use the canonical "Bearer" spelling and
therefore do not catch this.

**Fix:**
```go
// bearer.go extractToken — case-insensitive scheme match per RFC 7235 §2.1.
func extractToken(r *http.Request) string {
    const bearerPrefix = "Bearer "
    authHeader := r.Header.Get("Authorization")
    if len(authHeader) >= len(bearerPrefix) &&
        strings.EqualFold(authHeader[:len(bearerPrefix)], bearerPrefix) {
        return authHeader[len(bearerPrefix):]
    }
    return r.Header.Get("x-api-key")
}
```
Add coverage in `auth_internal_test.go`:
```go
func TestExtractToken_LowercaseBearerSchemeAccepted(t *testing.T) {
    t.Parallel()
    r := httptest.NewRequestWithContext(context.Background(), http.MethodGet, "/", nil)
    r.Header.Set("Authorization", "bearer my-token")
    if got := extractToken(r); got != "my-token" {
        t.Errorf("lowercase scheme: got %q, want %q", got, "my-token")
    }
}

func TestExtractToken_LowercaseBearer_DoesNotFallThroughToXAPIKey(t *testing.T) {
    t.Parallel()
    r := httptest.NewRequestWithContext(context.Background(), http.MethodGet, "/", nil)
    r.Header.Set("Authorization", "bearer auth-token")
    r.Header.Set("x-api-key", "key-token")
    if got := extractToken(r); got != "auth-token" {
        t.Errorf("D-15 precedence: lowercase Bearer must still win over x-api-key; got %q", got)
    }
}
```

### CR-03: SSE post-header emitter errors logged at Debug — operators miss real failures

**File:** `internal/adapter/anthropic/handlers.go:92-102`
**Issue:** After `runSSEEmitter` returns a non-nil error, the handler logs
unconditionally at Debug level:
```go
if err := runSSEEmitter(r.Context(), w, runHandle, wire.Model, a.cfg.Logger); err != nil {
    a.cfg.Logger.Debug("anthropic: sse emitter terminated", "err", err)
}
```
The comment justifies this with "ctx cancel is a normal disconnect, not
an error" — but the same return path swallows **real** writeEvent errors
(e.g., `fmt.Fprintf(e.w, ...)` failures from a broken pipe mid-stream,
JSON marshal errors from a malformed payload, the `errors.New("anthropic:
response writer is not flusher")` sentinel even though that branch should
have been handled by the caller's pre-emitter check). At default log level
(`slog.LevelInfo`), these are invisible. Operators investigating
"streaming responses sometimes truncate" have no signal to correlate. The
spec violation is mild but the observability gap is real: T-3.1-FRAMEERR
in the plan explicitly calls out that frame-write failures should be
auditable. Distinguishing `errors.Is(err, context.Canceled)` /
`context.DeadlineExceeded` from everything else and logging the latter at
Warn closes the gap.

**Fix:**
```go
if err := runSSEEmitter(r.Context(), w, runHandle, wire.Model, a.cfg.Logger); err != nil {
    switch {
    case errors.Is(err, context.Canceled), errors.Is(err, context.DeadlineExceeded):
        // Normal client disconnect or request deadline.
        a.cfg.Logger.Debug("anthropic: sse stream terminated by client", "err", err)
    default:
        // Real frame-write or marshal failure — operator-visible.
        a.cfg.Logger.Warn("anthropic: sse emitter error", "err", err)
    }
}
```
Add `"errors"` and `"context"` imports if not already wired (handlers.go
already imports neither — `errors` is used elsewhere in the package, but
add a focused import here).

### WR-01: genMessageID fallback returns a constant — concurrent crypto/rand failures cause ID collisions

**File:** `internal/adapter/anthropic/render.go:178-184`
**Issue:** When `crypto/rand.Read` fails, the function returns the literal
string `"msg_01_rand_unavailable"`. If the failure is transient and
multiple in-flight requests hit it within the same window, all of them
emit the SAME message id. SDK consumers, log correlation tooling, and
the loop24-client request-tracer treat the id as opaque-but-unique. A
duplicated id can short-circuit deduplication logic and confuse log
trails. Although crypto/rand.Read is documented as non-blocking and
should never fail on supported platforms (Linux getrandom, macOS
arc4random, Windows BCryptGenRandom), the regex normalization in
`messageIDRegexp` (sse_golden_test.go:37) already widens the pattern to
`[0-9a-fA-F_]+` "to tolerate the fallback id `msg_01_rand_unavailable`",
which confirms the path is acknowledged. A monotonically-distinct
fallback closes the collision risk without changing the happy path.

**Fix:**
```go
import "sync/atomic"

var messageIDFallbackCounter uint64

func genMessageID() string {
    var b [12]byte
    if _, err := rand.Read(b[:]); err != nil {
        n := atomic.AddUint64(&messageIDFallbackCounter, 1)
        return fmt.Sprintf("msg_01_rand_unavailable_%016x", n)
    }
    return "msg_01" + hex.EncodeToString(b[:])
}
```

### WR-02: SSE applyChunk emits empty content_block_start + content_block_stop when first chunk has nil pointer

**File:** `internal/adapter/anthropic/sse.go:204-270`
**Issue:** When the first chunk is `{Kind: ChunkKindText, Text: nil}` (or
ChunkKindThought with nil Thought), `applyChunk` runs step 3 (emit
content_block_start, mark blockOpen=true, set currentKind) BEFORE step 4
detects the nil pointer and returns nil. The block is now "open" with no
delta ever emitted. If the next chunk is a different supported kind, step
2 will close the empty block and bump the index — producing wire output
where `content_block_start` for index 0 is followed directly by
`content_block_stop` for index 0 with no `content_block_delta` between
them. The Anthropic SDK accepts this shape (an empty text block is valid),
but it is wasted bandwidth and trips loop24-client's hot-path assertion
that each `content_block_start` corresponds to ≥1 delta. The defensive
nil-pointer check should happen in step 1 BEFORE state mutation.

**Fix:**
```go
func (e *sseEmitter) applyChunk(c canonical.Chunk) error {
    // Step 0 (new): short-circuit on supported kinds with nil payload
    // BEFORE any state mutation, to avoid opening an empty block.
    switch c.Kind {
    case canonical.ChunkKindText:
        if c.Text == nil {
            e.logger.Debug("anthropic: sse text chunk with nil payload dropped")
            return nil
        }
    case canonical.ChunkKindThought:
        if c.Thought == nil {
            e.logger.Debug("anthropic: sse thought chunk with nil payload dropped")
            return nil
        }
    }
    // existing step 1 (block header identification) follows...
```
Add a regression test in sse_test.go asserting `applyChunk({Kind:Text,
Text:nil})` produces zero events and leaves `blockOpen=false`.

### WR-03: writeSSEError swallows marshal error onto a fixed body without logger context

**File:** `internal/adapter/anthropic/errors.go:107-120`
**Issue:** `writeSSEError` catches a json.Marshal failure and substitutes a
hardcoded fallback body. The fallback path is unreachable in practice
(marshaling a struct of strings is total), but if the unreachable becomes
reachable (e.g., a future schema change introduces a non-marshalable
field), the swallowed `err` is lost — no logger is available because the
function signature does not accept one. Other writeError paths in this
file also discard encoder errors (acceptable, since the client has
disconnected), but writeSSEError's discarded error is **pre-write** and
indicates a programmer-level bug that would otherwise be silent.

**Fix:** Either (a) accept a `*slog.Logger` parameter and log the marshal
failure at Error level, or (b) document explicitly via comment that the
fallback path is unreachable-by-construction and remove the runtime
guard. Option (a) is consistent with the package's general "log structured,
respond generic" pattern:
```go
func writeSSEError(w http.ResponseWriter, flusher http.Flusher, logger *slog.Logger, errorType, message string) {
    body, err := json.Marshal(errorEnvelope{...})
    if err != nil {
        if logger != nil {
            logger.Error("anthropic: writeSSEError marshal failure (should be impossible)", "err", err)
        }
        body = []byte(`{"type":"error","error":{"type":"api_error","message":"internal error"}}`)
    }
    // ...
}
```
Update call sites in sse.go finalizeStream accordingly.

### WR-05: handlers.go writes "invalid JSON: <err.Error()>" — risk of payload echo on future json.Decoder changes

**File:** `internal/adapter/anthropic/handlers.go:58-63`
**Issue:** The error envelope message is constructed as
`"invalid JSON: " + err.Error()`. The comment justifies this with "json.
NewDecoder errors do not embed body content per stdlib invariants" — that
is true TODAY (the decoder reports offsets and type names, not bytes), but
this is an unguaranteed implementation detail. A future Go release could
change the error message format to include surrounding bytes (proposals
to do so have surfaced periodically on the issue tracker). T-02-33's
contract is "never echo request content"; bridging it through a stdlib
invariant rather than an explicit sanitization is fragile. The
`TestHandleMessages_BadJSON_400` test only asserts the literal `{not-
valid-json` is absent — a future stdlib that echoes "near `{not-` of input"
would pass that substring check (substring NOT in error message ≠ never
in error message; depends on what slice the stdlib chose).

**Fix:** Sanitize the message to a static string or to a structured
position-only description:
```go
// Option A: static message, log raw err for diagnosis.
a.cfg.Logger.Debug("anthropic: invalid JSON in /v1/messages", "err", err)
writeError(w, http.StatusBadRequest, errInvalidRequest, "invalid JSON in request body")

// Option B: extract only the syntax error position (no surrounding bytes).
var syntaxErr *json.SyntaxError
if errors.As(err, &syntaxErr) {
    writeError(w, http.StatusBadRequest, errInvalidRequest,
        fmt.Sprintf("invalid JSON at offset %d", syntaxErr.Offset))
} else {
    writeError(w, http.StatusBadRequest, errInvalidRequest, "invalid JSON in request body")
}
```

### WR-06: anthropicEngineAdapter.Run does not propagate ctx cancellation symmetry — wrap on success returns a bare RunHandle

**File:** `cmd/loop24-gateway/main.go:308-314`
**Issue:** `anthropicEngineAdapter.Run` returns `anthropicRunHandleAdapter{run: run}` (no error) when the underlying `engine.Run` succeeds. That's
correct, but the inner `anthropicRunHandleAdapter` (lines 322-337) holds
ONLY the `*engine.Run`. When the SSE emitter loop returns on `ctx.Done`,
no signal is sent to the underlying engine run — the engine's own ctx
propagation must do the work. This is documented elsewhere but the
adapter has no defensive `Close` method to ensure the underlying run is
torn down even if the caller forgets to cancel ctx. For Phase 3.1 the
caller (handlers.go) does pass `r.Context()` to Engine.Run, so cancellation
propagates correctly — but the adapter swallows future flexibility. The
risk is low today; the comment on the type should explicitly note that
the adapter relies on caller-side ctx cancellation for run teardown.

**Fix:** Add a comment doc to `anthropicRunHandleAdapter` clarifying the
caller's contract:
```go
// anthropicRunHandleAdapter adapts *engine.Run to anthropic.RunHandle.
// IMPORTANT: this adapter holds NO Close method. The caller (the SSE
// emitter in adapter/anthropic) MUST ensure the context passed to
// Engine.Run is cancelled when the handle is no longer needed, so the
// underlying *engine.Run can tear down the kiro-cli session and free
// the pool slot. handleMessages satisfies this by passing r.Context().
type anthropicRunHandleAdapter struct {
    run *engine.Run
}
```

### WR-08: scriptedChunks comment contradicts implementation (unbuffered claim vs buffered reality)

**File:** `internal/adapter/anthropic/sse_test.go:27-38`
**Issue:** The comment says "The channel is unbuffered-but-pre-filled via
a goroutine that sends and closes; goleak verifies the sender goroutine
exits." The actual code makes a BUFFERED channel of length `len(chunks)`,
fills it inline (no goroutine), and closes it. There is no sender
goroutine. The comment is stale documentation that will mislead future
contributors trying to debug a leak warning (they'll search for a
non-existent sender goroutine).

**Fix:**
```go
// scriptedChunks builds a closed, pre-filled buffered channel of the
// supplied chunks. The buffer is sized to len(chunks) so all sends
// succeed before close — no producer goroutine is needed, and goleak
// has nothing to verify on this code path.
func scriptedChunks(chunks ...canonical.Chunk) <-chan canonical.Chunk {
    ch := make(chan canonical.Chunk, len(chunks))
    for _, c := range chunks {
        ch <- c
    }
    close(ch)
    return ch
}
```

**File:** `internal/adapter/anthropic/handlers_test.go:482-518`
**Issue:** `TestAuth_401_OnMessages` documents that Anthropic /v1/messages
returns the OLLAMA-shaped 401 envelope (`{"error": "Invalid or missing
API key"}`) as an intentional Phase 3.1 carry-forward, with Phase 8
planned to lift this into surface-aware error rendering. This is fine,
but no test exists that pins the **current** wire shape as a regression
guard against Phase 8 work accidentally breaking Phase 3.1 clients during
the transition. The single test asserts only `strings.Contains` of the
Ollama message substring. A byte-equal golden compare would catch any
drift (e.g., trailing newline change, key ordering, content-type
parameter additions).

**Fix:** Strengthen the existing assertion to byte-exact:
```go
wantBody := `{"error":"Invalid or missing API key"}` + "\n"
if got := w.Body.String(); got != wantBody {
    t.Errorf("401 body: got %q, want %q (Phase 3.1 contract carry-forward)", got, wantBody)
}
```
