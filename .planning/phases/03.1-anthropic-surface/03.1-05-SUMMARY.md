### Exact code change (render.go)

**contentBlock struct (line ~48, before):**
```go
Input    map[string]any `json:"input,omitempty"`
```

**contentBlock struct (line ~48, after):**
```go
// Input is *map[string]any (not map[string]any) so the omitempty tag
// distinguishes "this is not a tool_use block, drop the field" (nil
// pointer) from "this is a tool_use block with no arguments, emit
// {}" (non-nil pointer to empty map). With a plain map type,
// encoding/json's omitempty drops both nil AND len==0 maps — so
// `Input: map[string]any{}` would still vanish from the wire.
// Indirection through a pointer lets chatResponseToMessage carry
// "field present, value empty" into the marshaller (CR-01 fix —
// VERIFICATION.md gap 1).
Input *map[string]any `json:"input,omitempty"`
```

**chatResponseToMessage ContentKindToolUse branch (lines ~104-112, before):**
```go
case canonical.ContentKindToolUse:

        out.Content = append(out.Content, contentBlock{
            Type:  "tool_use",

            Input: part.ToolUse.Input,

```

**chatResponseToMessage ContentKindToolUse branch (after):**
```go
case canonical.ContentKindToolUse:
    if part.ToolUse != nil {
        // Anthropic spec requires tool_use.input as a JSON object even
        // when empty. Coerce nil/empty to map[string]any{} so json.Marshal
        // emits "input":{} rather than dropping the field via omitempty
        // (CR-01 fix — VERIFICATION.md gap 1). The pointer indirection
        // on contentBlock.Input is what makes omitempty preserve the
        // field for an empty-but-present map.
        input := part.ToolUse.Input
        if len(input) == 0 {
            input = map[string]any{}
        }
        out.Content = append(out.Content, contentBlock{
            Type:  "tool_use",
            ID:    part.ToolUse.ID,
            Name:  part.ToolUse.Name,
            Input: &input,
        })
    }
```

### Confirmation: `omitempty` intentionally preserved

The `json:"input,omitempty"` struct tag was kept exactly as in the original code (verified by `grep -c 'json:"input,omitempty"' internal/adapter/anthropic/render.go` → `1`). The plan explicitly required omitempty to remain so that Text/Thinking content blocks (whose `Input` field is unset, i.e. a nil pointer) continue to elide the field from their wire output rather than emitting `"input":null`. The `TestChatResponseToMessage_GoldenWireShape` test, which pins `{"type":"text","text":"Hello!"}` (no `input` key) for text blocks, continues to pass — confirming the omitempty preservation works.

## Decisions Made

- **Pointer-to-map for omitempty-presence semantics (deviation from plan-prescribed approach).** The plan called for coercing nil/empty `ToolUse.Input` to `map[string]any{}` and leaving the contentBlock field as `map[string]any`. This was tested and confirmed insufficient — Go's `encoding/json` `omitempty` tag treats len==0 maps as empty and drops the field regardless of nil vs allocated. The pointer indirection (`*map[string]any`) is the idiomatic Go fix: `omitempty` on a non-nil pointer to an empty map keeps the field; nil pointer drops it. This preserves the plan's behavioral truths exactly (text/thinking blocks elide `input`, tool_use always emits it) without resorting to a custom `MarshalJSON`. See deviation below.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Plan-prescribed coercion was insufficient because Go's `encoding/json` drops empty maps under `omitempty`**

- **Found during:** Task 2 GREEN verification
- **Issue:** The plan instructed: leave `Input map[string]any \`json:"input,omitempty"\`` as-is, and at the call site coerce `nil`/empty `Input` to `map[string]any{}`. This was implemented exactly as written, then `go test` was run — both sub-cases still FAILED with `field absent (omitempty dropped it)`. Validated against a minimal repro: Go's `encoding/json` treats `len(map) == 0` as the empty value for the omitempty check, so an explicit `map[string]any{}` is dropped just like a nil map. The plan's behavioral truth ("`"input":{}` always present for tool_use") and its decision rationale ("preserve omitempty for OTHER fields") are both still correct — only the prescribed mechanical implementation was wrong.
- **Fix:** Changed `contentBlock.Input` from `map[string]any` to `*map[string]any` (added doc comment explaining the indirection rationale). At the call site, after coercing `input := part.ToolUse.Input` then `if len(input) == 0 { input = map[string]any{} }`, assigned `Input: &input` (pointer to the coerced local). `omitempty` on a non-nil pointer is satisfied; on a nil pointer (text/thinking blocks where the field is never set), the field is dropped — matching plan intent.
- **Files modified:** `internal/adapter/anthropic/render.go` (the type change + one extra line of doc comment beyond plan spec; the call-site `len(input) == 0` guard is verbatim from the plan)
- **Verification:** `go test ./internal/adapter/anthropic/ -race -count=1` → green. Full repo `go test ./... -race -count=1` → green. `TestChatResponseToMessage_GoldenWireShape` (which would break if Text blocks emitted `"input":null`) still passes — confirming the pointer-type fix preserves omitempty behavior for non-tool_use blocks.
- **Committed in:** `407cbcf` (Task 2 GREEN commit)
