## Answers to Open Questions (committed)

### Q1 — Template resolution

**Decision:** Template lives at `$OTTO_INSTALL_ROOT/scripts/.env.otto-gw.example` and is found via the existing install-root resolver that the wrapper already computes at the top of `scripts/otto-gw` (the `_otto_source` symlink-follow block, lines 35-43). PS1 uses `Join-Path $InstallRoot 'scripts\.env.otto-gw.example'` (mirrors the existing `$InstallRoot = Split-Path -Parent $PSScriptRoot` at line 69).

The wrapper already resolves the install root to handle the one-liner installer's PATH symlink (`~/.local/bin/otto-gw` → `~/.otto-gw/scripts/otto-gw`). The template is a peer of the wrapper inside `scripts/`, so the same anchor that finds `bin/otto-gateway` finds the template.

- **Dev path:** `<repo>/scripts/.env.otto-gw.example` (since `OTTO_INSTALL_ROOT = <repo>`).
- **Shipped path:** `<install-root>/scripts/.env.otto-gw.example` (since `OTTO_INSTALL_ROOT = <install-root>`).
- **Override:** introduce `OTTO_TEMPLATE_FILE` env (and `--template PATH` CLI flag on `upgrade-env` + `migrate-to-overrides`) for the edge case where an operator wants to test an unreleased template without reinstalling.
- **Boot-error if missing:** if the template isn't found at any resolved path, both subcommands abort with a clear message (matches `init_cmd`'s existing template-missing handling at scripts/otto-gw:925-930).

### Q2 — Single-file deprecation-warn detection

**Decision:** Emit the WARN on `start` (and `restart`, by transitivity) when **all** of the following hold:

1. `.otto-gw.overrides.env` does **not** exist at any of the resolved overrides paths.
2. `.otto-gw.env` (or the resolved single-file path) **does** exist.
3. At least one of `AUTH_TOKEN`, `PII_HASH_KEY`, or `HTTP_ADDR` is uncommented in `.otto-gw.env` with a non-placeholder value, OR `KIRO_CMD` is uncommented and non-empty. (These are the keys `init` writes uncommented and that meaningfully indicate operator customization vs. a virgin template copy.)

The check piggybacks on the existing `env_is_uncommented` helper (scripts/otto-gw:164-168) — no new parsing primitives needed. Implementation is a single new function `detect_single_file_model` returning 0/1, called once from `load_config` (after env files are resolved, before `apply_cli_flags`), behind a `_warn_already_emitted` sentinel so the line never repeats inside one invocation.

The WARN looks like:

```
otto-gw: legacy single-file .env model detected — run `otto-gw migrate-to-overrides` to split secrets/overrides into .otto-gw.overrides.env. Single-file support will be removed in v1.7.
```

Explicitly **not** an error and explicitly **not** auto-migrating — operator decides when to run the command.

### Q3 — Init's secret-writing path

**Decision:** **Always write to overrides going forward**, even for fresh installs. The generated `.otto-gw.env` becomes a pure template copy with every line commented (including the secret placeholders that are already commented in the template today). Rationale:

- Keeps the generated file bit-identical to `scripts/.env.otto-gw.example` so `upgrade-env` can overwrite it without ceremony.
- Eliminates a forking branch in `init_cmd` ("does an overrides file exist? If yes write here, if no write there") — fewer states, fewer bugs.
- Matches the locked Decision 5 in CONTEXT.md ("`init` should write secrets to overrides instead of the main file") and Decision 4's idempotency guarantee for migrate-to-overrides.

`init_cmd` after this change:

1. Copy the template byte-for-byte to `.otto-gw.env` (still uses the existing template-copy + header-strip block at lines 936-946; the `set_env_line` calls at lines 949-957 are **removed**).
2. Write `AUTH_TOKEN`, `PII_HASH_KEY`, plus any operator-supplied values (`--kiro`, `--addr`, `--pii`, `--idle-timeout`, `--auth-enabled`, `--chat-trace`) to `.otto-gw.overrides.env` (uncommented, since they're the operator's intentional customizations). The new overrides file is mode 0600.
3. The generated `.otto-gw.env` is mode 0644 (it's a template copy with no secrets; no reason to lock it down).

Re-init (`--force`) on an existing overrides file preserves secrets the same way it does today (extract existing values, reuse unless `--regenerate-secrets`). Re-init on an existing single-file `.otto-gw.env` (no overrides yet) triggers an internal `migrate-to-overrides` pass first, then proceeds normally — this means an operator's first `init --force` after upgrading the wrapper automatically migrates their install. (Belt-and-suspenders with the start-time WARN.)

```
upgrade_env_cmd():
    parse_flags "$@"
    template=$(resolve_template_file) || abort "template not found"
    dest=${FLAG_DEST:-$(resolve_env_file)}
    [[ -z "$dest" ]] && dest="${DEFAULT_ENV_PATHS[0]}"   # cold-init fallback: write to project-local

    # Build add/orphan/unchanged lists by comparing key sets.
    template_keys_list=$(template_keys "$template")
    if [[ -f "$dest" ]]; then
        current_keys=$(env_keys_present "$dest")
    else
        current_keys=""
    fi

    added=$(comm -23 <(printf '%s\n' "$template_keys_list" | sort) <(printf '%s\n' "$current_keys" | sort))
    orphaned=$(comm -13 <(printf '%s\n' "$template_keys_list" | sort) <(printf '%s\n' "$current_keys" | sort))
    unchanged=$(comm -12 <(printf '%s\n' "$template_keys_list" | sort) <(printf '%s\n' "$current_keys" | sort))

    Print added (count + key list), orphaned (count + key list), unchanged (count only).

    if [[ "$FLAG_DRY_RUN" == "1" ]]; then
        Print "(dry-run; nothing written)"; return 0
    fi

    # Confirm before overwriting (skip when --yes).
    if [[ "$FLAG_YES" != "1" && -f "$dest" ]]; then
        read -r -p "Overwrite $dest with current template? [y/N]: " reply
        case "$reply" in [Yy]|[Yy][Ee][Ss]) ;; *) abort "cancelled" ;; esac
    fi

    # Write orphan log BEFORE overwriting (so a SIGKILL between the two leaves the operator
    # with their old file AND a log of what's about to be lost).
    if [[ -n "$orphaned" ]]; then
        upgrade_log="$HOME/.otto-gw.upgrade.log"   # configurable via $OTTO_UPGRADE_LOG
        ts=$(date -u +%Y-%m-%dT%H:%M:%SZ)
        {
            echo "# $ts upgrade-env removed orphaned keys from $dest"
            for k in $orphaned; do
                v=$(extract_env_any "$dest" "$k" 2>/dev/null || echo "<unset>")
                echo "$k=$v"
            done
            echo "# ---"
        } >> "$upgrade_log"
        Print "  orphan values logged to: $upgrade_log"
    fi

    # Regenerate by copying the template byte-for-byte (no header rewrite — the
    # generated file is now a literal copy, matching the new init contract).
    cp "$template" "$dest"
    chmod 644 "$dest"

    Print "✓ wrote $dest"
    Print "  added:     N (key list)"
    Print "  orphaned:  M (key list, logged to $upgrade_log)"
    Print "  unchanged: K"
```

`comm` is a POSIX core util — available on macOS BSD and every Linux. Process substitution (`<(...)`) is bash-3.2-safe (it's bash-feature, not bash-4-feature).

**Sub-C: add `migrate-to-overrides` subcommand** as `migrate_to_overrides_cmd`. Behavior:

```
migrate_to_overrides_cmd():
    parse_flags "$@"
    template=$(resolve_template_file) || abort
    dest=${FLAG_DEST:-$(resolve_env_file)}
    [[ -z "$dest" || ! -f "$dest" ]] && abort "no .otto-gw.env found at $dest"
    overrides_dest=${FLAG_OVERRIDES_DEST:-"$(dirname "$dest")/.otto-gw.overrides.env"}

    # Idempotency check: dest is already a pure template copy AND overrides file already exists → nothing to do.
    if [[ -f "$overrides_dest" ]] && diff -q "$dest" "$template" >/dev/null 2>&1; then
        Print "already migrated (no-op)"
        return 0
    fi

    # Build the migration delta: every key uncommented in dest whose value differs
    # from the template default goes to overrides.
    keys_in_dest=$(env_keys_present "$dest")
    migrations=""   # space-separated KEY list (bash 3.2 — no arrays in functions where possible)
    for k in $keys_in_dest; do
        # Only consider uncommented lines (operator-set).
        env_is_uncommented "$dest" "$k" || continue
        cur=$(extract_env_any "$dest" "$k")
        def=$(extract_default_value "$template" "$k" 2>/dev/null || echo "<MISSING>")
        if [[ "$cur" != "$def" ]]; then
            migrations="${migrations:+$migrations }$k"
        fi
    done

    # Always carry secrets across, even if they happen to equal the placeholder default.
    for k in AUTH_TOKEN PII_HASH_KEY; do
        if env_is_uncommented "$dest" "$k"; then
            case " $migrations " in
                *" $k "*) ;;
                *) migrations="${migrations:+$migrations }$k" ;;
            esac
        fi
    done

    Print "would migrate to $overrides_dest: $migrations"
    if [[ "$FLAG_DRY_RUN" == "1" ]]; then
        Print "(dry-run; no changes written)"; return 0
    fi

    # Confirm overwrite of dest (overrides_dest is append-or-create, never destructive).
    if [[ "$FLAG_YES" != "1" ]]; then
        read -r -p "Backup $dest and regenerate from template? [y/N]: " reply
        case "$reply" in [Yy]|[Yy][Ee][Ss]) ;; *) abort "cancelled" ;; esac
    fi

    # Backup.
    backup="$dest.pre-migrate.$(date -u +%Y%m%dT%H%M%SZ)"
    cp "$dest" "$backup"
    Print "  backup: $backup"

    # Write migrations to overrides.
    for k in $migrations; do
        v=$(extract_env_any "$dest" "$k")
        set_overrides_line "$overrides_dest" "$k" "$v"
    done

    # Regenerate dest from template (pure copy).
    cp "$template" "$dest"
    chmod 644 "$dest"

    Print "✓ migrated $(echo "$migrations" | wc -w | tr -d ' ') keys to $overrides_dest"
    Print "✓ regenerated $dest from template"
    Print "✓ backup at  $backup"
```

Every `Edit` call's `file_path` should resolve under that root. If a tool reports "file not found" or the `<system-reminder>` block shifts working directories mid-task, STOP and verify cwd before continuing.

2. **`comm` portability.** `comm` is in POSIX core utils and ships on macOS BSD + every Linux. If a future install target (e.g. busybox-only containers) lacks it, swap for a `sort -u | uniq -c` equivalent. Not a current concern.

3. **bash 3.2 (stock macOS).** No associative arrays (`declare -A`), no `mapfile`/`readarray`, no `${var^^}`. All helpers use POSIX-portable string ops + `awk`. Task 1's helpers were drafted with this in mind.

4. **Template+overrides path resolution interaction.** If an operator uses `--env-file /custom/path/.env` AND has no `--overrides-file`, the wrapper should NOT auto-find an overrides file next to the custom env path — it should look in the DEFAULT_OVERRIDES_PATHS list per the spec (or `$OTTO_OVERRIDES_FILE`). The two flags are decoupled. Document this in the Task 1 implementation comment.

5. **PS1 live verification.** macOS dev box has no pwsh. All PS1 tasks rely on (a) static review against bash, (b) operator-side smoke on Windows after merge. This matches the locked policy from 260531-oax and is acceptable per CONTEXT.md Decision 7.

6. **Edge: a virgin install with no existing .env.** The first `otto-gw init` writes both files; `otto-gw upgrade-env` on this install is effectively a no-op (template == dest). `otto-gw migrate-to-overrides` on this install errors with "no .otto-gw.env found" if the operator hasn't run `init` yet — acceptable, matches the existing init-first contract.

7. **Edge: operator hand-edited `.otto-gw.env` after migrate.** Documented as undefined-but-harmless: their edits are overwritten on the next `upgrade-env`. The deprecation WARN doesn't re-fire after migration because the overrides file exists, but if they re-introduce an uncommented operator value into `.otto-gw.env`, the WARN does NOT detect this case (the WARN requires absence of overrides file). This is the deliberate v1.6 → v1.7 transition shape; closing the loophole is v1.7 work.

8. **Orphan log location.** Default `$HOME/.otto-gw.upgrade.log` is per-user. CI-style installs with no `$HOME` should set `$OTTO_UPGRADE_LOG` to a writable path. The wrapper should fall back to `./.otto-gw.upgrade.log` (project-local) if `$HOME` is unset, NOT silently swallow the log.
