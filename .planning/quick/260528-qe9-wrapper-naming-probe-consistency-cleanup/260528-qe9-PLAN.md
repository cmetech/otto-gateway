<objective>
Two small fixes surfaced during the Windows operator smoke test in this
session, bundled into one quick task with two atomic commits.

Goal A: align the bash wrapper's readiness-probe default with the
PowerShell wrapper's recent fix (commit `7185293`). `localhost` resolves
to IPv6 first on Windows; the gateway binds IPv4 only; switching the
default to the literal `127.0.0.1` eliminates the DNS family mismatch.
Harmless on Linux/macOS, prevents future WSL2 mirrored-mode surprises.

Goal B: align the gateway binary's compiled-in `KIRO_CMD` default with
what both wrappers' Preflight functions auto-detect. Wrappers look for
`kiro` on PATH; the binary's fallback was `kiro-cli`. Result: when
KIRO_CMD is unset, the binary's "executable not found" error names a
different binary than the wrapper's "not on PATH" warning, confusing
operators.
</objective>

<commit_messages>

**A:**
```
fix(otto-gw): default OTTO_ADDR probe to 127.0.0.1 (Linux/macOS consistency with .ps1 #7185293)

Matches the PowerShell wrapper fix in 7185293. `localhost` resolves
IPv6-first on Windows and the gateway binds IPv4 only, so the probe
never connects there. On Linux/macOS the resolution is consistent so
this is currently harmless, but the change is consistency-preserving
and prevents future WSL2 mirrored-mode surprises.
```

**B:**
```
fix(config): default KIRO_CMD to "kiro" (matches what wrappers auto-detect)

Both wrapper Preflight functions auto-detect via `command -v kiro` /
`Get-Command kiro`, but the gateway's compiled-in fallback was
`kiro-cli`. When KIRO_CMD was genuinely unset, the gateway exec error
named a different binary than the wrapper's "not on PATH" warning,
confusing operators. Aligning the default removes the divergence;
deliberate test inputs that still use "kiro-cli" as a fixture value
are unaffected (those are inputs, not assertions of the default).
```
