<objective>
Fire PostHooks on streaming responses across all three API surfaces (Anthropic SSE, Ollama NDJSON for /api/chat + /api/generate, OpenAI SSE) so that LoggingHook.After emits `plugin.after` records, ChatTraceHook.After emits `post_chain_out` records, and any future PostHook (audit, metrics) sees every request that completes — not just the non-streaming subset.

Purpose: Operators currently see zero `post_chain_out` records in chat-trace.log + zero `plugin.after` lines in main log despite many pre/before records. Root cause: `engine.Collect` calls PostHooks at collect.go:118-122, but every streaming adapter calls `eng.Run` + `runSSEEmitter`/`runNDJSONEmitter` then returns WITHOUT calling Collect. PostHook invocation is gated on a code path streaming requests don't take.

This is pre-existing, not introduced by 260529-ll2 (ChatTraceHook). ll2 made it operator-visible because chat-trace's product value IS the pre/post pair. The Anthropic non-streaming path has the same gap (`CollectAnthropicChat` doesn't call PostHooks either — D-07 exception bypassed `eng.Collect` and the PostHook traversal with it).

Output: A shared `engine.RunPostHooks` helper invoked by every adapter at end-of-stream (and by `CollectAnthropicChat` at end-of-aggregation), with each adapter materializing a `*canonical.ChatResponse` rich enough (Message.Content populated from emitted chunks) that downstream PostHooks have something meaningful to observe.

LOAD-BEARING CORRECTNESS RISK (call this out explicitly — analogous to the dataset.raw / chain-order invariants in prior plans):

The single biggest correctness risk is **streaming aggregator richness**. LoggingHook.After reads `resp.StopReason` + `duration_ms` (minimal). ChatTraceHook.After reads `resp.StopReason` AND `resp.Message.Content` (full content array). If a streaming aggregator only tracks stop_reason and discards content, the `post_chain_out` record will have empty content — defeating chat-trace's entire response-side observation product value. Every adapter aggregator built in this plan MUST accumulate text deltas, thinking deltas, AND tool_use chunks into `Message.Content` (Anthropic also into `Message.ToolCalls` per D-07); otherwise we ship a fix that compiles, passes the gate, and produces empty post_chain_out records. The Anthropic adapter's existing `CollectAnthropicChat` (internal/adapter/anthropic/collect.go) already does this aggregation correctly for the non-streaming path — Task 2 reuses its discipline as the template.
</objective>
