<key_correctness_note>
The SINGLE biggest correctness risk in this task is the **ChatTraceHook Pre-chain insertion point relative to PIIRedactionHook**. The whole product value of chat-trace.log is recording what the client *actually said* — pre-redaction. If ChatTraceHook is appended/prepended in the wrong slot (or if a later wiring change reorders the chain so PII redaction runs before trace emission), the chat-trace.log will silently start logging redacted content and the feature becomes worthless without any test failing. The current Pre order is RequestIDHook → AuthHook → PIIRedactionHook → LoggingHook; ChatTraceHook MUST be inserted at index 0 (before RequestIDHook) so it observes the canonical request before any in-place mutation. This is symmetric to the v1.5.4 dataset.raw call-out: the load-bearing invariant is positional, not behavioral, and a future refactor that "tidies" the Pre slice order can break it silently.

Mitigations:
1. Task 1's trace_test.go MUST include a regression test that exercises the full chain with a PII-bearing request and asserts that the ChatTraceHook NDJSON record contains the raw (non-redacted) string. Without that assertion, ordering drift produces a green build with a broken feature.
2. Task 4 (main.go wiring) MUST insert via explicit slice-prepend (`append([]engine.PreHook{chatTrace}, chain.Pre...)`) — not a "find the right index" pattern that could silently no-op if the chain literal grows.
3. The wiring comment in main.go MUST state the invariant verbatim: "ChatTraceHook is first in Pre to observe pre-redaction content. Do not reorder."
</key_correctness_note>
