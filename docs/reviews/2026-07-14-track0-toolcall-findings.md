> **⚠️ Correction (2026-07-15).** The original headline below concluded kiro
> *structurally refuses* prompt-embedded tools and that the fix needs ACP
> `mcpServers` capability negotiation. **That root cause was wrong.** Reading
> the legacy JS gateway (`loop_24/acp_server/acp-server-ollama.js`) — which
> DOES get tool calls in production — shows it *also* prompt-embeds tools; the
> difference is a coercion apparatus the Go gateway lacks, and on one point Go
> does the exact opposite. The corrected root cause + revised Track 3 scope are
> in the two "Corrected" subsections below. The raw per-surface capture
> evidence (0 tool-call frames) is accurate and unchanged — only the
> interpretation was wrong.

## Headline finding (CORRECTED): Go is missing the JS gateway's tool-call coercion apparatus

Every one of the 3 live round-trips (anthropic/openai/ollama) produced the
same outcome: kiro-cli did not invoke `get_weather` and declined in prose
("…appear[s] to be part of a hypothetical or test scenario embedded in the
user message…"). **But this is not kiro refusing embedded tools per se** — the
JS gateway drives kiro to emit tool calls using *the same* prompt-embedding
channel, plus an apparatus the Go gateway does not implement (and, on
permissions, actively inverts). Verified against
`loop_24/acp_server/acp-server-ollama.js`:

1. **Strict function-calling prompt** (JS `~800-818`): frames kiro as a
   function-caller for an EXTERNAL system, forbids its built-in tools, and
   specifies an exact ```` ```json {"tool_call": {...}} ```` ``` output protocol
   with 6 rules. Go emits only a bare one-liner
   (`build_acp.go:105-109`: `[Available tools]\nEmit a tool_call ACP
   notification…`) — weaker, and it asks for a *native ACP notification*
   rather than the JSON-in-text protocol JS relies on.
2. **Permission denial (the key inversion)** (JS `314-338`): when the caller
   supplied tools (`denyKiroTools = !!tools.length`), JS **rejects** kiro's
   `session/request_permission` (`granted:false`, prefers `reject_always`) so
   kiro cannot complete the task with its own built-in tools and is pushed to
   emit `{"tool_call":…}` JSON instead. **The Go gateway does the opposite** —
   `internal/acp/client.go:1176,1186` auto-**grants** every permission
   (`{optionId:"allow_always", granted:true}`), so kiro runs its own built-in
   tools (or, with nothing grantable, declines in prose). This is the primary
   gap.
3. **Corrective nudge + circuit breaker** (JS `1865-1873`, `1990-1996`,
   `MAX_TOOL_DENIALS=4`): if a turn burns on denials without producing a
   tool_call, JS re-prompts "your built-in tools are DISABLED — emit the
   tool_call JSON now," and cancels the turn if kiro keeps looping. Go has none.
4. **Free-text coercion** (`coerceToolCall` / `extractToolCallObjects`):
   extract the `{"tool_call":…}` JSON from the assistant text into structured
   `tool_calls`. This is exactly the Track 3 robustness work.

**Why the Go run captured 0 tool-call frames:** Go has only item 1 in a weak
form and *inverts* item 2 (auto-grant instead of deny), with no items 3–4. So
kiro was never pushed to emit a tool_call. The per-surface sections below are
accurate — 0 tool-call transport activity — but the cause is a missing/inverted
apparatus, **not** a structural kiro refusal and **not** a missing `mcpServers`
channel (JS proves prompt-embedding works). The `sawDeclaredName=true` result
under "tool-name fidelity" is just kiro echoing `get_weather` in prose, not an
invocation.

## Track 3 scope (CORRECTED)

The precondition for Track 3 is **not** an open spike (and not `mcpServers`) —
it is a known, specified port of the JS gateway's tool-call apparatus. Do that
first (call it **Track 3a**), which makes kiro emit tool_calls; then the
robustness items (**Track 3b**) can be sized against real captured output.

**Track 3a — tool-call elicitation apparatus (port from `acp-server-ollama.js`), NEEDED, blocks everything below:**
- Set a per-request `denyKiroTools` when the caller supplied tools, and answer
  `session/request_permission` with a **rejection** (`granted:false`,
  `reject_always`) instead of the current auto-grant at
  `internal/acp/client.go:1186`. (This is the load-bearing inversion.)
- Replace the bare `[Available tools]` one-liner (`build_acp.go:105-109`) with
  the JS strict function-calling prompt + explicit `{"tool_call":…}` JSON
  protocol + the 6 rules.
- Add the corrective re-prompt nudge + a `MAX_TOOL_DENIALS` circuit breaker.

**Track 3b — free-text robustness (size AFTER 3a lands + a real tool_call is captured):**

| Track 3b sub-item | Verdict | Note |
|---|---|---|
| Free-text `{"tool_call":…}` extractor hardening (`extractToolCallObjects`) | **needed (size after 3a)** | JS proves the transport is free-text JSON in assistant text, so an extractor IS required; exact hardening (multi-object, wrappers) sizes against captured output once 3a elicits it. |
| Kiro→client tool-name reconciliation | **likely-not-needed for free-text** | In the JS free-text path the model emits the caller's declared name directly (`{"tool_call":{"name":…}}`), so no `kind` remap is needed there. Remap only matters if native ACP `tool_call` frames are surfaced instead — decide when 3a output is captured. |
| Truncated-JSON repair | **needed (size after 3a)** | JS carries `14bc655`-style repair; whether Go needs it depends on how often kiro's emitted JSON is truncated — measure against captured tool_call frames. |
| Structured `tool_calls` surfacing on OpenAI/Ollama | **needed** | Independent of 3a: once a tool_call is coerced, it must surface as structured `tool_calls`, not `[tool: …]` narration (`collect.go`). |

**Bottom line:** the earlier "uncertain / needs an `mcpServers` spike"
conclusion is withdrawn. Prompt-embedding is the right channel (JS uses it);
the gap is the coercion apparatus (Track 3a), with permission-denial as the
single most important missing piece. A repeat of this capture harness against a
Go gateway patched with 3a will capture kiro's real tool_call shape and finalize
3b sizing.

**Headline: the elicitation apparatus works — all three surfaces now emit

load-bearing inversion) was never exercised live, because kiro never tried a

gateway on `127.0.0.1:18099` with `ACP_CAPTURE=true` and real `kiro-cli`, using

refusal ("I don't have a `get_weather` tool available…"). This is the

tool_call emission on at least one surface) — met on **all three**, not just
one. `internal/acp/client.go`'s `Track 3a: MAX_TOOL_DENIALS reached` log line
and the `session/request_permission` deny branch did **not** fire at all
during this run — `grep -i "permission\|denial" /tmp/track3a-gw.log` returned
nothing.

**Why:** the strict function-calling prompt (`build_acp.go`'s
`[Available tools]` block, rule 6: "Built-in tool attempts are rejected
automatically — if one is rejected, do NOT retry it; output the tool_call JSON
block instead") appears sufficient on its own to make kiro skip attempting its
own built-in tools and go straight to the JSON protocol, at least for
single-turn, single-tool prompts. Two additional ad hoc probes (not part of
the committed harness — driven manually via `curl` against the same live
gateway, same session) tried to provoke a built-in-tool attempt instead:
"list the files in the current working directory" (declared tool
`list_directory_contents`) and "run the shell command `pwd`... using whatever
tool you have access to, built-in or otherwise" (declared tool `run_shell`) —
explicitly inviting kiro to reach for a real built-in fs/shell tool. Both
produced the same outcome: kiro emitted the `{"tool_call":…}` JSON for the
*declared* tool name directly, with zero `session/request_permission` frames
in the capture ring either time.

**What this means for the Task 1 reconciliation ask:** the task brief asked to
reconcile the real `session/request_permission` frame shape
(`permissionParams`/`pickRejectOption`, guessed from the JS reference) against
a live-captured frame. **That reconciliation could not be performed** — no
`session/request_permission` frame was captured in this session, across five
total probes (3 committed + 2 ad hoc). `internal/acp/permission_test.go`'s
fixtures (`{"optionId":"reject_always","kind":"reject_always"}`,
`{"optionId":"allow_once","kind":"allow"}`, toolCall `{"title":"..."}`) remain
JS-reference-derived guesses, unverified against real kiro-cli wire data. The
deny branch and `MAX_TOOL_DENIALS` breaker in `client.go` are covered by unit
tests against those synthetic fixtures (`internal/acp/permission_handler_test.go`,
`internal/acp/permission_test.go`) but have **zero live-fire evidence** as of
this run. A follow-up should either (a) find a prompt/tool combination that
reliably makes kiro attempt one of its own built-in tools despite the strict
prompt (candidates not yet tried: multi-turn conversations, tools whose
declared name collides with a kiro built-in's actual name/description, or
tasks kiro judges as unsafe/high-stakes enough to insist on its own
tooling), or (b) accept that the strict prompt alone is carrying the whole
apparatus for the cases tested so far and treat the deny path as
defense-in-depth whose real-wire-shape verification remains open.
