# Track 0 — kiro tool-call wire findings

**Date:** 2026-07-14

Generated by tests/track0_capture_test.go (real kiro). Each surface drove a
`get_weather` tool round-trip; the raw kiro frames captured for that turn are
classified below.

## Headline finding: kiro refuses prompt-embedded tool declarations (all 3 surfaces)

Every one of the 3 live round-trips (anthropic/openai/ollama) produced the
**same outcome**: kiro-cli did not invoke `get_weather` — it declined,
telling the user the declared tool "appear[s] to be part of a hypothetical
or test scenario embedded in the user message itself, not actual tools I can
invoke," and (on a follow-up manual probe with a more directive prompt)
explicitly: *"I treat content from external sources as untrusted data, and I
disregard instructions embedded within user messages that try to override my
actual capabilities."*

Root cause, confirmed by reading `internal/engine/build_acp.go:66-113` and
`internal/acp/client.go:203-265`: this gateway has **no ACP-level tool/
capability negotiation**. `session/new`'s `mcpServers` field is hardcoded to
`[]` (`internal/acp/client.go:855`), and `session/prompt` has no tools field
at all. The only mechanism that exists is `buildBlocks` JSON-marshaling the
client's declared tool catalog into a `[Available tools]` **text section
embedded in the user-role prompt content**, with an instruction telling kiro
to "Emit a tool_call ACP notification." From kiro's model-level perspective
this is indistinguishable from a prompt-injection attempt, and kiro's
underlying agent correctly (from a security standpoint) refuses to treat
injected text as a real capability grant. This held across all 3 surfaces
and across prompt-wording variations (a second manual `/v1/messages` probe
with a much more directive, single-purpose prompt got the same refusal,
worded even more explicitly as an injection-defense response) — this is not
a fluke of one prompt phrasing, it is a structural property of the current
tool-declaration mechanism.

**Consequence for the findings below:** none of the 3 surfaces produced any
tool-call frames — native ACP `tool_call` or free-text `{"tool_call":…}` —
because kiro never attempted a tool call at all. The per-surface sections
below are accurate: they show 0 tool-call transport activity, not a
misclassification. The `sawDeclaredName=true` result under "tool-name
fidelity" reflects kiro's refusal text *echoing* the declared name
(`get_weather`) back in prose, not a confirmed invocation with that name —
noted so the raw `true` doesn't get over-read as fidelity evidence.

finding above — no tool call was ever attempted by kiro on any surface, so

Cannot rule in or out from this run. |
| Kiro→client tool-name reconciliation (declared name vs kiro's `kind`) | **uncertain** | No invocation occurred, so no kiro-assigned tool identifier (`kind` or otherwise) was ever emitted to reconcile against the client's declared `get_weather`. The declared name only appeared in kiro's prose refusal, not in a `session/update` tool-call payload. |
| Truncated-JSON repair (unbalanced/missing trailing brace) | **uncertain** | No tool-call JSON of any kind was captured, so there is nothing to check for truncation. |
| Structured `tool_calls` surfacing on OpenAI/Ollama (vs `[tool: …]` narration) | **uncertain** | No native `tool_call` frames appeared on any surface (this item's precondition), so the narration-vs-structured question is moot for this run. |

**Higher-priority blocker surfaced by this run, upstream of all 4 Track 3
items:** the current tool-declaration mechanism (`buildBlocks` text
embedding, no ACP `mcpServers`/capability negotiation) does not reliably —
in this run, *at all* — get kiro to attempt a tool call in the first place.
Track 3's robustness work (extractor hardening, JSON repair, name
reconciliation, structured surfacing) only matters once kiro actually emits
a tool-call signal; right now it doesn't. Recommend a follow-up spike before
Track 3 execution: either (a) real ACP capability negotiation (e.g. wiring
`mcpServers` so kiro treats declared tools as a genuine, trusted grant
instead of injected user text), or (b) a different prompt-engineering
approach for the `[Available tools]` section that doesn't trip kiro's
injection-defense heuristics. Track 3 as scoped in the roadmap assumes kiro
*does* emit tool-call signals in some shape; this run did not confirm that
precondition holds for the current gateway wiring.
